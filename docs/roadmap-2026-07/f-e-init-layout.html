1. Why

The orchestrator's entire choreography stack targets SEM_ semantic groups — roles (SEM_ARCHES), ensembles (SEM_ALL, SEM_ALL_LESS_FOCAL), bands and sides. The July roadmap's Prop-grouping assessment concluded this architecture is the right contract: it matches community practice, respects the hard xLights constraint that groups load only at startup (so a static pre-built vocabulary is the only practical design), and several specifics — subtractive ensembles, empirical targetability probing, one source of truth for names, flat groups — beat the obvious alternatives.

But the implementation covers only half of xlights-layout-semantics-spec.md. The five gaps, verbatim from the assessment:

  1. The classifier is prose, not code. Spec §3 (DisplayAs mapping → pixel-count disambiguation → name heuristics → group hints → LLM fallback) has no implementation; layout_semantics.py starts from already-classified Prop objects. This layout was classified once, by hand — the root of the single-layout coupling. No code writes the SEM_ modelGroups either.
  2. The §6 manifest is designed but never emitted or consumed — the biggest miss. layout_semantics.json exists nowhere. The Director receives a flat list of group names, so the LLM plans knowing nothing of scale, geometry, or symmetry; QA fakes capability gating with name prefixes.
  3. Group render modes aren't code-managed. Spec §5.7 requires "Per Preview" on ensembles and "Horizontal Per Model" on _LTR groups, but only GridSize is patched. A wrong mode silently breaks every sweep on that group.
  4. The choreography vocabulary is hardcoded to this layout's prop mix (METRIC_RING, backbeat/bed preferences). A different prop mix gets a weaker beat anchor by accident, not by decision.
  5. Validation is manual. The spec's §7 role-color and sweep tests are described but not implemented, even though the offline preview renderer and visual critic could run them — and misclassification poisons every downstream decision.

F-E closes all five as one guided CLI command, xlo init-layout — the step that turns a personal tool into a shareable one. Today, onboarding a new layout means a human (or an ad-hoc agent session) reading the spec and hand-editing xlights_rgbeffects.xml; that is exactly what docs/usage.md ("One-time layout setup", lines 53–69) documents as a "manual/agent step today".

2.2 What doesn't exist — the judgment half

There is no classifier, no spatial deriver, no group writer, no manifest in the package. A grep for layout_semantics.json across packages/ returns nothing; the only callers of build_sem_groups are tests. Historical note: the archived change openspec/changes/archive/2026-06-10-add-layout-semantics/ designed (and its tasks.md checks off) a classifier + spatial derivation + rgbeffects writer + manifest — but only the group-plan and patch functions survived into layout_semantics.py. The classification and group creation for the current layout were executed as a one-time agent/manual step; the July roadmap's assessment ("the classifier is prose, not code … group creation is agent/manual today") is the authoritative statement of what is actually in the tree, and this doc designs against that reality. The archived change's decisions (DisplayAs map, tree threshold, outlier handling, backup/atomic write, xLights-closed guard) remain valid prior art and are folded in below.

3.1 The classifier — spec §3 as real code (knowledge/layout_classify.py, proposed)

Parsing. The preview module's parse_models() is close but wrong-shaped for this: it requires networks.xml controllers to resolve start channels (irrelevant to classification) and drops models it can't channel-resolve. Propose a dedicated, dependency-free parser that also captures what classification needs and parse_models ignores — StringType, <subModel> children (preserved for F-F), and user group membership:

# knowledge/layout_classify.py  (proposed)

def parse_props(rgb_path: str | Path, *, layout_group: str = "Default") -> list[Prop]:
    """One Prop per <model> in the given preview (spec §8: default preview only).
    Populates name, display_as, nodes (CustomModel grid max / parm math per DisplayAs,
    mirroring preview/layout.py), wx/wy (WorldPosX/Y), groups (from <modelGroups>
    membership, reverse-indexed), submodel names. Roles/spatial left at defaults."""

Node counting reuses the proven rules in preview/layout.py:141-149: Custom → max(CustomModel grid) via the _parse_custom_model logic; Cube → parm1*parm2*parm3; everything else → parm1*parm2. Prop gains three fields (defaulted, backward-compatible): mirror_of: str | None = None, string_type: str = "", submodels: list[str] = field(default_factory=list).

Classification steps (each step only touches props not yet resolved, spec §3):

#  Step 1 — DisplayAs direct map (data, not code):
DISPLAYAS_ROLE = {
    "Arches": "ARCH", "Icicles": "ICICLES", "Candy Canes": "CANE", "Star": "STAR",
    "Spinner": "SPINNER", "Matrix": "MATRIX", "Horiz Matrix": "MATRIX",
    "Vert Matrix": "MATRIX", "Window Frame": "WINDOW",
}

#  Step 2 — tree pixel-count disambiguation:
MEGA_TREE_NODES = 600
def _tree_role(p: Prop, all_props: list[Prop]) -> str:
    # Tree 360 / Tree Flat / Tree 180: ≥600 nodes → MEGA_TREE, else MINI_TREE;
    # a sole tree that is also the layout's largest prop → MEGA_TREE regardless.

#  Step 3 — name heuristics (case-insensitive substring, spec table verbatim):
NAME_HEURISTICS: list[tuple[tuple[str, ...], str]] = [
    (("roof", "gutter", "eave", "ridge", "outline", "peak", "fascia",
      "column", "garage"), "OUTLINE"),
    (("window", "door"), "WINDOW"),
    (("flood", "wash", "up light", "uplight"), "FLOOD"),
    (("face", "sing", "carol", "mouth"), "SINGING_FACE"),
    (("sign", "tune"), "SIGN"),
    (("drive", "walk", "path", "fence", "yard line"), "PATH"),
    (("flake",), "SNOWFLAKE"),
]

#  Step 4 — group hints: an unresolved prop inherits the role its user group's
#  name matches under the same heuristics (e.g. member of "All Outline" → OUTLINE).

def classify(props: list[Prop]) -> ClassifyResult:
    """Steps 1–4 in order; sets role + confidence per prop; returns the resolved
    props plus the unresolved tail for step 5. Confidence: step 1–2 → 1.0,
    step 3 → 0.9, step 4 → 0.85, unresolved → CUSTOM_PROP @ 0.5."""

Capability classes (spec §2) are derived from role + geometry, replacing today's implicit assumption that names encode capability:

def capability(role: str, nodes: int, string_type: str) -> str:
    # MATRIX → 2D_SURFACE;  MEGA_TREE/SPINNER/STAR → 2D_RADIAL;
    # OUTLINE/PATH → LINEAR_HIGH if nodes ≥ 100 else LINEAR_LOW;
    # ARCH/CANE/ICICLES/SNOWFLAKE/WINDOW/MINI_TREE → LINEAR_LOW;
    # FLOOD/CUSTOM_PROP low-count → POINT;  SINGING_FACE → SPECIAL;
    # spec §8: non-RGB StringType → POINT override regardless of role;
    # spec §8: very large dense Custom (node density + area) → MATRIX/2D_SURFACE.

Step 5 — LLM fallback (new agent, orchestrator side, so xlights-core keeps zero LLM deps). One batched call for the whole unresolved tail, not one call per prop — these are compact records and a single worker-tier request:

# agents/layout_classifier.py  (proposed)
class PropRoleGuess(BaseModel):
    name: str
    role: Literal["OUTLINE","WINDOW","MEGA_TREE","MINI_TREE","MATRIX","ARCH",
                  "SPINNER","STAR","ICICLES","CANE","SNOWFLAKE","FLOOD",
                  "SINGING_FACE","SIGN","PATH","CUSTOM_PROP"]
    confidence: float          # 0..1
    rationale: str = ""

class PropRoleGuesses(BaseModel):
    guesses: list[PropRoleGuess] = []

def layout_classifier_agent():
    return build_agent("classifier", output_type=PropRoleGuesses, system_prompt=_PROMPT)

def render_input(unresolved: list[Prop]) -> str:
    # per prop, the spec §3.5 compact record:
    # {"name": ..., "DisplayAs": ..., "nodes": ..., "width_m": ..., "height_m": ...,
    #  "groups": [...]}

A new classifier role goes into models/config.yaml routed to the worker tier (same as analyst/generator). The enum-constrained output_type makes an invented role a schema failure, not a silent bad group. Per spec §3.5/§6, any guess with confidence < 0.8 — and every prop the LLM didn't return — goes to the review queue, never silently into a group. The fallback is optional: with no LLM key, init-layout still runs fully deterministic (steps 1–4) with the tail defaulting to CUSTOM_PROP + review, matching the archived change's v1 posture.

3.2 Spatial derivation — spec §4 (derive_spatial(), same module)

@dataclass
class SpatialSummary:
    width_units: float          # bbox extent in world units (spec's width_m if units are m)
    focal_x: float              # normalized x of the focal center
    symmetric: bool             # ≥1 mirror pair found
    excluded: list[str]         # parked/outlier models (→ review, spec §8)

def derive_spatial(props: list[Prop], *, invert_x: bool = False) -> SpatialSummary:
  1. Outlier exclusion first (spec §8, archived-change decision): models positioned > 2× the display span outside the main bounding box, or with zero nodes, are excluded from the bbox and appended to review — the X = −907 parked model must not stretch normalization.
  2. Normalize: p.x = (wx - min_x) / (max_x - min_x), p.y likewise from WorldPosY, ground → top. invert_x flips the x axis — this is the knob the §7 sweep test turns when a mirrored preview is detected (spec §8 "inverted previews").
  3. Bands: GROUND/MID/ROOF at y cuts 0.33/0.66; optionally adjusted upward if the OUTLINE models' minimum y clearly defines a roofline above 0.66 (spec §4.2 "adjusted if the outline models clearly define a roofline").
  4. Sides: LEFT x<0.45, CENTER 0.45–0.55, RIGHT >0.55 (spec §5.4), filling the existing Prop.side field.
  5. Sweep order: within each multi-instance role in {ARCH, MINI_TREE, CANE, WINDOW, SNOWFLAKE, SPINNER}, sort by x, assign sweep_order = 1..N — exactly what build_sem_groups already expects for its _LTR groups (line 60–63).
  6. Mirror pairs: same role, |x₁ + x₂ − 1| ≤ 0.05 (mirrored around the centerline) and |y₁ − y₂| ≤ 0.05 → set mirror_of on both props (spec §4.4 "record both directions").
  7. Center distance: normalized Euclidean distance from the focal center — the MEGA_TREE if present, else bbox center — into Prop.center_dist.
  8. Focal flags: MEGA_TREE, MATRIX, and any prop whose approximate visual area (scale-derived bbox from ScaleX/Y × model extent, the same inputs model_world_pixels() uses) exceeds ~15% of the display area.

3.3 The SEM_ group writer — spec §5 + §5.7 (write_sem_groups(), extends layout_semantics.py)

This is the piece with no precedent in the package: creating <modelGroup> elements rather than patching attributes on existing ones. It extends the exact pattern of patch_sem_gridsize/patch_view (ET parse → mutate → tmp + os.replace), plus the archived change's safety decisions (timestamped backup; xLights must be closed):

# knowledge/layout_semantics.py  (proposed addition)

LAYOUT_MODE_ORDERED  = "Horizontal Per Model"   # SEM_*_LTR — chases traverse member order
LAYOUT_MODE_ENSEMBLE = "Per Preview"            # SEM_ALL / SEM_BAND_* / ensembles — spatial map

def layout_modes(groups: dict[str, list[str]]) -> dict[str, str]:
    """Spec §5.7: name → mode. _LTR → ordered mode; everything else → ensemble mode."""

@dataclass
class WriteReport:
    created: list[str]; replaced: list[str]; kept_user_groups: int; backup: Path | None

def write_sem_groups(rgb_path, groups: dict[str, list[str]], *,
                     modes: dict[str, str] | None = None,
                     grid_size: int = SEM_GRID_SIZE,
                     backup: bool = True) -> WriteReport:
    """Idempotently (re)create the SEM_ modelGroups:
      1. timestamped backup rgbeffects.<ts>.bak (unless backup=False)
      2. remove every existing modelGroup whose name matches ^SEM_   (regenerable,
         spec §6: 'rerunning deletes and recreates all SEM_ groups, touching nothing else')
      3. append one <modelGroup> per plan entry:
         name=..., models=",".join(members), LayoutGroup="Default",
         GridSize=str(grid_size), + the §5.7 layout-mode attribute
      4. atomic tmp + os.replace
    NEVER touches non-SEM_ groups (spec §5.6: do not delete/rename the user's groups)."""

The layout-mode attribute is the one deliberately unresolved fact in this design. xLights stores the group dialog's "Preview/Buffer Layout" choice as a modelGroup XML attribute, but the exact attribute name and the serialized strings for "Per Preview" vs "Horizontal Per Model" must be established by round-tripping, not guessed: create one group in each mode in the target xLights build's UI, save, and diff the XML (implementation plan step 3.1). The doc pins the behavioral requirement — ordered groups must traverse members in order, ensemble groups must map effects across the display spatially — and the writer takes the mode strings from one confirmed constant pair. Note this is a different setting from the per-effect B_CHOICE_BufferStyle already handled in pipeline/render_style.py; the group-level default matters because it's what an effect gets when the buffer style is "Default" (the archived add-metric-rhythm design found the _LTR groups "don't render a chase" precisely because their group mode was never set).

xLights-must-be-closed guard. xLights rewrites rgbeffects.xml from memory on exit, silently clobbering offline edits. The guard is a cheap connectivity probe: attempt XLightsClient.get_version() with a short timeout; if it answers, refuse to write and print "close xLights, then re-run" (or in the guided flow: wait-and-poll with a countdown). This inverts the pipeline's usual expectation (xLights running), so init-layout must not share _run()'s async with XLightsClient() assumption.

Two known ET caveats, accepted deliberately: xml.etree.ElementTree does not preserve comments or attribute order. This is already the shipped behavior of patch_view/patch_sem_gridsize on the same file, and xLights regenerates the file itself anyway; the timestamped backup covers the residual risk.

Cache interaction: rewriting groups changes the group-name set, which changes pipeline/groups.py::_fingerprint — the targetable-groups probe re-runs automatically on the next pipeline run. init-layout should say so in its summary ("first run will re-probe group targetability").

Written to the show directory (spec §6: "the only layout representation downstream LLM planners receive"), kept under ~10 KB, plus a copy under the cache root so hermetic runs and the dashboard can read it without the show folder mounted. Consumption is three independent, individually landable integrations:

(a) Director traits/props block. A compact renderer, ~1 KB of prompt:

# agents/director.py  (proposed)
def render_layout_block(manifest: LayoutManifest, groups: list[str]) -> str:
    """Per targetable SEM_ group: role, member count, ~nodes, band, symmetry/order.
    e.g.  SEM_ARCHES: 6 arches, ~50 nodes each, ground band, mirror-symmetric,
          ordered variant SEM_ARCHES_LTR (left→right)
          SEM_FOCAL: MegaTree (1600 nodes, 2D radial) + Matrix_P10 (2D surface)
    Plus one display line: 'display ~18m wide, focal center x=0.50, symmetric'."""

def render_input(brief, groups, placeable_types,
                 manifest: LayoutManifest | None = None) -> str:
    # unchanged flat AVAILABLE GROUPS list (the hard constraint the plan is
    # validated against), plus, when a manifest exists:
    #   "\n\nLAYOUT (what each group physically is — ground your looks in this):\n"
    #   + render_layout_block(manifest, groups)

The flat list stays (it is the authoritative "choose targets only from these" constraint fed from the live probe at pipeline/run.py:414); the block adds the grounding. Wired in run.py behind load_manifest() — absent manifest, byte-identical prompt to today, so cached briefs stay valid.

(b) QA capability gating. Replace the name-prefix proxy in qa/rules.py with manifest-derived capability classes, keeping the prefix table as the no-manifest fallback:

# qa/rules.py  (proposed)
def _target_res(target: str, manifest) -> set[str]:
    """res classes of the target: the prop's own class, or the union of member
    classes for a group. Cached per manifest."""

def _is_linear(target: str, manifest=None) -> bool:
    if manifest and (res := _target_res(target, manifest)):
        return res <= {"LINEAR_HIGH", "LINEAR_LOW"}      # ALL members linear
    return target.startswith(_LINEAR_PREFIXES)            # legacy fallback

This also unlocks rules the prefix hack can't express: texture effects legal on a mixed group whose canvas is dominated by a 2D surface; POINT-only groups rejecting anything but washes; SPECIAL (faces) excluded from general placement — enforcement moves from "what the group is named" to "what the pixels are". evaluate(instructions, plan) grows an optional manifest=None keyword; existing tests pass unchanged.

(c) Choreography data. Mirror pairs and sweep orders feed true call-and-response (left prop ↔ its recorded mirror, not just SEM_SIDE_LEFT/RIGHT) and center-out waves ordered by center_dist. First consumer: the weave's counterphase/directional sweeps and pipeline/triggers.py group rotation; the seam is that both currently take group name lists, so the manifest only needs to order those lists.

3.5 Derived choreography vocabulary — constants as fallback

Today semantic_groups.py hardcodes this layout's judgment calls. The generalization is a small pure function over the manifest, with today's tuples as both the tie-break prior and the no-manifest fallback:

# pipeline/semantic_groups.py  (proposed addition)
@dataclass(frozen=True)
class ChoreoVocabulary:
    metric_ring: tuple[str, ...]
    backbeat_preference: tuple[str, ...]
    bed_preference: tuple[str, ...]
    peak_broad: tuple[str, ...]
    accent_groups: tuple[str, ...]
    hero_group: str
    bass_band_group: str

DEFAULT_VOCAB = ChoreoVocabulary(METRIC_RING, BACKBEAT_GROUP_PREFERENCE, ...)  # today's constants

def derive_vocabulary(manifest: LayoutManifest | None) -> ChoreoVocabulary:
    if manifest is None:
        return DEFAULT_VOCAB
    # RING: rank the rhythm-family role groups present (ARCH, CANE, MINI_TREE,
    #   SNOWFLAKE, SPINNER, WINDOW) by score = f(instance count, x-spread, node budget):
    #   count → a walkable family needs ≥2 members; spread → the walk should traverse
    #   the yard; node budget → readable at distance. Take the top 4.
    # BACKBEAT: SEM_SIDE_* when ≥2 sides are populated (spatial contrast beats family
    #   contrast), else the point/accent families not consumed by the ring.
    # BED: SEM_BAND_GROUND if the ground band holds ≥~25% of props, else SEM_ALL;
    #   PEAK_BROAD is the reverse (broadest first), preserving today's deliberate asymmetry.
    # HERO: SEM_FOCAL if any focal prop, else the largest 2D_RADIAL/2D_SURFACE prop.

Consumers change minimally: beats.py::select_rhythm_groups(section, available_groups) gains a vocab: ChoreoVocabulary = DEFAULT_VOCAB parameter (its internal references to METRIC_RING/BACKBEAT_GROUP_PREFERENCE/etc. switch to vocab.*), and weave.py:398's BED_PREFERENCE lookup does the same. run.py computes the vocabulary once per run from the loaded manifest. Every existing if g in avail guard stays — the vocabulary proposes, the live probe disposes. The decision the roadmap asked for lands here: a layout with no arches gets its beat anchor by ranking, not by accident of tuple order.

3.6 The guided CLI flow

xlo init-layout [--show-folder PATH] [--dry-run] [--yes] [--no-validate]
                [--no-llm] [--invert-x] [--review-web]
  1. Locate. --show-folder, else ask a running xLights via get_show_folder() (and remember it for the write step's close-xLights guard), else prompt.
  2. Analyze. parse_props → classify → LLM fallback (skipped with --no-llm or no key) → derive_spatial. Print the classification table: name, DisplayAs, nodes, role, res, band/side, confidence, source step.
  3. Review. For each prop with confidence < 0.8 (and each excluded outlier): a numbered terminal prompt offering the role enum, "accept as CUSTOM_PROP", or "exclude". --yes accepts all suggestions unattended (they still land in the manifest's review array so the state is visible; spec §7.4 requires resolution before the manifest is "final" — an unreviewed manifest is marked "review": [...] non-empty and the CLI exits with a warning).
    Terminal vs browser: the brief_editor.py stdlib-http.server pattern (line 205 serve()) was weighed and deferred: the review payload is a short list of enum choices with no visual assets to show (the validation render doesn't exist yet at this stage), so a numbered prompt is strictly simpler; the browser form wins only when we can show each prop highlighted in a preview render, which is a natural v2 once the validator exists (--review-web reserved for it).
  4. Plan + diff. build_sem_groups(props) → diff against the SEM_ groups currently in the file (see §3.8 migration). --dry-run stops here, printing the diff and the would-be manifest.
  5. Write. Enforce the closed guard → write_sem_groups + patch_view → emit_manifest. Print the backup path.
  6. Validate. §3.7 below (skippable via --no-validate); on a sweep-test failure, offer to re-run with --invert-x automatically (spec §7.3's prescribed fix).
  7. Finish. "Restart xLights to load the groups; the first xlo run will re-probe targetability."

CLI wiring is the existing pattern in cli.py::main(): a new sub.add_parser("init-layout", ...) beside run/regen/edit-brief, dispatching to pipeline/init_layout.py::run_init_layout(args). Unlike run, it must not require an LLM key (deterministic path) and must not require xLights to be running.

3.7 Automated §7 validation — offline, deterministic-first

The key realization: validation needs no xLights at all. PreviewRenderer(fseq, rgbeffects, networks) renders any channel data against the layout, and we know each model's absolute channel range from parse_controllers() + resolve_start_channel() (preview/layout.py:57-92). So the validator synthesizes the test frames' channel data directly instead of asking xLights to render a test sequence:

# pipeline/layout_validate.py  (proposed)

    """DETERMINISTIC sweep check: per frame, the lit-pixel world-x centroid

Split of labor, following the repo's judgment-vs-realization convention:

  • Deterministic (always): the sweep-order centroid check — pure geometry, exact, enters the pass/fail gate. Also cheap structural checks: every SEM_ member exists as a model; no SEM_ group is empty; SEM_ALL excludes SINGING_FACE/SIGN; every _LTR member order matches sweep_order.
  • Vision (optional, LLM key present): send the role-color contact sheet to the visual critic (agents/visual_critic.py — multimodal, advisory) with the question spec §7.2 poses: "does each colored region correspond to its labeled role?" plus the known common failures (mega vs mini tree confusion, outline-as-PATH). Findings are advisory and printed; they flag, never auto-mutate.
  • Human (always available): the contact sheet on disk; the CLI prints its path.

Caveat, inherited from the roadmap's I8 discussion: offline-render fidelity is approximate (that's why RealRender exists). For role-color/sweep purposes — "which pixels lit, where" — the offline renderer is exact by construction (it maps channels to world positions deterministically); fidelity concerns apply to effect appearance, which validation doesn't judge. One live render remains the ground-truth gate at the end of the first real run, unchanged.

3.8 Migration: the existing hand-built layout must converge

The first user of init-layout is the current layout, which already has hand-created SEM_ groups the whole pipeline depends on. Requirement: running xlo init-layout --dry-run against it produces a plan whose group membership matches the file byte-for-byte or with explained diffs only — misclassification would show up as a membership diff, making the dry-run diff double as the classifier's acceptance test on real data. Mechanics:

  • --dry-run renders a three-way diff per group: only-in-file / only-in-plan / member-order-changed.
  • Convergence fixture: a sanitized copy of the real rgbeffects.xml (names kept, any personal metadata stripped) checked in under tests/fixtures/, with a golden manifest — the same golden-snapshot discipline as test_golden_pipeline.py.
  • Divergences found during bring-up are resolved by fixing the classifier or recording an explicit per-layout override, never by weakening the diff. Overrides live in the manifest's review resolutions (a small layout_overrides.json beside the manifest: {"prop_name": {"role": ...}}, applied after step 4, before the LLM step).
  • A converged re-run is a no-op write: write_sem_groups compares the serialized SEM_ subtree before replacing and skips the file write (and the backup) when identical — same idempotency contract as patch_sem_gridsize returning 0.
  • One accepted divergence to resolve at bring-up: the file may contain hand-set attributes on SEM_ groups the plan doesn't know about. Policy: the writer owns SEM_ groups entirely (spec §6 regenerability); anything worth keeping must become part of the plan (e.g. grid size, layout mode) — discovered extras are printed in the diff so nothing silently vanishes.

5. Testing strategy

Hermetic-first, per repo convention (pytest = no xLights, no network; pytest -m live opt-in).

  • Fixture rgbeffects files (small, synthetic, committed):
    • layout_basic.xml — mega tree (Tree 360, 1600 nodes), 6 arches at mirrored x positions, 4 canes, outline Single Lines named Roof Left/Gutter R, a Window Frame, a Custom named Snowflake 1: exercises steps 1–3, bands, sweep order, mirror pairs, focal flags.
    • layout_tricky.xml — sole 400-node tree (sole-largest → MEGA_TREE), a parked model at X = −900 (outlier → review + excluded), a dense whole-house Custom mesh (→ MATRIX by density), a non-RGB StringType prop (→ POINT override), a model in a non-Default LayoutGroup (excluded per §8), a group named "All Outline" hinting an unnamed Custom.
    • layout_real.xml — the sanitized copy of the user's actual layout (81 models) + its golden manifest: the convergence test.
    • layout_modes_roundtrip.xml — the slice-3 spike artifact: groups saved by real xLights in both §5.7 modes, asserting our writer emits byte-compatible attributes.
  • Classifier tests — table-driven: every DisplayAs mapping; tree threshold at 599/600/601; each name-heuristic family; group-hint inheritance; unresolved → CUSTOM_PROP @ 0.5 → review; capability matrix incl. the LINEAR_HIGH/LOW node cut and the StringType override.
  • Spatial tests — normalization with and without outliers; band cuts; side cuts at 0.449/0.451; sweep order; mirror tolerance edges (0.049/0.051); center_dist with/without a mega tree; invert_x flips sweep order and mirror pairs consistently.
  • Writer tests — write → re-parse → equal plan; re-run → no-op (mtime unchanged); user groups and views untouched; backup created once per real write; SEM_ groups fully replaced (a stale SEM_OLD disappears); GridSize + mode attributes present; output re-parses under parse_models() (the preview parser as a cross-check consumer).
  • Manifest tests — schema round-trip; <0.8 confidence → review; <10 KB on the real fixture; load_manifest returns None on absence and on version-mismatch (forward-compat guard).
  • Validation tests — write_fseq_v2_uncompressed → load_fseq round-trip; sweep centroid check passes on layout_basic, fails (and recommends invert) on the same layout with x negated; role frames light exactly the member channels; the contact sheet renders (skip if [preview] extras absent, mirroring test_preview.py).
  • Consumption tests — Director: render_input without manifest is byte-identical to today (snapshot); with manifest contains the expected group lines and stays under a size budget. QA: manifest-gated _is_linear flags a texture on a linear-membered group with a non-SEM name, and un-flags a matrix-dominated group that the prefix rule would wrongly pass. Vocabulary: derive_vocabulary(None) == DEFAULT_VOCAB; an arch-less synthetic manifest ranks canes/minis into the ring; golden pipeline snapshot unchanged on the real-layout manifest.
  • CLI tests — argument wiring (also an I6 item), review-queue answer parsing, and --dry-run on the convergence fixture printing an empty diff, driven through main(argv) with a stubbed flow.
  • Live (opt-in, -m live) — against a real xLights + a scratch show folder: write groups with xLights closed, start xLights, confirm get_group_names() contains the SEM_ set, run targetable_groups, place one effect per _LTR group and export a preview. Perceptual sweep direction gets one human watch, per the repo's live-verify convention.

7. Acceptance criteria

  • ☐ xlo init-layout --dry-run on the current real layout prints an empty (or explicitly accepted) membership diff — the convergence/no-op requirement.
  • ☐ xlo init-layout on a fresh fixture show folder produces: SEM_ groups in rgbeffects.xml with GridSize and correct §5.7 layout modes, a "SEM Master" view, and a valid layout_semantics.json < 10 KB — with xLights closed, atomically, with a backup.
  • ☐ Re-running the command is a byte-level no-op on an unchanged layout (no new backup, no write).
  • ☐ Every spec §3 step is a tested code path; props resolved below 0.8 confidence appear in the review queue and the manifest's review array, never silently in groups.
  • ☐ Spatial fields (band, side, sweep_order, mirror_of, center_dist, focal) populate correctly on the fixtures, including outlier exclusion and --invert-x.
  • ☐ The deterministic sweep validation fails on an x-inverted fixture and passes after --invert-x; the role-color contact sheet is written and (with a key) critiqued.
  • ☐ The Director prompt contains the layout block when a manifest exists and is byte-identical to today's prompt when it doesn't; cached briefs remain loadable.
  • ☐ qa/rules.py gates texture-on-linear via manifest capability classes when available, prefix fallback otherwise; existing QA tests unmodified and green.
  • ☐ derive_vocabulary() reproduces today's constants from the real-layout manifest (golden pipeline snapshot unchanged) and produces a sane ring on an arch-less synthetic layout.
  • ☐ Deterministic classifier path works with no LLM key configured.
  • ☐ docs/usage.md's "One-time layout setup" section describes the command instead of the manual procedure.
  • ☐ Live-verified once: groups load after xLights restart, are probe-targetable, and a generated show runs against them.

8. Open questions

  1. Layout-mode serialization — the exact modelGroup attribute name/values for "Per Preview" vs "Horizontal Per Model" in the target xLights build. Resolved by the slice-3.1 round-trip spike before any writer code merges (deliberately not guessed here).
  2. Manifest home — spec §6 says the show directory; the pipeline caches under data/analyses/. Proposal: show dir is canonical, cache copy for hermetic consumers — confirm the copy doesn't create a staleness trap (mtime check on load?).
  3. SEM_ALL_LESS_FOCAL* vs the spec — the subtractive ensembles exist in code but not in spec §5.5. Proposal: treat code as the spec's superset and amend the spec doc; the writer emits them (it already must, for convergence).
  4. Numbered-group removal — the 2026-06-10 change deleted the user's 47 0[1-8]_ groups as a one-time, explicitly-consented cleanup. init-layout must not generalize that (spec §5.6 forbids touching user groups). Is a separate --prune-groups REGEX opt-in worth having, or leave cleanup manual?
  5. Multiple previews / layoutGroups — spec §8 says default preview only. Is a --layout-group NAME escape hatch needed at v1, or defer until a real second-preview user appears?
  6. Roofline-adjusted band cuts — implement the §4.2 adjustment at v1 or ship fixed 0.33/0.66 cuts and revisit when a layout misbands? (Leaning: fixed cuts + a manifest field recording them, so the adjustment is a data change later.)
  7. Where does the Director block stop? ~1 KB of traits is the roadmap's number; should group_motifs-style per-prop detail (colors that read well on sparse props, e.g.) ever ride along, or is that the guides' job? (Leaning: traits only; guides own craft.)
  8. Review UX v2 — browser review with per-prop preview-render highlights once the validator exists (--review-web), reusing the brief_editor.py pattern. Worth scheduling with F-I (live progress UI) which extends the same pattern?
