1. Why

A full pipeline run makes on the order of 10–20 LLM requests (3–4 analysts, synthesizer, director, one generator call per section, then judge + visual critic + regens per refine iteration) and hundreds of xLights REST calls (one addEffect per placed effect via effect_emitter.apply_instructions, plus probe/save/render/export calls). Today a single transient failure at any of those points either degrades quality silently or kills the run:

  • A 429/529 on one panel analyst drops that analyst from the brief (agents/panel.py:142–145) — the synthesizer fuses the survivors and the show is built on a thinner brief, with only a single log.warning that doesn't even name which analyst died.
  • A 529 on the director or synthesizer propagates out of run_pipeline and aborts the run after audio analysis (and possibly stem separation) already spent minutes of work.
  • A momentary xLights hiccup (app busy during render, socket reset) during apply_instructions raises XLightsConnectionError/XLightsTimeout from client._request (client.py:90–97) and fails the whole emit stage — even though the very next request would have succeeded.

The only explicit retry in the entire codebase is lyricsgenius.Genius(token, timeout=timeout, retries=1, ...) in lyrics.py:60–62 — for the one integration that is genuinely optional. LLM calls rely on whatever PydanticAI's provider client defaults happen to be, and the xLights client has a typed error taxonomy and a write lock but zero backoff. This item adds one thin, shared retry primitive and applies it at three seams.

2. Current state

2.1 xLights client: typed errors, serialized writes, no retry

xlights_core/exceptions.py defines the taxonomy callers branch on:

XLightsError
├── XLightsConnectionError      # nothing listening / transport failure (client.py:92–97)
├── XLightsTimeout              # connect/read timeout (client.py:90)
├── XLightsNotImplemented       # HTTP 504 "not implemented" (client.py:118–122)
└── XLightsResponseError        # any other non-200; carries status_code + message
    ├── XLightsTargetMissing    # 503 "target element" on addEffect (client.py:124)
    └── XLightsUnsavedChanges   # 504 disambiguated by message (client.py:118)

XLightsClient._request (client.py:73) maps httpx.TimeoutException → XLightsTimeout, httpx.ConnectError → XLightsConnectionError, any other httpx.HTTPError → XLightsConnectionError. All mutations funnel through _mutate, which serializes on self._write_lock (an asyncio.Lock, client.py:58) with a generous write timeout (DEFAULT_WRITE_TIMEOUT = 300.0, line 35) because "render/check on a real (large) open sequence can take a while":

async def _mutate(self, cmd, params=None):
    async with self._write_lock:
        return await self._request(cmd, params=params, timeout=self._write_timeout)

Note xLights' quirk: it reports operational errors as 503 (busy, "Unknown model.", "No sequence open.") and overloads 504 for both not-implemented and unsaved-changes — so "HTTP 5xx ⇒ retryable", the usual web heuristic, is wrong here. The 5xx codes carry semantic, non-transient meanings.

2.2 LLM calls: PydanticAI defaults only

models/registry.py::build_agent constructs a plain Agent(model_string, output_type=..., system_prompt=..., model_settings=...) — no retry configuration. The installed PydanticAI (2.5.0) raises, from pydantic_ai.exceptions (verified in-venv):

  • ModelHTTPError(status_code, model_name, body) — provider HTTP errors (429 rate limit, 500, 529 overloaded…), subclass of ModelAPIError;
  • UnexpectedModelBehavior — the model exceeded output-validation retries, returned malformed structured output, etc.;
  • UsageLimitExceeded, ContentFilterError, FallbackExceptionGroup — policy/limit outcomes;
  • plus underlying httpx transport errors that can escape as connection failures.

PydanticAI's own retries setting governs output-validation re-prompting (asking the model to fix schema errors), not transport-level retries — a different mechanism from what this item adds, and the two must not be conflated: schema failures should never get a transport retry (same input ⇒ likely same failure, at full token cost).

2.3 The panel drop path

# agents/panel.py:130–145 (today)
async def _run(spec: AnalystSpec):
    async with sem:
        res = await spec.agent.run(spec.render(sa, lyrics))
        return spec.key, res.output

results = await asyncio.gather(*[_run(s) for s in analysts], return_exceptions=True)
for r in results:
    if isinstance(r, Exception):
        log.warning("analyst failed, dropping: %s", r)   # ← which analyst? unknown
        continue

Because gather(return_exceptions=True) returns the bare exception, the analyst's key is lost by the time the drop is logged — the operator can't tell whether structure or lyrics went missing. If all analysts fail, run_panel raises RuntimeError("all panel analysts failed...") (line 150) — correct, but today one burst of 429s across 3 concurrent analysts (they launch within the Semaphore(3) window) can produce exactly that.

2.4 The bulk-write path

effect_emitter.apply_instructions loops place_preset() per instruction, catching only the semantic skip set _SKIPPABLE = (PresetPlacementError, XLightsTargetMissing, ValueError, KnobValueError, KeyError) (line 16). Transport errors (XLightsConnectionError, XLightsTimeout) are deliberately not in that set — they abort the whole emit. That posture is right (a dead xLights shouldn't be 400 skipped effects) but brittle for a one-request blip.

3. Proposed design

3.1 Retryable vs. non-retryable — the classification table

DomainRetryNever retryRationale
LLM (PydanticAI)ModelHTTPError with status_code in (408, 429, 500, 502, 503, 529); httpx.TimeoutException/httpx.ConnectError/httpx.RemoteProtocolError escaping the provider clientUnexpectedModelBehavior (schema/validation — deterministic-ish, expensive), ModelHTTPError 400/401/403/404/413/422 (bad request/auth/model id), UsageLimitExceeded, ContentFilterError, pydantic.ValidationErrorRate-limit/overload/timeout are time-correlated and clear on their own; schema and auth failures repeat identically and re-bill full input tokens.
xLights RESTXLightsConnectionError, XLightsTimeout (reads); XLightsConnectionError only (mutations — see idempotency note)XLightsNotImplemented, XLightsTargetMissing, XLightsUnsavedChanges, XLightsResponseError generally (xLights' 503/504 are semantic, §2.1)Only transport-level failures are transient; every typed subclass encodes a state that retrying cannot change (and UnsavedChanges needs a decision, not a repeat).
Genius lyrics(unchanged — library-internal retries=1)—Already graceful: any failure returns None and the panel runs without the lyric analyst.

One deliberate judgment call: an XLightsResponseError whose message indicates busy-ness could arguably be retried, but the message strings are not a stable API; keep the first iteration conservative and extend the predicate only with observed, logged evidence (the I5 degradations summary will surface exactly which errors occur in practice).

3.2 The primitive (proposed: xlights_core/retry.py)

One thin async helper in xlights-core (the bottom of the dependency graph — core cannot import from the orchestrator, but the orchestrator can reuse core's helper for LLM calls). No tenacity dependency: the need is ~30 lines, the codebase style is stdlib-first, and tenacity's decorator indirection fights the injected-fake test pattern.

# PROPOSED — packages/xlights-core/src/xlights_core/retry.py
from __future__ import annotations
import asyncio, logging, random
from collections.abc import Awaitable, Callable
from typing import TypeVar

log = logging.getLogger(__name__)
T = TypeVar("T")

async def with_retry(
    fn: Callable[[], Awaitable[T]],
    *,
    retryable: Callable[[BaseException], bool],
    attempts: int = 3,                 # total tries, incl. the first
    base_delay: float = 1.0,           # first backoff, seconds
    factor: float = 2.0,               # exponential growth
    max_delay: float = 20.0,
    jitter: float = 0.5,               # ± fraction of the computed delay
    label: str = "",                   # for the log line ("judge", "addEffect", ...)
) -> T:
    """Run fn(); on a retryable exception sleep (exponential backoff, full jitter
    band) and re-run, up to `attempts` total. Non-retryable exceptions and the
    final failure propagate unchanged (same type — callers' except clauses keep
    working). Each retry logs at WARNING with the label, attempt count, and cause."""
    for attempt in range(1, attempts + 1):
        try:
            return await fn()
        except BaseException as exc:
            if attempt == attempts or not retryable(exc):
                raise
            delay = min(max_delay, base_delay * factor ** (attempt - 1))
            delay *= 1 + random.uniform(-jitter, jitter)
            log.warning("%s: transient failure (attempt %d/%d), retrying in %.1fs: %s",
                        label or getattr(fn, "__name__", "call"), attempt, attempts, delay, exc)
            await asyncio.sleep(delay)
    raise AssertionError("unreachable")

Shape notes:

  • Callable-wrapping, not decorating. Call sites do await with_retry(lambda: agent.run(prompt), retryable=llm_transient, label="judge"). This keeps the retry visible at the seam, composes with the I1 telemetry split (record() happens after the successful attempt; failed attempts' tokens are the provider's loss report, not ours), and doesn't wrap injected fakes invisibly.
  • Exception transparency. The original exception type propagates on exhaustion — panel.run_panel's isinstance(r, Exception), _refine_loop's best-effort except Exception blocks, and effect_emitter's _SKIPPABLE all keep working unchanged.
  • Full jitter (±50% band) because the panel launches 3 analysts near-simultaneously — synchronized retries would re-collide on the same rate-limit window.
  • Defaults 3/1s/2×/20s ⇒ worst-case added latency ≈ 1s + 2s ≈ 3–4.5s per call — negligible against LLM latencies and the 300s xLights write timeout.

Two predicate helpers live next to their taxonomies:

# PROPOSED — xlights_core/retry.py (transport predicate; core knows its own exceptions + httpx)
def xlights_transient(exc: BaseException) -> bool:
    return isinstance(exc, (XLightsConnectionError, XLightsTimeout))

# PROPOSED — xlights_orchestrator/models/registry.py (or a sibling llm_retry.py):
# the orchestrator owns the pydantic_ai dependency, so the LLM predicate lives here
_TRANSIENT_HTTP = {408, 429, 500, 502, 503, 529}
def llm_transient(exc: BaseException) -> bool:
    if isinstance(exc, ModelHTTPError):
        return exc.status_code in _TRANSIENT_HTTP
    return isinstance(exc, (httpx.TimeoutException, httpx.TransportError))

3.3 Where it wraps — agent runs

Rather than editing all seven LLM call sites (enumerated in the I1 doc §2.2), add one wrapper in models/registry.py and route the high-value, run-fatal sites through it:

# PROPOSED — models/registry.py
async def run_agent(agent, prompt, *, role: str, attempts: int = 3):
    """agent.run(prompt) with transient-only retry. Fakes without real transports
    never raise transient classes, so tests pass through on the first attempt."""
    return await with_retry(lambda: agent.run(prompt),
                            retryable=llm_transient, attempts=attempts, label=role)
Call siteTreatmentWhy
Director (pipeline/run.py:415), synthesizer (panel.py:158), generator (generate.py:158)run_agent(..., attempts=3)Failure aborts the run (or a whole section) — full backoff is worth it.
Panel analysts (panel.py:135)run_agent(..., attempts=2) — §3.5Roadmap: "retry once before being dropped"; the drop path remains the backstop.
Judge (run.py:229)run_agent(..., attempts=3)A judge failure currently escapes _refine_loop (it is not in a best-effort block) and kills the run mid-refine.
Visual critic (visual.py:94), section redesigner (run.py:163)run_agent(..., attempts=2)Both already sit under best-effort except Exception guards (run.py:227, 264) — one cheap retry converts "advisory capability silently lost this iteration" into "usually fine"; more would just slow the loop.

3.4 Where it wraps — xLights REST, and the mutation-lock interaction

Wrap at the transport boundary inside the client, not at call sites — every read and mutation funnels through two methods:

# client.py — PROPOSED shape
async def _request_with_retry(self, cmd, params=None, timeout=None, *, retryable):
    return await with_retry(lambda: self._request(cmd, params, timeout),
                            retryable=retryable, attempts=3, label=cmd)

# reads (get_models, get_model, get_show_folder, getOpenSequence, ...):
#   retryable = xlights_transient          (ConnectionError AND Timeout)
# mutations (_mutate):
async def _mutate(self, cmd, params=None):
    async with self._write_lock:
        return await self._request_with_retry(
            cmd, params, self._write_timeout,
            retryable=lambda e: isinstance(e, XLightsConnectionError))  # NOT Timeout

Three deliberate points:

  1. Retry happens inside the lock. The write lock exists because "xLights has one shared open sequence" (client.py:57) — interleaving another mutation between a failed attempt and its retry could change the open-sequence state the retried command assumes (e.g. a closeSequence sneaking between two addEffect attempts). Holding the lock across backoff (worst case ~4.5s extra) preserves the ordering invariant; other writers queue exactly as they do today behind a slow render.
  2. Mutations do not retry on timeout. XLightsTimeout after a mutation is ambiguous — the command may have been applied (e.g. an addEffect that landed but whose response was lost would, on retry, either double-place an effect on another layer via _free_layer stacking, or return worked=false). XLightsConnectionError from httpx.ConnectError means the TCP connection was never established, so the request provably did not reach xLights — always safe. (The catch-all "other transport failure → XLightsConnectionError" at client.py:96–97 should be split out as part of this work: re-raise send-failures distinctly, or exclude them from the mutation predicate — see Open questions #1.)
  3. render_all/export_video_preview are excluded from mutation retry entirely (long-running; a timeout usually means "still rendering", and re-issuing piles work onto the app). They keep today's behavior.

effect_emitter.apply_instructions and pipeline/groups._probe need no change: their per-call semantics (skip vs abort, XLightsTargetMissing as the only exclusion reason) sit above the now-self-healing transport.

3.5 Panel analysts: retry-before-drop, and name the body

# agents/panel.py — PROPOSED
async def _run(spec: AnalystSpec):
    async with sem:
        res = await run_agent(spec.agent, spec.render(sa, lyrics),
                              role=f"analyst:{spec.key}", attempts=2)
        return spec.key, res.output

results = await asyncio.gather(*[_run(s) for s in analysts], return_exceptions=True)
outputs: dict[str, object] = {}
for spec, r in zip(analysts, results):            # ← pair result with its spec: gather
    if isinstance(r, Exception):                  #    preserves input order, so zip is safe
        log.warning("analyst %r failed after retry, dropping: %s", spec.key, r)
        continue
    key, out = r
    outputs[key] = out

Changes: (a) one transient retry inside the semaphore slot before the analyst is dropped; (b) the drop log names which analyst was lost, at WARNING level (this is a whole-capability loss — exactly the I5 convention, and the event should also be reported to the I5 degradations collector once that lands: degradations.note("analyst:" + spec.key, ...)); (c) the retry stays inside the semaphore so concurrency never exceeds the free-tier cap of 3 even while retrying.

3.6 What stays untouched

  • lyrics.py — lyricsgenius(retries=1) already retries internally and the whole path degrades to None; adding our wrapper would double-retry. Leave as is; note the asymmetry in the module docstring.
  • PydanticAI's output-validation retries — untouched and orthogonal (§2.2).
  • Best-effort except Exception blocks — none get retries by default; they are the subject of I5, not I2.

4. Implementation plan

  1. xlights_core/retry.py: implement with_retry() + xlights_transient() as §3.2; export from xlights_core/__init__.py alongside the client/exceptions.
  2. XLightsClient: add _request_with_retry; route read commands through it with xlights_transient; change _mutate to the connection-only predicate inside the lock; exempt render_all/export_video_preview. Add a constructor knob retry_attempts: int = 3 (0/1 disables — tests that assert single-call behavior can opt out).
  3. Split transport mapping in _request (client.py:96): map generic httpx.HTTPError to a distinguishable state (e.g. keep XLightsConnectionError but set a request_sent: bool attribute, or introduce XLightsTransportError) so the mutation predicate can be precise about "provably never sent". Decision per Open question #1.
  4. models/registry.py: add llm_transient() and run_agent(); unit-test the predicate against constructed ModelHTTPErrors.
  5. Route the seven call sites through run_agent() with the attempt counts from §3.3 (this dovetails with I1's telemetry split at the same lines — if I1 lands first, run_agent is also the natural single home for telemetry.record(role, res), collapsing seven instrumentation points to one).
  6. Panel: apply §3.5 (zip specs with results; WARNING with analyst key).
  7. Docs: note retry behavior + env knob (if any; see Open question #3) in docs/usage.md; update openspec/specs/show-orchestration/xlights-sequence-editing deltas if the specs assert single-attempt semantics.

5. Testing strategy

Hermetic-first, per repo convention (-m "not live"; fake agents as in tests/test_refine.py; httpx.MockTransport-injected client as in tests/test_client.py — the client constructor already accepts an injected httpx.AsyncClient precisely for this, client.py:47–56).

  • tests/test_retry.py (new) — pure with_retry semantics with a counting fake and base_delay=0:
    • succeeds first try ⇒ one call, no sleep;
    • transient, transient, success ⇒ three calls, returns value;
    • non-retryable first try ⇒ one call, original exception type propagates;
    • exhaustion ⇒ attempts calls, last exception propagates unchanged;
    • backoff sequence monotone and capped (patch random.uniform and capture asyncio.sleep args via monkeypatch);
    • predicate tests: llm_transient(ModelHTTPError(429, "m")) true; 400 false; UnexpectedModelBehavior false; xlights_transient over the whole exception taxonomy.
  • tests/test_client.py (extend) — MockTransport handlers that fail N times then succeed:
    • read (getModels): httpx.ConnectError ×2 then 200 ⇒ result returned, 3 transport calls observed;
    • mutation (addEffect): ConnectError then 200 ⇒ placed once (assert the mock saw exactly one successful add);
    • mutation timeout ⇒ no retry: raise httpx.ReadTimeout once, assert XLightsTimeout after exactly one attempt;
    • 503 "Unknown model." ⇒ no retry, XLightsResponseError immediately;
    • lock-hold test: start a mutation that retries (with a short real delay); concurrently fire a second mutation; assert the second's transport call happens only after the first completes (ordering via a recorded call log) — pins the retry-inside-lock invariant.
  • tests/test_orchestrator.py / panel tests (extend) — fake analyst agent whose run() raises a transient ModelHTTPError(529, ...) once then succeeds ⇒ brief contains that analyst's output and no drop warning; always-failing analyst ⇒ dropped after exactly 2 calls with a WARNING naming its key (use caplog); all-analysts-fail ⇒ still RuntimeError.
  • Refine loop: fake judge raising 529-once ⇒ iteration completes (previously fatal); fake visual critique raising twice ⇒ findings skipped for the iteration, loop continues (unchanged outer behavior, now with one retry observed).
  • No-sleep guarantee for the suite: all retry tests pass base_delay=0 or monkeypatch asyncio.sleep; add a conftest guard if suite time regresses.
  • Live (-m live, optional): kill-and-restart xLights mid-get_models loop is not practically scriptable; instead a live test that points the client at a socket that drops the first connection (tiny local TCP stub) to exercise the real httpx path once.

6. Risks & mitigations

RiskImpactMitigation
Retrying a non-idempotent xLights mutation double-applies it (duplicate effect, double close)HighMutations retry only on ConnectError (request provably unsent); timeouts surface immediately; render_all/export excluded entirely.
Backoff inside the write lock stalls other writersLow–MediumBounded (≤ ~4.5s worst case, vs. the existing 300s write timeout); alternative (retry outside the lock) breaks ordering — documented trade-off, chosen deliberately.
Retries mask a persistently sick provider/xLights, tripling time-to-failureMediumAttempts hard-capped at 3; every retry logs WARNING with cause; the terminal failure is unchanged in type and content.
Rate-limit retry storms across concurrent analystsMediumFull jitter; retries stay inside the Semaphore(3); analysts get only 2 attempts.
PydanticAI exception surface shifts between versions (it did between 0.x and 2.x)MediumPredicate isolated in one function with direct unit tests constructing real exception instances; a rename fails a test, not a run.
Token cost of retried LLM callsLowOnly transient-class failures retry (typically billed zero or trivially by the provider); schema failures — the expensive repeat — are explicitly excluded. I1 telemetry counts only successful attempts' usage.
Hidden coupling: run_agent wrapper changes fake-agent call counts in existing testsLowFakes don't raise transient classes ⇒ single attempt, identical behavior; verified by running the full suite unmodified.

7. Acceptance criteria

  • ☐ xlights_core.retry.with_retry exists with exponential backoff + jitter, propagates original exception types, and is unit-tested for all four outcome shapes (§5).
  • ☐ A transient 429/529/timeout on director, synthesizer, generator, or judge no longer aborts a run on first occurrence (proven by fake-agent tests).
  • ☐ A panel analyst gets exactly one transient retry before being dropped, and the drop is logged at WARNING naming the analyst key.
  • ☐ xLights reads retry on connection error and timeout; mutations retry on connection error only, inside the write lock; XLightsTargetMissing/UnsavedChanges/NotImplemented/generic 503 never retry.
  • ☐ Ordering invariant pinned by test: no second mutation's request is sent between a first mutation's attempts.
  • ☐ Schema/validation failures (UnexpectedModelBehavior, ValidationError) are never transport-retried (test-pinned).
  • ☐ Full hermetic suite passes with no fixture changes and no measurable slowdown (retry tests use zero delays).
  • ☐ Every retry emits a WARNING log line with label, attempt count, and cause (feeds the I5 degradations summary later).

8. Open questions

  1. The catch-all transport mapping (client.py:96: any httpx.HTTPError → XLightsConnectionError) conflates "never sent" with "sent, response lost". Split into a new subclass (cleaner, but a public-taxonomy addition) or a boolean attribute on XLightsConnectionError? Leaning: new XLightsTransportError(XLightsConnectionError) for the sent-but-failed case so existing except XLightsConnectionError callers keep catching both.
  2. Should the generator retry budget differ during refine? A regen failure inside _refine_loop currently propagates out of _regen and kills the run (it is not in a best-effort block, run.py:266). With retry it's rarer; do we additionally want I5-style skip-this-revision handling there, or is fail-fast correct for a corrupted-state risk? (Fail-fast is current behavior; keep, revisit with I5.)
  3. Env knobs — expose XLO_RETRY_ATTEMPTS/XLO_RETRY_BASE like XLO_REFINE_SKIP_OBJECTIVE, or keep constants until a need appears? Leaning constants-first (roadmap R3/R4 is still consolidating existing knobs).
  4. Anthropic retry-after header — ModelHTTPError.body may carry it; honoring it beats blind exponential backoff for 429s. Worth it in v1, or note as an extension once telemetry shows 429s actually occur?
  5. Should save_sequence during _report/RealRender.refresh retry? They sit in best-effort blocks (run.py:174–179, visual.py:63–66) where failure only degrades sampling — retry adds latency per iteration for an advisory capability. Proposal: inherit the mutation policy (connection-only) and measure.
