1. Why

An attended xlo run --refine is a 10–30 minute pipeline whose only live surface is stdout. Worse, in attended mode the run stops and holds the terminal hostage at blocking input() prompts. The two stage-gate checkpoints, in pipeline/run.py:

# run.py:110-116 — the interpret review gate (input() at line 115)
async def _interpret_review(desc_md: str, brief) -> bool:
    """Attended interpret gate: show the song description; approve to proceed, abort to stop
    (hand-edit description.md / adjust and re-run). Returns True to continue."""
    print("\n" + "=" * 70 + "\n" + desc_md + "\n" + "=" * 70)
    print("Review the song description above (also written to description.md).")
    ans = input("Proceed to creative direction? [Y/n] (n = stop so you can correct) ").strip().lower()
    return not ans.startswith("n")

# run.py:102-107 — the design review gate (input() at line 106)
async def _design_review(brief_md: str, plan) -> bool:
    """Attended design gate: show the creative brief; approve to generate, abort to correct."""
    print("\n" + "=" * 70 + "\n" + brief_md + "\n" + "=" * 70)
    print("Review the creative brief above (also written to creative_brief.md).")
    ans = input("Proceed to effect generation? [Y/n] (n = stop so you can correct) ").strip().lower()
    return not ans.startswith("n")

Two more input() sites share the same posture: the per-iteration refine checkpoint _interactive_checkpoint (run.py:88–99, input() at line 96) and _final_approval (run.py:119–121, input() at line 120). All four calls are synchronous input() inside async functions — the whole event loop is parked, which is tolerable only because nothing else is scheduled. Meanwhile the operator has no view of stage progress, per-section generation, QA scores per iteration, or refine-loop decisions (revert / stall / plateau / skip-gate) except scrolling log lines.

The project already solved the "stdlib browser surface" problem once: brief_editor.py serves a schema-driven brief form on localhost with zero dependencies. F-I extends that exact pattern to (a) a live pipeline view and (b) browser-based approve/edit checkpoints, so an attended run is something you glance at in a tab rather than babysit in a terminal.

2. Current state

2.1 The brief editor — the stdlib-browser pattern to extend

brief_editor.py (222 lines) is stdlib-only by design (module docstring: "Stdlib only (http.server); no framework, no CDN"). Its shape:

  • Page as a Python string — _PAGE (lines 28–147) is one raw-string HTML document with inline CSS and inline vanilla JS; render_page() (lines 150–156) substitutes __BRIEF__/__SCHEMA__/__COLORS__/__PATH__ placeholders with json.dumps/html.escape output.
  • Handler factory — _handler(brief_path, schema, colors) (lines 172–202) closes over its state and returns a BaseHTTPRequestHandler subclass: do_GET serves the page at /, do_POST handles /save by reading Content-Length bytes, validating via save_brief() (lines 159–169, which runs ShowPlan.model_validate and writes atomically via a .json.tmp rename), and replying JSON. log_message is overridden to silence request logging (line 174).
  • Server + browser — serve() (lines 205–222) binds ThreadingHTTPServer(("127.0.0.1", port), …) with port=0 by default (OS-assigned; the real port is read back from httpd.server_address[1], line 213), opens the browser after 0.4 s via threading.Timer(0.4, lambda: webbrowser.open(url)) (line 216), then blocks in httpd.serve_forever() until Ctrl+C.

Note what it does not do: it never returns edits to a running pipeline. It is a standalone xlo edit-brief command (cli.py:52–64); Save writes creative_brief.json back to disk, and the pipeline consumes the edit on the next run through the stage cache (run.py:410–411 loads ShowPlan.model_validate_json(sp_cache.read_text()); run.py:420 even logs "edit + re-run to apply"). F-I closes that loop for a live run.

2.2 The pipeline stages and checkpoint wiring

run_pipeline() (run.py:320–486) is a sequential stage skeleton: analyze (lines 350–377, incl. lyric attach and instrumental refine), groups probe (line 386), interpret (lines 391–402, cached panel → description.md), design (lines 409–418, cached Director → creative_brief.md + editable JSON), generate (lines 426–433), apply/render (lines 437–446), refine (lines 449–477, opt-in), finalize (lines 479–485). The checkpoints are injected callables with None meaning "no gate":

    if not await interpret_checkpoint(desc_md, st.music_brief):

and identically for the design gate at run.py:419–423. The refine loop _refine_loop (run.py:135–305) resolves its per-iteration checkpoint as decide = checkpoint or _interactive_checkpoint (line 143), and carries the closures _regen (148), _redesign (156), _report (172 — flush-save, real-render refresh, QA evaluate), _obj (186), and _record (201 — the revision-log tap). _auto_checkpoint (run.py:81–85) takes the Judge's verdict verbatim for unattended runs.

2.3 What --auto gates

--auto is declared at cli.py:80 ("unattended refine (no human checkpoints)") and read in exactly three places when threading into run_pipeline:

So --auto means: no interpret gate, no design gate, and Judge-verbatim refine decisions. The final-save prompt is also implicitly gated: run.py:481 asks _final_approval only when checkpoint is None and refine — i.e. attended refine runs. These four injection points are precisely where the browser checkpoints slot in without touching --auto semantics.

2.4 What the UI would display

qa/__init__.py::evaluate (lines 16–35) returns a QAReport (refine.py:21–26): objective_score, advisory_score, findings: list[Finding], and subscores (sync/placement/rules/variety, plus coverage when a sampler exists). Per refine iteration, _record (run.py:201–209) already serializes a RevisionLogRecord (revision_log.py:33–51) with iteration, scores, findings, judge verdict, revisions, regenerated sections, obj_before/obj_after/obj_delta, reverted, and human_decision — the revision-log tail is effectively the event stream F-I needs, just currently written only to revision_log.jsonl/.md (revision_log.py:106–113).

3. Proposed design

Everything below is proposed — none of these classes exist today. Constraint carried over from brief_editor.py: stdlib only (http.server, threading, queue, json, webbrowser), no web framework, no CDN; the served page may use inline vanilla JS and inline SVG, but no external resources.

3.1 Event model — ProgressBus (proposed, new module progress.py)

# Proposed: packages/xlights-orchestrator/src/xlights_orchestrator/progress.py
@dataclass(frozen=True)
class ProgressEvent:
    seq: int                    # monotonic, assigned by the bus (SSE id: + reconnect replay)
    ts: str                     # UTC isoformat
    type: str                   # "stage" | "section" | "score" | "refine" | "checkpoint"
                                # | "checkpoint_resolved" | "log" | "done"
    stage: str = ""             # "analyze" | "interpret" | "design" | "generate" | "apply"
                                # | "refine" | "finalize"
    section: int | None = None  # section index for per-section events
    payload: dict = field(default_factory=dict)   # JSON-serializable details

class ProgressBus:
    """Thread-safe, append-only event log with queue-based fan-out. The async pipeline
    emits from the event-loop thread; SSE handler threads consume via per-client queues."""
    def __init__(self) -> None:
        self._lock = threading.Lock()
        self._events: list[ProgressEvent] = []
        self._subs: list[queue.Queue[ProgressEvent]] = []
    def emit(self, type: str, *, stage: str = "", section: int | None = None,
             **payload) -> ProgressEvent: ...        # append + fan-out, never raises
    def subscribe(self, since: int = 0) -> tuple[list[ProgressEvent], queue.Queue]: ...
    def unsubscribe(self, q: queue.Queue) -> None: ...

class NullProgressBus:
    """No-op twin (--auto, tests) — mirrors NullRevisionLog (revision_log.py:116-120)."""
    def emit(self, *a, **kw) -> None: return None

Injection, not module-level. run_pipeline already injects every observability seam explicitly (revlog, checkpoint, clock — run.py:135–140, 320–344); a module-level singleton would leak events across concurrent tests and hide the dependency. Add a keyword-only progress=None parameter to run_pipeline and _refine_loop, resolved as progress = progress or NullProgressBus(), exactly the revlog = revlog or NullRevisionLog() idiom at run.py:193. --auto and every existing test pass nothing and see zero behavior change. emit() must be exception-swallowing (log-and-continue), matching the revision log's "pure observability — never breaks a run" contract (revision_log.py:112–113).

3.2 The server — SSE over long-poll (proposed, new module live_server.py)

Two stdlib-compatible options were weighed:

Long-poll (GET /events?since=N, blocks up to ~25 s)Server-Sent Events (GET /events, response held open)
Stdlib fitTrivial — every request is an ordinary short-lived handler; ThreadingHTTPServer's thread-per-request model is designed for itAlso fine — an SSE response is just a normal HTTP response whose handler never returns: write headers, then keep writing data: frames to self.wfile. One thread pinned per client, but there is exactly 1 client (occasionally 2 tabs)
Client codeHand-rolled fetch loop with since-cursor, backoff, dedupBuilt-in EventSource: auto-reconnect, Last-Event-ID replay for free
Chatter/latencyOne request per event burst (or per timeout); events between polls batch up; more moving parts to get low-latency rightPush-latency ~0; one connection for the whole run
Disconnect detectionImplicit (requests stop)Needs heartbeats: write a comment frame : hb\n\n every ~15 s; a BrokenPipeError/ConnectionResetError on write ends the handler and unsubscribes the queue
Proxies/bufferingRobust anywhereCan be buffered by intermediaries — irrelevant on 127.0.0.1

Recommendation: SSE. On localhost with one viewer, SSE's only real cost (a pinned thread per client) is negligible, and it deletes the entire client-side polling machinery in exchange for ~15 lines of handler loop. Long-poll's robustness advantages only matter across proxies we don't have. Concretely:

# Proposed: packages/xlights-orchestrator/src/xlights_orchestrator/live_server.py
class LiveProgressServer:
    """ThreadingHTTPServer (daemon_threads=True) on 127.0.0.1:0, following
    brief_editor.serve() (brief_editor.py:205-222) but running in a background
    thread so the async pipeline continues underneath it."""
    def __init__(self, bus: ProgressBus, gate: CheckpointGate, *, port: int = 0): ...
    def start(self, *, open_browser: bool = True) -> str:   # returns the URL
        # httpd = ThreadingHTTPServer(("127.0.0.1", port), handler); read back
        # server_address[1]; threading.Thread(target=httpd.serve_forever,
        # daemon=True).start(); threading.Timer(0.4, webbrowser.open) as in
        # brief_editor.py:216
    def stop(self) -> None: ...                              # shutdown() + server_close()

# Handler routes (handler factory closing over bus/gate, like brief_editor._handler):
#   GET  /                      → the live page (Section 3.3), text/html
#   GET  /events                → SSE stream: replay bus.subscribe(since=Last-Event-ID),
#                                 then loop: q.get(timeout=15) → "id: {seq}\ndata: {json}\n\n";
#                                 on queue.Empty → ": hb\n\n" heartbeat; on write error →
#                                 bus.unsubscribe(q) and return
#   POST /checkpoint/<id>       → resolve a pending checkpoint (Section 3.4)
#   GET  /revlog?tail=20        → last N revision_log.jsonl records (read from disk)

SSE responses set Content-Type: text/event-stream, Cache-Control: no-cache, and omit Content-Length (the brief editor's _send helper at brief_editor.py:177–183 always sets it; the SSE path writes headers directly instead). daemon_threads = True ensures pinned SSE threads never block interpreter exit.

3.3 The page

One self-contained HTML string in a Python module constant, rendered with placeholder substitution exactly like _PAGE/render_page (brief_editor.py:28–156). Inline CSS + inline vanilla JS, no external resources. Regions, all driven by one EventSource("/events") dispatcher keyed on event.type:

  • Stage timeline — the seven stages as a horizontal strip; stage events flip pending → running → done, with elapsed time per stage.
  • Per-section grid — one cell per ShowPlan section (count arrives in the design stage-end payload); section events mark generated / regenerating / escalated / reverted, with the section's look as a tooltip.
  • QA sparkline — score events append (iteration, objective, advisory) points; an inline SVG <polyline> is rebuilt client-side per event (client-side SVG keeps the server dumb; ~20 lines of JS). Subscores render as a small labelled bar row from QAReport.subscores.
  • Revision-log tail — the last N iteration records, populated from refine events (whose payloads mirror RevisionLogRecord) with GET /revlog as the catch-up path for a page opened mid-run: iteration, obj_before → obj_after (delta), kept/reverted, regenerated sections, judge verdict, human decision.
  • Checkpoint panel — hidden until a checkpoint event arrives; see 3.4.

3.4 Replacing the blocking checkpoints — CheckpointGate (proposed)

The state machine, per checkpoint:

  1. Pipeline posts. The browser-backed checkpoint callable (same signatures as today's _interpret_review(desc_md, brief), _design_review(brief_md, plan), _interactive_checkpoint(report, verdict, ledger)) registers a pending checkpoint with the gate and emits a checkpoint event: {id, kind: "interpret"|"design"|"refine"|"final", body_md, options}.
  2. UI shows approve/edit. The panel renders body_md (pre-rendered to HTML server-side, or shown in a <pre> — see Open questions) plus the kind's buttons: interpret/design → Proceed / Stop (+ a link to xlo edit-brief's form for design, reusing brief_editor as-is: edit, save, then Stop and re-run picks up the cache per run.py:410); refine → Approve revisions / Stop / Keep as final, mirroring the [A/s/k] prompt at run.py:96; final → Save / Discard.
  3. Server sets the result. POST /checkpoint/<id> with {"action": "..."} validates the id matches the pending checkpoint (stale posts → 409) and puts the action on the gate's queue.Queue.
  4. Pipeline resumes. The pipeline side waits with action = await asyncio.to_thread(self._q.get) — a thread-safe blocking wait that does not park the event loop (unlike today's raw input()). It maps the action to the existing return types (bool for the stage gates, Decision(action="accept"|"approve", revisions=...) per refine.py:42–45 for refine) and emits checkpoint_resolved.
# Proposed sketch
class CheckpointGate:
    def __init__(self, bus: ProgressBus) -> None:
        self._q: queue.Queue[str] = queue.Queue()
        self._pending: PendingCheckpoint | None = None   # id, kind, valid actions
    async def wait(self, kind: str, body_md: str, options: list[str]) -> str:
        cp_id = self._register(kind, options)
        self._bus.emit("checkpoint", payload={"id": cp_id, "kind": kind,
                                              "body_md": body_md, "options": options})
        action = await asyncio.to_thread(self._q.get)     # blocks a worker thread, not the loop
        self._bus.emit("checkpoint_resolved", payload={"id": cp_id, "action": action})
        return action

def browser_interpret_review(gate) -> Callable:           # factory per checkpoint kind
    async def _review(desc_md: str, brief) -> bool:
        return await gate.wait("interpret", desc_md, ["proceed", "stop"]) == "proceed"
    return _review

Terminal fallback is kept. _interpret_review, _design_review, _interactive_checkpoint, and _final_approval remain unchanged in run.py and remain the wired defaults whenever the live server is not running: --no-browser, --auto, or a failed server bind all leave today's behavior intact. Additionally, each checkpoint event is mirrored to stdout as a one-line notice with the URL, so an operator who closed the tab knows the run is waiting. (A dual stdin+browser race — first answer wins — is deliberately out of scope; two authorities for one decision invites lost answers. See Open questions.)

5. Testing strategy

The repo's convention is hermetic-first: tests/test_brief_editor.py tests the editor with "No real browser/server" (its docstring, line 1) by exercising the pure seams — render_page placeholder substitution and save_brief round-trip/validation (lines 30–58). Follow that split, plus a real-server tier that stays hermetic because everything is loopback:

  • Bus unit tests (tests/test_progress.py) — emit/subscribe ordering, seq monotonicity, late-subscriber replay via since, fan-out to two queues, unsubscribe, emit-never-raises (a poisoned subscriber queue), thread-safety (emit from N threads, assert count and order).
  • Pipeline emission tests — extend the existing hermetic pipeline tests (test_orchestrator.py / test_refine.py style: fake client, injected analyze/interpret/generator/qa/judge): pass a real ProgressBus, run, assert the expected event sequence (stage bracketing, one section event per section, one score per refine iteration, refine payloads matching the RevisionLogRecord the injected revlog captured, done last). Assert the default (progress=None) run emits nothing and the golden snapshot is byte-identical.
  • Server tests (tests/test_live_server.py) — start LiveProgressServer on port 0 (OS-assigned, no collisions — same trick as brief_editor.py:212), read back the port, and drive it with urllib.request against 127.0.0.1: GET / contains the substituted page (no leftover placeholders, mirroring test_render_page_embeds_brief_schema_colors); open /events as a raw socket or unbuffered urlopen, feed 3 synthetic events through the bus, read and parse the frames (assert id: ordering, data: JSON payloads, heartbeat comment after a quiet timeout with the heartbeat interval injected small); reconnect with Last-Event-ID: 2 and assert replay starts at 3.
  • Checkpoint round-trip without a browser — register a pending checkpoint via gate.wait(...) in an asyncio task; assert the checkpoint event appears on the SSE stream; urllib-POST {"action":"proceed"} to /checkpoint/<id>; assert the task resolves True and checkpoint_resolved is emitted. Repeat for the refine kind asserting a proper Decision; POST a stale/unknown id and assert 409 with the pending checkpoint unconsumed.
  • Fallback tests — with no live gate wired, monkeypatch builtins.input and assert _interpret_review/_design_review still gate exactly as today (this also back-fills part of the I6 cli/brief_editor coverage gap).

No test opens a browser, touches the network beyond loopback, or needs an API key (the injected-agent pattern from run.py:145–146 already guarantees the latter).

7. Acceptance criteria

  • Attended xlo run --refine prints a localhost URL and opens a live page showing stage timeline, per-section generation, per-iteration objective/advisory scores with subscores, and the revision-log tail, updating without manual refresh.
  • The interpret and design reviews (today's input() at run.py:115 and run.py:106) are answerable from the browser; choosing Stop returns st early exactly as the current gates do (run.py:403–405, 419–423).
  • The refine checkpoint and final save approval (run.py:96, run.py:120) are likewise answerable from the browser, with the refine panel offering approve/stop/keep matching _interactive_checkpoint's semantics.
  • --no-browser reproduces today's terminal-only attended behavior byte-for-byte; --auto starts no server, opens no browser, and blocks on nothing (cli.py:27–29 wiring unchanged).
  • A page opened (or reloaded) mid-run reconstructs full state from event replay; a run that finishes shows a terminal done state.
  • All new tests are hermetic (loopback-only, port 0, no API keys, no browser); the golden pipeline snapshot and the existing brief_editor tests pass unchanged.
  • The live page loads zero external resources (verifiable by grepping the page constant for http/// URLs).

Stdlib has no Markdown renderer; options are a <pre> block (zero risk, adequate), a ~40-line subset renderer (headings/bold/lists), or linking the on-disk .md. Lean: subset renderer, shared by the revlog tail.
  • Should the design checkpoint embed the brief editor? The cleanest v1 links to a separately-served edit-brief form (edit → save → Stop → re-run picks up the cache per run.py:410–411). Mounting brief_editor's handler under the live server (/brief) and adding an "edit → save → proceed" path that reloads st.show_plan in-place before the design gate returns would remove the re-run — but mutating st mid-checkpoint needs care with the instructions cache (run.py:426–433 keys off the song, not the brief hash). Defer unless cheap.
  • Dual stdin+browser answering. Racing input() against the gate queue (first answer wins) is possible via a daemon stdin-reader thread, but abandoned input() threads are messy. v1: browser-only when the server is up (with stdout notices), terminal-only otherwise.
  • Event scope for F-J. Should ProgressEvent carry the I5 degradation events and (post-I1) token/cost payloads now, or add types later? The type+payload shape is open-ended by design; recommend defining only the types in Section 3.1 and letting I1/I5 add theirs.
  • Persist the event stream? Dumping bus._events to progress.jsonl alongside revision_log.jsonl at run end is nearly free and would feed the F-G dashboard; deferred to keep F-I's surface minimal.
