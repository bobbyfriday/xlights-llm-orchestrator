xLights LLM Orchestrator

Multi-agent generation of xLights light-show sequences from a song · implementation plan

Context

A local tool that turns a song into a finished xLights light-show sequence using multiple LLM agents working in parallel — analyzing the song, generating effects per song-section/model-group, testing/critiquing the result, and iterating to a final sequence. Two deliverables:

1 · MCP server
Wraps the xLights local automation API so any MCP client (Claude Code included) can drive xLights.
2 · LLM orchestrator
Standalone Python CLI that fans out parallel agents over the MCP/REST surface to generate and refine a sequence.

Greenfield project (/Users/rob/git/xlights_llm_orchestrator is empty).

Key research findings (drive the design)

  • xLights exposes an HTTP REST automation API at http://127.0.0.1:49913 (instance "A"; 49914 = "B"). Each command is GET /<cmd>?param=val or POST /<cmd> with a JSON body; responses are JSON with a res/status code. (Source: src-ui-wx/automation/xLightsAutomations.cpp in the xLights repo.)
  • ~75 commands exist. Load-bearing ones: getModels/getModel, getViews, newSequence/openSequence/loadSequence/getOpenSequence/saveSequence/closeSequence, addEffect (one effect → one target model/group, one layer, with a raw settings string + palette string, startTime/endTime in ms), getEffectSettings/setEffectSettings, renderAll/batchRender, checkSequence, playSequence, exportVideoPreview (renders a preview video → visual feedback loop), getControllers.
  • prepareAudio, setAudioVolume, setAudioSpeed are NOT implemented (return 504). So audio analysis must live in our tool, not be delegated to xLights.
  • A sequence = .xsq (editable XML) + .fseq (compiled output). Effects live on models/groups, on layers, with ms start/end. Timing tracks hold beat/bar/section marks.

Decisions (confirmed with user)

  • Python 3.11+; orchestration on PydanticAI (agent + model layer) + pydantic-graph (graph orchestration with state persistence). Standalone CLI.
  • Provider-agnostic / multi-model. Each agent role maps to a model spec (provider + id + settings) in config, so Claude, Gemini, etc. are swappable per role without code changes. Provider-specific features are optional optimizations, never load-bearing.
  • Monorepo, 3 separately-installable packages so the MCP server stays lightweight (no LLM/audio deps) and the shared REST client/preset library is single-sourced. The MCP server is independently useful/publishable.
  • The hardest correctness problem — emitting valid xLights settings strings — is solved by mining the user's existing .xsq corpus into a parameterized preset library.
  • Build the MCP server first, then layer the orchestrator on top.
⚠️ Model usage is load-bearing but provider-specific, applied via PydanticAI per-model ModelSettings:
  • Claude defaults: claude-opus-4-8 (planner/critic), claude-sonnet-4-6 (generators/tester), with thinking={"type":"adaptive"} + output_config={"effort": …}; do not pass temperature/top_p/top_k/budget_tokens (they 400). Re-verify against the claude-api skill at impl time.
  • Gemini defaults: gemini-2.5-pro (planner/critic), gemini-2.5-flash (generators) with its own thinking-budget config (verify current IDs at impl time).
  • The registry holds these as per-provider settings so swapping a role's provider also swaps in the right knobs.

Both the MCP server and the orchestrator's tester/critic talk to xLights and need the preset library, so that shared code lives once in xlights-core. MCP tools are thin wrappers over core.client; the orchestrator depends on core directly.

The hardest problem — valid effect settings strings

Free-form generation of E_TEXTCTRL_…/C_BUTTON_Palette1_… strings will hallucinate keys and silently render wrong. Instead, a curated parameterized preset library mined from real .xsq files:

  1. xsq_extractor.py (offline): parse the user's .xsq corpus, extract each effect's exact settings+palette strings per effect type — valid by construction.
  2. Template-ize: identify the keys that vary meaningfully (speed, scale, direction, palette slots), freeze the rest, store as JSON presets with typed params.
  3. preset_library.render(name, params, palette) does safe substitution; validators.py rejects out-of-range/unknown params before addEffect.
  4. Agents only see the preset menu (xl_list_effect_presets) and call xl_add_effect_from_preset with typed params — never raw strings. A gated, validated xl_add_effect_raw exists as an escape hatch.
  5. Validation layers: effect ∈ catalog → params ∈ ranges → target ∈ live getModels & timing in bounds → checkSequence as the final server-side gate.

Orchestration engine (how the conductor runs the agents)

Two kinds of components: deterministic (code, cheap, no tokens) and agents (LLM, at decision points). Principle: deterministic gates first, LLM judgment only on what passes — never spend a vision-critic call on a sequence that fails checkSequence.

Topology — deterministic graph + LLM workers

Control flow is a pydantic-graph Graph of typed BaseNodes, not decided by a model. The dependency graph is known (analyze → interpret → design → generate → test → decide → finalize), giving reproducibility, cost control, debuggability, and built-in checkpoint/resume. An autonomous "Director decides next action" variant is out of scope for v1.

Agents — PydanticAI, provider-agnostic

Every agent is a pydantic_ai.Agent with a system prompt, an output_type (pydantic → validated structured output, no parsing), and its tools. The model is injected from models/registry.py (role → provider model + per-provider ModelSettings). Two shapes: structured one-shot (analysts, synthesizer, director, generators, judge) and tool-using (visual critic — read-only tools only).

Execution substrate (used inside nodes)

  • asyncio.gather + Semaphore — a node fans out its agents concurrently, capped. Helper run_panel(coros) wraps gather+semaphore+return_exceptions so one failure doesn't sink the batch.
  • ProcessPoolExecutor — blocking CPU/GPU DSP (demucs, whisperX, essentia, librosa, vamp) runs off the event loop.
  • asyncio.Lock write-queue — all xLights mutations pass through one lock in core.client (one shared open sequence).

Generation concurrency — "plan then apply" (the key safety pattern)

Generators do not call addEffect mid-reasoning. Each returns a validated EffectInstruction[] for its section (pure, parallel, no lock). The Generate node then validates and applies all instructions serially through the write lock. This separates thinking (parallel) from writing (serial, deterministic, retryable) — removing write races and making a failed section a cheap re-plan.

Shared state — pydantic-graph State as the typed blackboard

Holds the run's artifacts: SongAnalysis → MusicBrief → ShowPlan → effects → findings → ledger. Nodes read the slice they need and write structured results back via GraphRunContext. The refine loop is just nodes returning the next node (Decide → Generate), bounded by MAX_ITERATIONS + the anti-oscillation ledger.

Resilience & observability

PydanticAI retries transient/validation errors; the semaphore bounds pressure. StatePersistence (sqlite) checkpoints state per node → resume mid-pipeline + step-through replay. Optional Pydantic Logfire captures model, tokens, tool calls, latency.

Deterministic components (NOT agents)

AudioAnalyzer (VAMP + librosa) → SongAnalysis (raw features)

  • VAMP plugins (via the vamp module, VAMP_PATH → xLights' bundled Vamp Plugins/ so timing matches xLights): QM Bar-and-Beat Tracker (downbeats/bars), QM Tempo/Beat, QM Key Detector, QM Onset Detector, QM Segmenter / Segmentino (structural sections), NNLS Chroma + Chordino (chords), QM Tonal Change (transition cues).
  • librosa: per-band RMS/energy arc, spectral, MFCC/timbre, HPSS, self-similarity matrix (repetition).
  • Track ID: AcoustID → MusicBrainz → artist/title/genre (style prior + lyric query).
  • Stems: demucs → vocals/drums/bass/other → per-stem energy (instrument-reactive lighting) + clean vocal for alignment.
  • High-level: Essentia ML → mood, danceability, genre.
  • Lyrics: lyricsgenius text → forced-align to vocal stem (aeneas/whisperX) for word/line timing. Fallbacks: syncedlyrics → ASR → none. Feeds xLights phoneme/lyric timing tracks → singing-face / text displays.
  • fusion.py reconciles all into one grid with per-feature confidence. Outputs measurements only; every extractor degrades gracefully (missing token/key/plugin/GPU → skip with a warning).

QA / metrics (run every iteration)

  • Validators — preset ∈ catalog, params ∈ ranges, target ∈ live getModels, timing in bounds. Before every addEffect.
  • Static QA — checkSequence warnings/errors → structured list keyed to model/time.
  • Render QA — per-model render data → activity level (dark props?), clipping, channel conflicts.
  • Sync QA — every effect's start/end vs beat grid + section boundaries; % transitions on beat (no LLM needed).
  • Variety metrics — effect histogram, % models used, energy dynamic range → numeric monotony detection.

The Refiner is not a separate agent — refinement re-invokes the section Generator with its RevisionBrief (what was wrong, what to try, what NOT to repeat).

Parallelism design

  • Primary axis = song section. A chorus should feel unified across all props at that moment; the Director's per-group motifs keep a group coherent across sections without a separate pass.
  • Write serialization: generators reason in parallel but serialize writes through one async lock; sections write disjoint time-ranges, so order is irrelevant.
  • Critics run fully parallel — read-only, no lock contention.
  • Provider-aware prompt caching: where supported (Claude), cache the preset catalog + system prompt and warm it (fire one generator, then fan out as cache reads); skipped on providers without it.

The test + iterate loop

"Testing a song" = the deterministic QA suite + the parallel critic panel, fused by the Judge into a score and a scoped work-list. The loop is scoped (only flagged sections regenerate) and bounded (MAX_ITERATIONS≈3), with an anti-oscillation ledger.

PLAN
 0  AudioAnalyzer -> SongAnalysis (raw features):              [CHECKPOINT: confirm tempo/sections]
       identify -> stems -> VAMP ‖ librosa ‖ essentia ‖ lyrics(genius+align on vocal stem)
 0b Analysis panel (parallel): Structure&Theme ‖ Rhythm&Dynamics ‖ Harmony&Mood ‖ Lyric&Narrative
       -> Music Synthesizer -> MusicBrief                       [CHECKPOINT: pick theme]
 1  Director -> ShowPlan                                        [CHECKPOINT: edit creative direction]
 2  new_sequence + create_timing_track  (marks from VAMP beats/bars/sections)

DRAFT
 3  fan-out Generators (parallel reasoning, serialized writes) -> effects; save_sequence

TEST  (cheap gates first, then LLM panel)
 4  renderAll + Static QA + Render QA + Sync QA + Variety metrics          [deterministic]
       └─ HARD FAIL (render error / checkSequence errors)?
            build RevisionBriefs -> scoped regen -> back to 3   (NO LLM critics spent)
 5  export_video_preview + sample frames (per section + per energy peak)
 6  parallel critic panel:  Visual ‖ Sync ‖ Technical                     [LLM]
 7  Judge fuses findings + metrics -> score + scoped RevisionBriefs + verdict

DECIDE
 8  if score >= THRESHOLD or iter >= MAX or human approves -> finalize
    else: record briefs in ledger; scoped regen -> back to 3   [CHECKPOINT: approve/redirect]

FINALIZE
 9  save_sequence + batch_render (.fseq)                        [CHECKPOINT: approve final]

Convergence & stopping: score ≥ threshold, OR no net score gain, OR MAX_ITERATIONS, OR human approval. Each RevisionBrief carries scope (section, group, layer, time-range), the issue, a suggested fix, and a "do-not-repeat" note. Checkpoints are interactive in the CLI; --auto skips them.

Verification (end-to-end)

  1. Live API smoke test: with xLights running, a CLI helper hits getVersion/getModels on 127.0.0.1:49913 and prints the layout.
  2. Preset round-trip tests: mined presets render to strings identical to their .xsq source.
  3. MCP integration: add the server to a Claude Code MCP config; open a sequence, add a preset effect, checkSequence clean, effect visible.
  4. Audio reference: analyze a known-BPM track; detected tempo within tolerance.
  5. Full run: xlo run --song … produces a rendered .fseq that plays in xLights; preview matches the ShowPlan at section boundaries.

Risks

  • Invalid settings strings (highest): mined preset library + pre-send validation + checkSequence; grow coverage by mining more .xsq.
  • Shared-sequence write races: single async write-lock; disjoint per-section assignment.
  • Audio misdetection: VAMP (same engine as xLights) + librosa cross-check with confidence; human checkpoints 0/0b; manual overrides.
  • VAMP availability: auto-discover xLights' bundled plugins via VAMP_PATH; degrade to librosa-only.
  • Heavy/optional deps (demucs, whisperX, Essentia): ship as [full] extra; cache artifacts; feature-flag + degrade.
  • Lyrics accuracy: gate Genius query on AcoustID match; align on the vocal stem; fall back syncedlyrics→ASR→none; gate lyric-driven effects on confidence.
  • External-service/token risk: cache lookups; respect rate limits; optional enrichment only. (Spotify excluded — audio-features deprecated.)
  • Cost/latency: worker tier for generators, planner only for planner/critic; cache catalog; bound iterations + concurrency; route high-volume roles to a cheaper provider.
  • Multi-provider parity: PydanticAI normalizes tool/structured-output; provider features stay optional; validate each role's model supports what it needs (e.g. vision); pin a known-good default routing.
  • PydanticAI/pydantic-graph maturity: keep domain logic framework-thin so the orchestration layer is replaceable; pin versions.
  • xLights API drift / 504s: typed XLightsNotImplemented; all audio in our module; recorded-response tests.

Open items to resolve at implementation start

  • Path to the user's existing .xsq show folder (input to xsq_extractor.py, Phase B).
  • Path to xLights' bundled Vamp Plugins/ folder for VAMP_PATH (Phase D) — typically bin/Vamp/Vamp Plugins/ in the xLights install.
  • LLM provider keys (Phase F): ANTHROPIC_API_KEY and/or GEMINI_API_KEY (or Vertex creds) per models/config.yaml. At least one required.
  • Optional API tokens (Phase D, pipeline degrades without them): Genius token (lyrics), AcoustID key (track ID). In .env.
  • GPU availability for demucs/whisperX (Phase D) — determines CPU-fallback timeouts.
