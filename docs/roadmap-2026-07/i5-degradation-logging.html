1. Why

The orchestrator's best-effort posture is deliberate and correct: lyrics, stems, visual critique, real-render export, timing tracks, and caching are all enrichments — a run should finish without them rather than die. But the implementation of that posture is 45 except Exception blocks across 20 files (measured 2026-07-05; matches the roadmap count exactly), and they are wildly inconsistent about saying what was lost: 12 of them are completely silent (pass or a bare fallback with no log line at all), the rest scatter across log.debug/log.info/log.warning with no rule for which level means what.

The failure mode is concrete: when a show "looks off," the operator today cannot distinguish "stems failed, so the brief had no instrument features" from "the visual critic never ran because ffmpeg was missing" from "coverage QA was flying blind because the .fseq didn't resolve" — all three degrade silently or near-silently. The roadmap's target state: it turns "the show looks off" into "stems failed, brief had no instruments." Two cheap steps get there: (1) one logging convention applied to every block, (2) a per-run degradations summary — what was skipped and why — printed at the end and written next to the run's other artifacts.

2.3 What "the run log doesn't say why" looks like today

A run where stems, the real render, and coverage sampling all failed produces — at default INFO level — two mid-run lines (real render unavailable: ... at INFO, one stems WARNING from core, possibly scrolled off among placement logs) and zero mention of coverage blindness. Nothing at the end of the run aggregates them. The operator reads the final scores as if all subsystems had been live.

3. Proposed design

3.1 The logging convention (one rule, applied 45 times)

  • log.debug("<what> skipped/fallback: %s", exc) — cosmetic degradations: cache misses/write failures, per-call fallbacks inside an already-reported capability, expected recompute paths. Silent pass is banned; every block logs at least debug with the exception value.
  • log.warning("<capability> lost: %s", exc) — a whole capability is gone for the run or the iteration: no lyrics, no stems, no visual critique, offline-only render, coverage blind, dropped analyst, default view, failed media patch. Warnings for per-iteration capabilities fire per occurrence but the collector dedupes (§3.2).
  • Message shape: start with the capability name used in the collector (greppable), include %s-formatted exc (never bare except: pass), keep the repo's existing # noqa: BLE001 — <reason> comment convention on the except line.
  • Level ≠ audience trick: INFO stays for positive progress lines (as today: "timed lyrics attached", "targetable_groups: 12/19 …"); degradations are only ever DEBUG or WARNING, so grep -i warning over a run log is a complete degradation list even without the collector.

3.2 The degradations collector (proposed: xlights_orchestrator/degradations.py)

Same ContextVar pattern as I1's telemetry (no signature churn; reaches closures like make_visual_critique._vc and make_lit_sampler.sampler), same best-effort guarantee (the collector itself may never raise):

# PROPOSED — new module xlights_orchestrator/degradations.py
from __future__ import annotations
import logging
from contextvars import ContextVar
from pydantic import BaseModel

log = logging.getLogger(__name__)

class Degradation(BaseModel):
    capability: str            # taxonomy key: "audio:lyrics", "visual:real-render", ...
    detail: str                # first occurrence's message ("%s" % exc, truncated ~300ch)
    count: int = 1             # occurrences (dedup key = capability)
    stage: str | None = None   # optional: "interpret" | "generate" | "refine" | "finalize"

class DegradationLog:
    def __init__(self) -> None:
        self.items: dict[str, Degradation] = {}      # keyed by capability → dedup
    def note(self, capability: str, detail: str, *, stage: str | None = None) -> None: ...
    def summary(self) -> list[Degradation]: ...       # insertion-ordered

_current: ContextVar[DegradationLog | None] = ContextVar("xlo_degradations", default=None)

def start_run() -> DegradationLog: ...
def current() -> DegradationLog | None: ...

def note(capability: str, exc_or_detail: object, *, stage: str | None = None,
         level: int = logging.WARNING) -> None:
    """Log AND record in one call — the convention's warning branch is exactly:
        degradations.note("visual:critique", exc)
    Logs `"%s lost: %s" % (capability, exc)` at `level`; records into the active
    DegradationLog if any (no active log → log-only, e.g. bare xlo regen helpers).
    BEST-EFFORT: wraps itself in except Exception → log.debug."""

Capability taxonomy (closed set, documented in the module docstring — free-form keys would defeat dashboard aggregation): audio:lyrics, audio:stems, audio:instrumental-refine, groups:probe, emit:view, qa:coverage-blind, qa:render-flush, visual:critique, visual:real-render, refine:redesign, refine:analyst-drop (from I2's panel change), generate:triggers, finalize:media, finalize:timing-tracks, finalize:xsq-patch, cache:post-refine.

End-of-run summary — emitted at the tail of run_pipeline() (just before return st, after finalize) and likewise in regen_section():

== degradations (3) =====================================
  audio:stems          all separation backends failed: demucs-mlx: ... (×1, interpret)
  visual:real-render   real render unavailable: export_video_preview ... (×3, refine)
  qa:coverage-blind    no .fseq for 'MyShow' (×1, refine)
=========================================================
(or) run completed with no degradations

Rendered via log.warning when non-empty (one block, not per-line — a run that degrades should end loudly) and log.info "no degradations" otherwise. Additionally written best-effort as cache_root()/<song_key>/degradations.json ([{capability, detail, count, stage}, ...] plus run_id/ts), sitting beside revision_log.jsonl and I1's usage.json so F-G can correlate "expensive run" × "degraded run" × "low score".

Optional revision-log tie-in (cheap, recommended): the finalize RevisionLogRecord gains degradations: list[str] = [] (capability keys only) — same backward-compatible default posture as I1's fields, giving the JSONL analyzer the signal without parsing a second file.

3.3 The core/orchestrator boundary rule

xlights-core must not import the orchestrator's collector (dependency direction) and shouldn't grow its own: core's blocks get the convention only (debug/warning per §2.2). The orchestrator reports core-owned capabilities by observing outcomes at the seam, which it already can:

  • audio:stems — after AudioAnalyzer().analyze(song_path, stems=True) in run_pipeline step 1: if stems and not getattr(st.song_analysis, "stems", None): degradations.note("audio:stems", "no stems in analysis (separation failed or unavailable)", stage="interpret").
  • audio:lyrics — the existing run.py:367 block plus the "still no lines" condition at :371 already bracket the outcome.
  • Render-order patches — patch_view/patch_xsq_render_order return bool; finalize checks the return value it currently discards.

This keeps core reusable (the MCP server links core without any orchestrator machinery) while the run-level story stays complete.

4. Implementation plan

  1. Create degradations.py (§3.2): Degradation, DegradationLog, ContextVar plumbing, note(), render_summary() (pure list → text, for testing), the capability-key docstring taxonomy.
  2. Install per run: degradations.start_run() at the top of run_pipeline() and regen_section(); summary emission + degradations.json write at both exits (including the early-return checkpoints at run.py:405,423 — a declined checkpoint still prints what degraded up to that point).
  3. Sweep pipeline/run.py (9 blocks) to the §2.2 targets — the highest-value file; includes the audio:stems seam check from §3.3.
  4. Sweep pipeline/visual.py (6), pipeline/groups.py (5) — including the groups.py:43 re-raise (delete the block, or narrow it to re-raise; update its docstring and the "returns []" contract in callers — run_pipeline needs no change since it currently just stores the list).
  5. Sweep the remaining orchestrator files: effect_emitter.py:84, qa/coverage.py:31 (once-per-run flag on the closure), weave.py:316, media.py, timing.py, finalize.py, triggers.py, lyrics.py, regen.py, revision_log.py (level check only).
  6. Sweep xlights-core (convention only, no collector): analyzer.py ×4, stems.py ×2, lyrics_align.py, preview/knowledge files per §2.2.
  7. Wire I2 synergies if landed: panel analyst drop → note("refine:analyst-drop", ...); retry-exhaustion warnings need no collector entry (the terminal failure surfaces elsewhere).
  8. Optional finalize-record field: add degradations: list[str] = [] to RevisionLogRecord and populate at both finalize call sites (skip-gate and end-of-loop).
  9. Docs: taxonomy table + "reading a degradations summary" paragraph in docs/usage.md; add a repo convention note (CLAUDE.md/openspec project conventions) that new best-effort blocks must log per §3.1 — the mechanism that keeps the count honest as the roadmap's "zero TODO markers" culture intends.

5. Testing strategy

Hermetic-first (pytest -m "not live", injected fakes per tests/test_refine.py / test_orchestrator.py conventions):

  • tests/test_degradations.py (new):
    • note() with no active log is a no-op that still logs (caplog assertion) and never raises (call it with a raising fake detail object).
    • Dedup: two note("visual:critique", ...) ⇒ one item, count == 2, first detail kept.
    • render_summary(): empty ⇒ "no degradations" text; non-empty ⇒ one line per capability with count and stage.
    • ContextVar isolation: two sequential start_run()s don't leak items across runs.
  • Pipeline integration (extend test_orchestrator.py): run the hermetic pipeline with an injected visual_critique fake that raises ⇒ finalize summary contains visual:critique ×N and the run still completes; with all fakes healthy ⇒ empty summary, degradations.json absent or empty list.
  • Per-block caplog tests for the upgraded silent blocks (representative, not exhaustive): qa/coverage with a raising sampler logs the blind warning exactly once across many evaluate calls; effect_emitter with a client whose view-create raises logs emit:view and falls back (extend the existing emitter tests' fake client).
  • groups.py:43 re-raise: update tests/test_targetable_groups.py — client whose get_group_names raises now propagates XLightsConnectionError (was: returned []); keep the probe-failure fallback tests unchanged (they pin the still-degrading path).
  • Back-compat pins: golden pipeline snapshot unchanged (logging-only edits must not touch artifacts); if the optional revision-log field ships, reuse I1's old-JSONL-line-validates test pattern.
  • Log-level audit test (cheap guard): a unit test that greps the two packages' source for except Exception blocks whose body contains neither a log. call, a note( call, nor a raise — pinning "no silent swallows" as a structural invariant rather than a review habit. (Simple AST walk; skip server.py/brief_editor.py re-raise patterns automatically since they contain raise.)

7. Acceptance criteria

  • ☐ Zero fully-silent except Exception blocks remain in packages/*/src: every block logs (≥ debug with the exception value), reports via note(), or re-raises — enforced by the structural audit test.
  • ☐ Every whole-capability loss in the §2.2 inventory logs at WARNING and appears in the end-of-run summary with its taxonomy key.
  • ☐ A completed run (including early-checkpoint exits and xlo regen) ends with either the degradations block or an explicit "no degradations" line, and writes degradations.json beside the revision log when non-empty.
  • ☐ qa/coverage.py blindness is reported exactly once per run; effect_emitter default-view fallback is reported; run.py:178 flush failure is reported.
  • ☐ targetable_groups raises on a failed group listing (test-pinned) while retaining all existing probe-failure fallbacks.
  • ☐ xlights-core gains no orchestrator imports; its blocks follow the level convention only.
  • ☐ Full hermetic suite + golden pipeline snapshot pass; ruff check clean; no new runtime dependencies.
  • ☐ The summary demonstrably answers the roadmap's motivating question: an injected stems failure produces a run whose tail says audio:stems — ... instead of nothing.

8. Open questions

  1. Merge with I1's collector? Telemetry (UsageLog) and degradations (DegradationLog) are structurally twin ContextVar collectors installed at the same two entry points. One runinfo.py module holding both would halve the plumbing — but couples two independently-landable items. Proposal: keep separate modules, extract a shared _ContextCollector helper only if a third collector ever appears.
  2. Stage attribution — pass stage= explicitly at each call site (proposed, zero magic) vs. a stage ContextVar the pipeline sets as it advances (less call-site noise, more machinery)? Explicit wins for 20-odd sites.
  3. Attended-mode surfacing — should the interactive checkpoints (_interactive_checkpoint, _design_review) print degradations-so-far, so a human can abort a stems-less run before paying for generation? Cheap once the collector exists; F-I's browser progress UI is the eventual home.
  4. Severity beyond two levels — is a third tier ("degraded-but-scored" vs "advisory-only lost") worth encoding, or does the capability key already imply it? Leaning: the key implies it; keep two levels.
  5. run.py:440 escalation criteria — after N observed finalize:media occurrences in real degradations.json files, promote to re-raise? Record the decision rule now so the data collection has a purpose.
