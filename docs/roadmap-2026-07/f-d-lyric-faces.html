1. Why

Singing faces are the community's signature vocal-song move: a character prop whose mouth shapes follow the vocal line. The effects catalog names the effect plainly — "Faces: Phoneme-driven mouth rendering for singing face models, with eye and outline options … Vocal lip-sync" (xlights-effects-catalog.md §8/roster table) — and the craft roadmap scopes it as the biggest single build of the vocal dimension (item 7, Complexity L). The pipeline already does the hard audio work: fetched lyric text is force-aligned against the separated vocal stem with word-level timestamps (lyrics_align.py), and that timing already drives lyric color triggers and the "Lyrics" reference timing track.

What's missing is three specific things, none of them mysterious: (1) phoneme-level timing — xLights' Faces effect animates mouth shapes from a timing track whose marks are phoneme labels, one level finer than the words we have; (2) a placement route — Faces is in ASSET_BOUND_TYPES (constants.py:9), excluded from the mined catalog because its settings reference a face definition and a timing track that only exist in a specific layout/sequence (F-B builds the route); and (3) hardware — a prop with a face definition to sing on, which the current layout does not have. This document designs the full path and is explicit about what is buildable now versus conditional on props arriving.

2. Current state

2.1 Word-level timing exists; that is the ceiling today

xlights_core/audio/lyrics_align.py: Whisper (MLX, mlx-community/whisper-small-mlx) transcribes the vocal stem with word_timestamps=True (_transcribe, lines 42–52); each lyric line is fuzzy-matched into the transcript word stream with a monotonic cursor (_match_lines, lines 55–87), and — crucially for F-D — the matched per-word spans are persisted:

# lyrics_align.py:80–83 — inside the matched-line record
# persist the matched per-word timing (already computed) so lyric
# triggers can hit the WORD, not just the line — see triggers.py
"words": [{"word": w[0], "start": round(w[1], 2), "end": round(w[2], 2)}
          for w in words[best_i:j + 1]]

Output shape: SongAnalysis.lyrics = {lines: [{text, start, end, words: [...]}], sections: [...], repeated: [...]}. Alignment is fully graceful — any failure returns None and the run proceeds (lines 129–131). There is no phoneme anywhere in the codebase (no g2p dependency, no ARPAbet/Papagayo vocabulary).

2.2 Timing tracks are authored offline at finalize

pipeline/finalize.py::finalize_sequence (lines 21–40): after save_sequence, the run closes the sequence and patches the .xsq offline — "attaching media / reordering via the live API crashes xLights" — including reference timing tracks via build_timing_tracks + patch_xsq_timing_tracks (pipeline/timing.py). The patcher already mirrors the corpus timing-track XML exactly and is idempotent (same-named tracks are removed before re-adding, timing.py:167–169) and atomic (temp + os.replace). Today each track is a single <EffectLayer> of labeled marks:

# timing.py:170–179
ET.SubElement(display, "Element", {"type": "timing", "name": tr.name, ...})
el = ET.SubElement(effects, "Element", {"type": "timing", "name": tr.name})
layer = ET.SubElement(el, "EffectLayer")
for m in tr.marks:
    ET.SubElement(layer, "Effect", {"label": m.label, "startTime": ..., "endTime": ...})

The existing "Lyrics" track (_lyric_track, lines 123–127) is line-granular. A Papagayo-style lyric track in xLights is the same Element type="timing" but with three EffectLayer children — phrases, words, phonemes — which the current dataclass (TimingTrack(name, marks), one flat list) cannot express. That is the concrete extension point (§3.4).

2.3 Faces is excluded from placement, by design

knowledge/constants.py:9: ASSET_BOUND_TYPES = frozenset({"Faces", "Pictures", "Video", "Shader", "DMX"}) — "settings reference external resources not guaranteed to exist in a target sequence." For Faces those references are (a) the target model's face definition (the faceInfo mapping in the user's rgbeffects.xml that says which nodes/submodels form each mouth shape, the eyes, the outline) and (b) a timing track name to drive the phonemes from. The extractor skips them at mining (xsq_extractor.py:78–80), so there is no Look, and place_preset's get_look would raise KeyError. F-B's build_faces_settings + place_direct is the designed route; its Faces builder deliberately requires verified references (F-B §3.2/§3.5 ring 2).

2.4 Hardware honesty: this layout has no singing faces

The spec reserves the role and quarantines it from choreography:

  • xlights-layout-semantics-spec.md §2: role SINGING_FACE ("Singing faces / talking props — Custom with face submodels"), capability class SPECIAL: "singing faces. Excluded from general choreography; driven by dedicated face tracks only." §5: "SEM_ALL — every model except SINGING_FACE and SIGN … Do not put SINGING_FACE models into any SEM_ group except where explicitly listed. Faces are driven independently." Name heuristics (§3): "face, sing, carol, mouth → SINGING_FACE."
  • Code implements the quarantine but nothing else: layout_semantics._NON_ENSEMBLE = ("SINGING_FACE", "SIGN") (line 47) excludes the role from every ensemble group, and _ROLE_GROUP (line 43) has no SEM_ group for it — correctly, since it must never receive beds/weaves/accents.
  • The actual layout: tests/fixtures/getModels.json lists Arches, Matrix, Mega Tree, House Outline (+ two groups); no face-named prop, and no faceInfo handling exists anywhere in the codebase to detect one.

Consequently every rendering deliverable of F-D is conditional on hardware. What is not conditional: the phoneme extractor, the multi-layer timing-track writer, the Faces settings builder, and the gating logic — all buildable and hermetic-testable now, and the phoneme timing track is independently useful (it makes the saved .xsq hand-editable for faces the moment a prop arrives, and it is exactly what a human sequencer would import from Papagayo). A realistic landing is: ship the track + builder + gated pass; the pass no-ops on this layout until a face prop with a definition appears. Note: technically a Faces effect can render "mouth movements" on a matrix/mega-tree with a face definition drawn for it — if the user ever defines one on the Matrix, the same pass lights up unchanged.

3. Proposed design

3.1 xLights Faces anatomy (what we must produce)

The Faces effect renders a mouth shape per frame chosen from the model's face definition. The mouth-shape vocabulary is the Papagayo/Preston Blair set xLights uses for its phoneme timing: AI, E, etc, FV, L, MBP, O, rest, U, WQ — ten labels, where rest is the closed mouth between words and etc is the consonant catch-all. Driven mode: the effect binds to a timing track whose phoneme layer's mark labels are these tokens; at each frame xLights shows the mouth image/node-set mapped to the active label. The settings string therefore references (key names to be frozen from the F-B probe protocol — author one by hand, read it back from the .xsq EffectDB): the phoneme source (timing track name), the face definition name, eyes mode, and outline toggle. The catalog adds one placement rule we adopt as-is: "when Faces is active, surrounding props dim to ≤30% so the singer owns focus" (xlights-effects-catalog.md §8 Faces).

3.2 The phoneme-extraction gap: two candidate approaches

A. G2P lookup + heuristic distribution (recommended first)B. Forced aligner (Montreal Forced Aligner)
MechanismCMUdict lookup (via g2p-en or pronouncing; g2p-en falls back to a trained model for OOV words) → ARPAbet phonemes per word → map ARPAbet → Papagayo set → distribute across the word's already-aligned Whisper spanAcoustic model force-aligns audio + transcript to phone-level timestamps directly
New depsOne small pure-Python package + bundled CMUdict (~4 MB)MFA: Kaldi-based, conda-distributed, model downloads, heavy native deps — hostile to this repo's uv workspace and CI
AccuracyWord boundaries exact (Whisper-aligned); intra-word phoneme boundaries approximatePhone boundaries genuinely acoustic
Failure modeSlightly early/late mouth shapes within a word (tens of ms at yard-viewing distance)Whole-pipeline fragility; alignment failures on separated stems with bleed
FitMatches the repo's graceful-degradation posture; hermetic-testable (dict lookup is deterministic)Overkill until someone can see the difference on a real prop

Recommendation: A now, B never-unless-proven-needed. Community practice calibrates this: hand sequencers use Papagayo/xLights' own "breakdown words" — also dictionary lookup with even distribution — and it reads fine on props. The distribution heuristic can be slightly better than uniform: weight vowel-class phonemes (AI/E/O/U/WQ) ~2× consonant classes (MBP/FV/L/etc) within the word span, since vowels carry the sung duration; insert rest in inter-word gaps > ~120 ms.

    "UW": "U", "UH": "U",
    "W": "WQ", "R": "WQ", "ER": "WQ",
    "F": "FV", "V": "FV",
    "M": "MBP", "B": "MBP", "P": "MBP",
    "L": "L",
    # everything else (S, Z, T, D, K, G, N, NG, CH, JH, SH, TH, DH, HH, Y) → "etc"
}

def word_phonemes(word: str) -> list[str]:
    """Papagayo labels for a word via CMUdict/g2p-en; ['etc'] fallback for
    unpronounceable tokens. Pure + deterministic for in-dictionary words."""

def phoneme_marks(lines: list[dict]) -> list[tuple[str, int, int]]:
    """(label, start_ms, end_ms) marks from aligned lyric lines' word spans:
    per word, distribute its phonemes across [start, end] (vowels 2× weight);
    'rest' marks fill inter-word gaps > REST_GAP_MS."""

The mapping table is committed and unit-tested; it is judgment-free realization, consistent with the repo's split.

3.3 Vocal-songs-only gating

The pass runs only when all hold, each already computable:

  1. Vocal song: sa.lyrics["lines"] non-empty with word spans (the same signal that flips the panel's instrumental flag; cf. director.render_input's instrumental = not brief.featured_lyric_moments and not brief.narrative_summary, director.py:24 — but prefer the analysis-side signal: aligned words are what the track is built from).
  2. Singing prop present: a model classified/nameable as SINGING_FACE (spec §3 name heuristics: face/sing/carol/mouth) and carrying a face definition — detected by a new face_definitions(rgb_path) -> dict[str, list[str]] helper parsing faceInfo elements in rgbeffects.xml (proposed home: knowledge/layout_semantics.py, beside the other rgbeffects readers). No definition → no Faces, ever — the reference must be verified, never invented (F-B ring 2).
  3. Track scheduled: the run will write the phoneme track at finalize (i.e. timing_tracks=True and save path known), since the placed effect references it by name.

Instrumental songs skip at gate 1 with a single degradation-log line (roadmap I5 style). This layout skips at gate 2 today.

3.4 Timing-track authoring: extend the finalize path

Extend timing.py minimally:

  • TimingTrack gains layers: list[list[TimingMark]] | None = None (or a sibling MultiLayerTimingTrack) — when present, patch_xsq_timing_tracks emits one <EffectLayer> per layer instead of the single layer at timing.py:175. Existing single-layer tracks are untouched (back-compat by default value).
  • New builder _phoneme_track(sa, end_ms) -> TimingTrack | None: name "Lyric Phonemes", three layers — phrases (the aligned lines, reusing _lyric_track's marks), words (the per-word spans), phonemes (phonemes.phoneme_marks) — the exact Papagayo import shape a human would recognize. Appended to the candidate list in build_timing_tracks (timing.py:134–141), so it inherits idempotent replace + atomic write for free.

The ordering wrinkle (and why it resolves): effects are placed live during generation, but timing tracks are patched offline after save (finalize.py — live timing edits crash xLights). So the placed Faces effect references a track that does not exist in the open sequence yet. That is acceptable by design: the settings string binds by name; when the user opens the saved .xsq, the track (patched moments after save) resolves and the faces sing. The cost: in-run renders — the visual critic, coverage sampler, and RealRender — never see mouth movement, only whatever the effect renders trackless (typically rest/outline). Mitigations: (a) faces are SPECIAL-class and excluded from choreography QA anyway — coverage/variety must ignore the face prop (it is outside every SEM_ ensemble already); (b) the acceptance eyeball happens on the finalized file, called out in the live-verify checklist. A future alternative — placing the Faces effect itself via offline .xsq patch at finalize, after the track — is sketched in Open Questions.

3.5 The placement pass

# PROPOSED — pipeline/faces.py
PHONEME_TRACK_NAME = "Lyric Phonemes"

def place_faces(st: State, face_models: dict[str, str]) -> list[EffectInstruction]:
    """One Faces instruction per singing-face model per SUNG region (the aligned
    lines' spans, merged with ≤2 s gaps), not the whole song — faces close and go
    dark in instrumental breaks. direct_settings = build_faces_settings(
    timing_track=PHONEME_TRACK_NAME, face_definition=face_models[model], ...);
    on_top=True (a face must never sit under a wash — though ensembles exclude it
    by construction, an explicit Generator instruction could still target it)."""

Companion rule from the catalog: during sung regions, dim surrounding props to ≤30% — v1 scopes this to a gentle brightness multiplier on concurrent SEM_ALL-bed instructions only (full "singer owns focus" choreography is a later refinement; it interacts with improve-musicality's treatments). Instructions are tagged with the owning section's section_index per span so refine-loop splices recreate them (replace_section semantics), and the pass re-runs after splices, idempotent by construction (it fully owns all Faces-type instructions: filter-and-rebuild).

4. Implementation plan

Stages 1–3 are buildable and shippable now (no hardware); stage 4 is conditional.

  1. Phoneme extractor (xlights_core/audio/phonemes.py): add g2p-en (or pronouncing) to xlights-core deps; commit _ARPA_TO_PAPAGAYO; implement word_phonemes + phoneme_marks with vowel-weighted distribution and rest-gap insertion. Hermetic tests: known words → exact label sequences; OOV token → ['etc']; distribution sums to the word span; rest marks in gaps; deterministic across runs. Heavy import deferred inside functions, mirroring lyrics_align._transcribe's pattern.
  2. Multi-layer timing track: extend TimingTrack/patch_xsq_timing_tracks for N EffectLayers (back-compat default); add _phoneme_track to build_timing_tracks gated on aligned words existing. Tests: XML round-trip (parse the patched file, assert 3 layers + labels), idempotent re-patch, no track when no lyrics. This alone delivers user value: a Papagayo-shaped track in every vocal .xsq.
  3. Faces builder + detection (with F-B): freeze the Faces settings template from a hand-authored probe on a scratch model with a throwaway face definition (F-B probe protocol); implement build_faces_settings (required refs, no defaults) and face_definitions() over rgbeffects.xml; ring-1 round-trip + ring-2 reference tests with a fixture rgbeffects containing a synthetic faceInfo.
  4. Placement pass (pipeline/faces.py): gates (§3.3), sung-region merging, instruction construction, wiring into generate_instructions + splice re-run; degradation log lines for each gate. Hermetic tests with a fake layout that has a face model: instructions span sung regions only; instrumental → none; no definition → none; this repo's real fixture layout → none (the honest default).
  5. Live verify (hardware-conditional): on a layout with a real face prop — or a face definition drawn on the Matrix as a stand-in — run a vocal song end-to-end; open the finalized .xsq; confirm the phoneme track imported, the Faces effect bound to it, mouths track the vocal by eye, and surrounding dim applies. Until then, the golden stays untouched (the fixture song is instrumental-shaped and the fixture layout has no faces — assert that explicitly in a test).

5. Testing strategy

  • Hermetic-first (pytest -m "not live"): everything except stage 5. Extractor tests are pure dictionary/math; track tests parse real XML written by the patcher (pattern of existing timing-track tests); builder tests reuse F-B's parser round-trip; pass tests use fake State/layout fixtures — no xLights, no network, no Whisper (aligned lyrics dicts are fed as fixtures, as tests/test_lyric_alignment.py already does).
  • Negative-space tests are first-class: instrumental song → no track additions beyond today, no instructions; layout without faces (the shipped fixtures) → pass no-ops; missing face definition → refused with a logged reason. These encode the honesty constraints so they cannot regress.
  • Schema-drift guard: a stored known-good lyrics payload (with words) validated in tests, per roadmap I6's convention, since the extractor now depends on that shape.
  • Golden: expected zero change (both gates fail on the fixture); if any diff appears, that is a bug, not a regeneration case.
  • Live (opt-in -m live, hardware-conditional): a validate_direct-style scratch placement of the frozen Faces template; the finalized-file eyeball checklist from stage 5.

7. Acceptance criteria

Unconditional (this layout, now):

  • ☐ phonemes.word_phonemes/phoneme_marks produce deterministic Papagayo-label marks from aligned word spans; vowel-weighted distribution and rest-gap insertion covered by tests.
  • ☐ A vocal song's finalized .xsq contains a 3-layer "Lyric Phonemes" timing track (phrases/words/phonemes) that round-trips the XML parser; instrumental songs get no such track; re-finalize is idempotent.
  • ☐ build_faces_settings exists (F-B), requires verified timing_track + face_definition, and its output round-trips parse_settings; face_definitions() reads faceInfo from a fixture rgbeffects.xml.
  • ☐ place_faces gates correctly: instrumental → no-op; no face model/definition → no-op with a degradation log line; the shipped fixture layout produces zero Faces instructions and an unchanged golden.
  • ☐ Hermetic suite green: pytest -m "not live".

Hardware-conditional (when a face prop with a definition exists):

  • ☐ Faces instructions span exactly the sung regions (merged ≤2 s gaps), bound to "Lyric Phonemes" and the model's real definition, placed on_top.
  • ☐ Opening the finalized .xsq in xLights shows mouths tracking the vocal by eye; surrounding bed dims during sung regions.
  • ☐ Refine-loop/xlo regen splices recreate the face instructions without duplication.

8. Open questions

  • Should the Faces effect also be patched offline at finalize (after the track) instead of placed live — trading the live-render blindness for a second offline-patch pathway? v1 keeps live placement for emitter uniformity; revisit if the trackless in-run render looks wrong in review bundles.
  • Dictionary choice: pronouncing (pure CMUdict, zero downloads, OOV → etc) vs g2p-en (neural OOV fallback, model download). Leaning pronouncing for CI hygiene — measure the OOV rate on real lyric corpora first.
  • Where does surrounding-dim live long-term — this pass, or improve-musicality's treatment system (a feature-like "vocal focus" treatment)? v1: local multiplier; revisit after F-A Phase 2.
  • Should the phoneme track write even for layouts without faces (pure hand-editing value) — i.e. decouple stage 2 from the feature flag entirely? Recommend yes (it is this doc's stage-2 default); confirm no xLights UI clutter objection from the user.
  • Multiple singing faces (duets): round-robin lines across props, or all props sing everything? Defer until hardware exists; the pass signature (face_models: dict) already permits both.
