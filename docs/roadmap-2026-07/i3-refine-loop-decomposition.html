1. Why

_refine_loop (pipeline/run.py lines 135–305, ~170 lines) is now the most complex, highest-risk function in the codebase. It began life (change 2026-06-08-add-refine-loop) as a simple test→judge→regenerate cycle with three terminators; five subsequent changes each threaded a new guard through the same function body:

  • Objective-regression revert with re-emit of the reverted best (revision-log change, 2026-06-09).
  • Plateau detector (llm-cost-bundle, 2026-06-13).
  • Design escalation to the Director's section_redesigner (test_design_escalation.py era).
  • Skip-high-objective gate (add-refine-skip-high-objective, implemented but not yet archived).
  • Visual-revision backstop flooring (floor_visual_revisions) and revision-log observability (_record, _bundle).

The result: seven distinct control-flow guards, five nested closures (_regen, _redesign, _report, _obj, _record, plus _bundle and _design_implicated), and a 20-parameter signature. Every guard is only testable through the whole loop (see tests/test_refine.py — each test constructs a full fake State/Judge/emitter to probe one branch). The improve-musicality proposal's Phase 2 (section treatments + energy-keyed coverage QA) and Phase 1's new advisory metrics flow through exactly this code — the roadmap explicitly sequences I3 before Phase 2. Meanwhile roadmap R3/R4 (constants consolidation) is only half done: pipeline/tuning.py gathers the show-feel dials, but the refine thresholds and the per-effect metadata tables remain scattered across four modules with cross-module imports.

2.4 The safety net

  • tests/test_golden_pipeline.py — exact snapshot of the deterministic generate stage (103 instructions in fixtures/golden_instructions.json), explicitly built as “a SAFETY NET for refactors that are supposed to be behavior-preserving (e.g. splitting run.py)”. Constants moves must not change it.
  • tests/test_refine.py — 15 hermetic tests covering the loop end-to-end: iterate/accept, cap+stall termination, plateau stop (and non-stop while scores move), checkpoint stop overriding the Judge, and five tests on the skip gate incl. the env override. It imports _refine_loop, REFINE_SKIP_OBJECTIVE, _refine_skip_objective from xlights_orchestrator.pipeline.run (lines 9, 364).
  • tests/test_visual.py, tests/test_design_escalation.py, tests/test_revision_log.py also drive _refine_loop directly; tests/test_effect_speed.py:5 imports SPEED_KEYS from pipeline.beats.

3. Proposed design

3.1 New module: pipeline/refine_loop.py

Extract the loop and its guards into a dedicated module; run.py keeps run_pipeline as the stage skeleton and re-exports the old names so every existing import keeps working. Each guard becomes a named, individually-testable function or small class. All code below is proposed (signatures sketched, not final):

# pipeline/refine_loop.py  (proposed)
"""The refine loop: test → judge → regenerate, with independent safety guards.

Guards are named functions/classes so each is unit-testable without a full loop run.
Termination is never delegated to the Judge: hard cap + plateau + stall + human stop.
"""

# -- pure, stateless guards ---------------------------------------------------

def should_skip_refine(first_pass_objective: int, skip_objective: int | None) -> bool:
    """Gate #1: an already-good draft skips the whole loop (add-refine-skip-high-objective)."""

def refine_skip_objective() -> int:
    """XLO_REFINE_SKIP_OBJECTIVE override, falling back to tuning.REFINE_SKIP_OBJECTIVE."""

def plateau_signature(report: QAReport, verdict: JudgeVerdict) -> tuple:
    """(objective, advisory, frozenset((section, issue[:64]))) — equal signatures = plateau."""

def design_implicated(plan: ShowPlan, section_index: int, findings: list[Finding]) -> bool:
    """True when a rules-finding for this section names one of its planned effect_types
    (currently the _design_implicated closure, run.py:165)."""

# -- stateful guards, one class each ------------------------------------------

class BestTracker:
    """Owns best/best_applied/best_obj/open_is_best and the stall counter (guards #4, #5, #9)."""
    def __init__(self, instructions, applied, objective: int,
                 *, regress_margin: int = tuning.REGRESS_MARGIN,
                 stall_limit: int = tuning.STALL_LIMIT): ...
    def assess(self, obj: int) -> Outcome:            # Outcome(reverted: bool, gained: bool)
        """Pure decision: revert vs keep vs gain, per REGRESS_MARGIN semantics."""
    def keep(self, instructions, applied, obj: int) -> None: ...
    def revert(self) -> tuple[list[EffectInstruction], dict]: ...
    @property
    def stalled(self) -> bool: ...                    # stall >= stall_limit

class EscalationLedger:
    """Owns `ledger` (anti-oscillation, guard #6) + `redesigned` (once-per-run, guard #7)."""
    def record(self, rev: RevisionBrief) -> None: ...
    def prior_sections(self) -> set[int]: ...
    def should_escalate(self, si: int, plan, findings) -> bool:
        """si not yet redesigned AND (repeat offender OR design_implicated(...))."""
    def mark_redesigned(self, si: int) -> None: ...

# -- side-effecting collaborators (still injectable, as today) -----------------

class ReportBuilder:
    """Wraps _report/_obj: flush save_as → real_render.refresh → qa_eval(..., sampler=…),
    preserving the legacy 5-arg qa signature for injected fakes (run.py:172–187)."""
    async def report(self, st: State) -> QAReport: ...
    async def objective(self, st: State) -> int: ...

class IterationRecorder:
    """Wraps _record/_bundle: RevisionLogRecord assembly, review-bundle path guard.
    Pure observability — never raises into the loop."""
    def record(self, i: int, report, verdict, **kw) -> None: ...

async def apply_revisions(st: State, revisions: list[RevisionBrief], *,
                          regen, redesign, ledger: EscalationLedger,
                          findings: list[Finding], log=log) -> None:
    """Guard #7 + splice: per revision, maybe escalate design (structure pinned),
    then replace_section(st.instructions, si, await regen(rev)) and ledger.record(rev)."""

async def refine_loop(st: State, *, client, emitter, ..., skip_objective=None) -> None:
    """Same signature and behavior as today's _refine_loop; now ~60 lines of orchestration
    over the named guards above."""

The loop body after extraction reads as the sequence the archived design docs describe: skip gate → [report → visual findings → judge → decide → plateau check → floor visual revisions → apply revisions (escalating where warranted) → clamp/finalize/emit → assess & keep-or-revert → record → stall check] → restore best → finalize record.

3.2 What stays injectable

The dependency-injection seams the hermetic tests use are contracts, not incidental: judge, qa, regenerate, redesign, checkpoint, visual_critique, revlog, sampler, clock. They remain keyword parameters of refine_loop. The lazy-agent construction rules also remain: the default generator agent is only built when no regenerate is injected (run.py:144–146, “constructing a real Agent needs an API key, which hermetic tests don't have”), and section_redesigner() stays lazy inside the redesign path (run.py:160).

3.3 Constants consolidation — destination map

Two destinations, honoring tuning.py's scope line rather than fighting it:

  1. pipeline/tuning.py gains a “refine-loop control” section for the loop thresholds — they are tunable behavior dials, exactly what tuning.py is for.
  2. A new per-effect metadata table — proposed pipeline/effect_meta.py (orchestrator-side, since every consumer is in the orchestrator) — one data-driven row per effect type, replacing four parallel dicts/sets. Documented like the markdown guides: each row cites its corpus provenance (the SPEED_KEYS ranges and DIRECTION_KNOBS values are corpus-observed; see archived changes 2026-06-12-add-settings-hygiene and 2026-06-12-add-directional-sweeps).
# pipeline/effect_meta.py  (proposed)
@dataclass(frozen=True)
class EffectMeta:
    speed: tuple[str, float, float, str] | None = None   # (key, lo, hi, fmt) — from SPEED_KEYS
    directions: dict[str, tuple[str, str]] | None = None # direction -> (key, value) — DIRECTION_KNOBS
    energy_band: tuple[int, int] | None = None           # catalog §2 — ENERGY_BAND
    duration_class: Literal["hit", "phrase", "cellable", "free"] = "free"
    bed_capable: bool = False                            # weave._BED_EFFECTS
    native_bounce: bool = False                          # weave._NATIVE_BOUNCE
    chase_family: bool = False                           # weave._CHASE_FAMILY

EFFECT_META: dict[str, EffectMeta] = { "SingleStrand": EffectMeta(...), ... }

# derived views — keep every existing name working:
SPEED_KEYS        = {k: m.speed for k, m in EFFECT_META.items() if m.speed}
DIRECTION_KNOBS   = {k: m.directions for k, m in EFFECT_META.items() if m.directions}
ENERGY_BAND       = {k: m.energy_band for k, m in EFFECT_META.items() if m.energy_band}
DURATION_HIT      = {k for k, m in EFFECT_META.items() if m.duration_class == "hit"}
DURATION_PHRASE   = {k for k, m in EFFECT_META.items() if m.duration_class == "phrase"}
DURATION_CELLABLE = {k for k, m in EFFECT_META.items() if m.duration_class == "cellable"}
MOTION_EFFECTS    = DURATION_CELLABLE | {"Fire", "Galaxy"}

4. Implementation plan

  1. effect_meta.py first (pure data move). Create the table + derived views; change beats.py, weave.py, qa/rules.py to import from it; leave re-export aliases in each original module (from .effect_meta import SPEED_KEYS # re-export: tests + external callers). Run the full suite — the golden test must be byte-identical (no XLO_REGEN_GOLDEN).
  2. Refine thresholds into tuning.py. Move REGRESS_MARGIN/STALL_LIMIT/REFINE_SKIP_OBJECTIVE (+ PHRASE_BARS/CELL_BARS/MOTION_SHARE_MIN) under a new # -- refine loop control section, keeping each constant's provenance comment. Re-export from run.py.
  3. Extract the pure guards. should_skip_refine, plateau_signature, design_implicated, refine_skip_objective into pipeline/refine_loop.py; call them from the still-monolithic loop. Add direct unit tests for each (no loop harness needed).
  4. Extract the stateful guards. BestTracker (revert/keep/gain/stall — port the exact comparison operators: revert is < best - margin, gain is > best + margin, held objective keeps but doesn't reset stall) and EscalationLedger. Unit-test the boundary cases: obj == best - margin keeps, obj == best + margin keeps-without-gain, stall reset only on gain.
  5. Extract collaborators + the loop. ReportBuilder, IterationRecorder, apply_revisions, then move the loop body itself; run.py keeps _refine_loop = refine_loop, regenerate_section (shared with xlo regen via pipeline/regen.py) and the checkpoint functions (_auto_checkpoint is imported by cli.py:13).
  6. Verify + docs. Full suite (65 test files) green with an unchanged golden fixture; update docs/architecture pipeline notes; note the new module in the roadmap scorecard. Consider an OpenSpec change doc (refactor-refine-loop) since show-refinement is a spec'd capability — the spec's behavior is unchanged, so tasks are code-motion only.

5. Testing strategy

  • Behavior-preservation: the entire existing suite runs unmodified at every step — especially test_refine.py, test_visual.py, test_design_escalation.py, test_revision_log.py (loop consumers) and test_golden_pipeline.py (generate stage; the constants moves touch generate-side modules). No golden regeneration is expected anywhere in this change; if step 1 requires one, the table transcription has a bug.
  • New unit tests (hermetic, no agents): tests/test_refine_guards.py covering BestTracker boundary arithmetic, plateau_signature equality/inequality (issue truncation at 64 chars), design_implicated (rules metric + effect-type mention required; wrong section / non-rules metric → False), EscalationLedger.should_escalate (once-per-run, repeat-offender path, implicated path), should_skip_refine (None disables; equality skips).
  • Import-compat test: a small test asserting the historical import paths still resolve (from xlights_orchestrator.pipeline.run import _refine_loop, REFINE_SKIP_OBJECTIVE; from xlights_orchestrator.pipeline.beats import SPEED_KEYS; from xlights_orchestrator.qa.rules import ENERGY_BAND, DURATION_CELLABLE) so a later cleanup can't silently break cached-workflow scripts.
  • Table-integrity test: assert the derived views from EFFECT_META equal a frozen copy of today's literals (checked in once, then deleted after a release or kept as the drift guard) — this is what makes step 1 mechanically safe.

8. Open questions

  1. tuning.py vs a separate refine_tuning section/module. The roadmap says “into pipeline/tuning.py”; tuning.py's docstring currently scopes itself to artistic dials. Amend the docstring (recommended — one file to look in), or add a sibling module?
  2. Where does effect_meta.py live? All consumers are orchestrator-side today, but xlights_core.knowledge already owns corpus-derived effect knowledge (preset library, value curves, colors). If F-E layout onboarding ever needs effect metadata core-side, core is the better home; starting orchestrator-side is the smaller diff.
  3. Context object now or later? refine_loop keeps the 20-keyword signature for compatibility. A RefineContext dataclass would clean it up but forces edits in run_pipeline and every test — worth doing only if improve-musicality Phase 2 adds more parameters anyway.
  4. Should _design_implicated's substring match (e in f.detail, run.py:169) become structured? Findings could carry an effect_type field instead of prose matching — a small semantic improvement, but it changes Finding and is out of behavior-preserving scope. Defer, note for improve-musicality.
  5. Archive the pending changes first? add-refine-skip-high-objective is implemented-but-unarchived and specifies lines this change moves. Archiving it before I3 keeps the OpenSpec history clean.
