1. Why

The codebase is fully annotated (a June-roadmap R1 deliverable), but CI runs mypy with continue-on-error: true — the workflow comment itself says the backlog "is being triaged before this becomes a hard gate." Advisory type-checking decays: nobody reads a red-but-ignorable check, new findings pile onto the backlog invisibly, and the annotations slowly drift from the truth they claim. The roadmap's assessment is blunt: "findings accumulate until the signal is noise."

Concretely, the backlog already contains real smells, not just pedantry: agents/panel.py types the agent field as bare object and then calls .run() on it; pipeline/generate.py dereferences st.show_plan.sections eleven times on a field typed ShowPlan | None; models/registry.py builds AnthropicModelSettings via an untyped **kw expansion into a TypedDict — exactly the construction that silently broke when Opus rejected sampling params (the comment at registry.py:60 memorializes a live 400). A hard gate turns this class of drift into a red PR.

Separately, none of the three packages ships a py.typed marker (PEP 561), so even a fully-clean codebase presents as untyped to any downstream consumer — including our own cross-package boundary: when mypy checks xlights-orchestrator against an installed xlights-core (rather than the workspace source layout), core's annotations are invisible without the marker.

2.4 Packaging (why py.typed is currently absent everywhere)

All three packages build with hatchling and enumerate their package dir explicitly, e.g. packages/xlights-core/pyproject.toml:

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[tool.hatch.build.targets.wheel]
packages = ["src/xlights_core"]

Hatchling includes all files under a listed package directory in the wheel (not just *.py — this repo already relies on that: models/config.yaml and the agents/prompts/ guides ship as package data). So shipping the marker is literally touch src/<pkg>/py.typed per package — no build-config change needed. A one-line check in CI (below) guards against a future packaging refactor silently dropping it.

3. Proposed design

3.1 Gate semantics: zero-findings, not a ratchet

With only 41 mechanical findings, a baseline/ratchet system (mypy-baseline files, per-module overrides that get deleted over time) is overkill and leaves ignore-debt around. The design is: burn the list to zero in one change, then flip CI to hard-fail in the same PR. The gate is "mypy exits 0", full stop. Inline # type: ignore[code] is permitted only with a same-line reason comment, and warn_unused_ignores (below) makes stale ignores themselves errors, so ignore-count cannot silently grow.

3.2 Config hardening (modest, deliberate)

# root pyproject.toml — PROPOSED [tool.mypy]
[tool.mypy]
python_version = "3.11"
files = ["packages"]
ignore_missing_imports = true        # keep: librosa/vamp/demucs/lyricsgenius have no stubs
warn_unused_ignores = true           # NEW: stale `# type: ignore` becomes an error
warn_redundant_casts = true          # NEW: same for casts
check_untyped_defs = true            # NEW: bodies of any未annotated defs still get checked
# NOT enabled (deliberately): strict, disallow_untyped_defs, disallow_any_*.
# The gate freezes today's honesty level; strictness增量 are separate, per-package steps.

(Replace the two placeholder words with plain English in the actual file — shown here to flag they are comments.) Rationale for not jumping to disallow_untyped_defs: the roadmap claims full annotation, but enforcing it is a separate, noisier diff; the gate's value is stopping new drift, which the zero-findings rule already does. Add types-PyYAML to the dev group so the yaml stubs are present regardless of which transitive dependency currently drags them in (today's clean result for registry.py:15 is accidental).

3.3 py.typed for all three packages

  • packages/xlights-core/src/xlights_core/py.typed (empty file)
  • packages/xlights-orchestrator/src/xlights_orchestrator/py.typed
  • packages/xlights-mcp/src/xlights_mcp/py.typed

Effects: (a) PEP 561 consumers (including editors and any external user of xlights-core) see the packages as typed; (b) if the workspace ever switches from source-tree checking to checking against built wheels, cross-package types keep resolving; (c) it is an implicit contract that the types are maintained — which the hard gate now guarantees. Verification that the marker actually ships: uv build --package xlights-core && unzip -l dist/xlights_core-*.whl | grep py.typed.

  2. Category-A burn-down (Optional State): add require() to pipeline/state.py (signature: def require(value: T | None, name: str) -> T); apply in generate.py (realize_section, generate_instructions), run.py:131,257,261, regen.py, cli.py:32. ~18 findings.
  3. Category-B seams: panel.py Protocol + gather narrowing; registry.py:61 literal TypedDict construction; revision_log.py RevisionSink Protocol + run.py:459 annotation. ~4 findings.
  4. Mechanical remainder: the per-file fixes from §2.3 (annotations, or 0/or {} guards, assert proc.stdin is not None, the layout.py variable rename, qa/__init__.py dict type). ~19 findings.
  5. Run the full suite (uv run --no-sync pytest, then the golden pipeline specifically) — every fix above is intended as zero-behavior; the suite is the proof. Run ruff check.
  6. Add the three py.typed markers; verify wheel inclusion via uv build + unzip -l for each package.
  7. Harden [tool.mypy] (§3.2) and fix anything the new warnings surface (expected: near-zero; warn_unused_ignores may flag leftovers from step 2–4).
  8. Flip CI (§3.4) and rewrite the stale advisory comments in ci.yml and pyproject.toml.
  9. Optional coverage step (§3.5) as a separate commit so it can be dropped independently.
  10. Docs/spec hygiene: note the gate in docs/roadmap-2026-07.md's scorecard on next revision; this item needs no OpenSpec capability delta (tooling-only), matching how R1 (CI itself) shipped.

5. Testing strategy

The "tests" here are mostly the existing suite plus CI mechanics; hermetic-first as always (-m "not live"):

  • Behavior-preservation: the full hermetic suite (65 test files) must pass unchanged after every burn-down commit. The golden pipeline snapshot (tests/test_golden_pipeline.py) is the strongest guard: require() narrowings and guard reorderings must not perturb any artifact byte.
  • New unit tests where the fix adds runtime semantics:
    • require(None, "show_plan") raises RuntimeError naming the field; require(x, ...) is identity — 4-line test in tests/test_orchestrator.py.
    • registry._settings() still yields the exact settings dict for a thinking+effort role and None for a bare one (pins the TypedDict restyle; extend the existing registry coverage or add to I6's planned test_registry).
    • panel.run_panel with a fake raising analyst still drops-and-continues (existing tests cover; re-run).
  • Gate self-test: after the flip, verify locally that an injected error (e.g. temporarily add x: int = "a") fails uv run --no-sync mypy with exit ≠ 0 — i.e. the workflow step actually gates (guards against a silent files= misconfiguration checking nothing).
  • Environment parity check: a one-line make/CI assertion that mypy checked ≥ 87 files (mypy ... | tail -1 parse or --no-error-summary counterpart) — protects against the §2.2 failure mode where a thinner venv quietly checks fewer modules and reports a different truth. (Optional; cheap.)
  • Wheel test: uv build each package and assert py.typed present (script or manual checklist in the PR; not a permanent CI step unless packaging churn recurs).

7. Acceptance criteria

  • ☐ uv run --no-sync mypy exits 0 in the CI environment (all 87+ source files checked, 0 errors).
  • ☐ .github/workflows/ci.yml has no continue-on-error on the mypy step; the "advisory" comments in ci.yml and pyproject.toml are rewritten to describe the hard gate.
  • ☐ A deliberately-injected type error makes the CI job fail (verified once during rollout).
  • ☐ py.typed exists in all three packages' source trees and is present in each built wheel.
  • ☐ warn_unused_ignores and warn_redundant_casts enabled; any # type: ignore in the tree carries an error code and a reason.
  • ☐ Full hermetic pytest suite and the golden pipeline snapshot pass unchanged; ruff check clean.
  • ☐ No new runtime dependencies; only types-PyYAML (and optionally pytest-cov) added to the dev group.
  • ☐ (If coverage step adopted) coverage summary visible in the PR job summary with no threshold enforcement.

8. Open questions

  1. State-typing altitude: require() is the minimal fix, but the deeper design question is whether State should grow typed accessors (State.plan → ShowPlan, raising) or even split into per-stage dataclasses (pre-design vs post-design). That intersects I3's refine-loop extraction — propose deciding there, keeping I4 minimal.
  2. Type-check tests too? files = ["packages"] excludes tests/. Checking tests would catch fixture drift (I6's schema-drift concern) but adds a large annotation surface of intentional fakes. Proposal: out of scope for I4; reconsider with I6.
  3. xlights-mcp strictness: at ~240 lines and 0 current findings, should it jump straight to disallow_untyped_defs = true via a per-package override as the pilot for gradual strictification? Cheap; low blast radius.
  4. mypy vs. adding pyright: editors mostly run pyright, whose inference differs (it would likely flag more). One checker as the gate is enough; is pyright-advisory worth a follow-up look, or noise?
  5. Coverage destination: job summary only (proposed) vs. an artifact XML for future trend tooling — decide when F-G dashboard scoping happens; artifacts are cheap to add later.
