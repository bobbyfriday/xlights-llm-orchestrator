1. Why

Every full pipeline run today requires a live, GUI-visible xLights instance reachable over its automation REST API (XLightsClient, packages/xlights-core/src/xlights_core/client.py). That single dependency has three costs:

  • No CI full-pipeline evals. The hermetic tests fake the client (see §2.4), so CI exercises the orchestration logic but never a real render. A golden test can freeze instruction JSON (tests/test_golden_pipeline.py), but nothing in CI can answer "does this change make the show better?" — the eval loop (F-G/F-H) is capped at instruction-level diffs.
  • No cloud execution. The run must be co-located with a desktop app; today that is a macOS machine with xLights open (the .fseq is even written into the macOS sandbox container, pipeline/media.py:21 — SANDBOX_DATA = Path.home() / "Library/Containers/org.xlights/Data").
  • Slow refine iterations. Each refine iteration re-emits every effect over REST — one getModels plus one addEffect round-trip per placement (editing.py:82 and :85), then a full renderAll (effect_emitter.py:134, write timeout 300 s, client.py:35) and a saveSequence flush before QA can sample coverage (run.py:174-179). With ~270+ effects per show this is minutes of serialized REST traffic per iteration.

The tantalizing fact: an offline renderer that needs no xLights already exists (xlights_core.preview) and already feeds the visual critic and the coverage sampler. The core question this spike must answer honestly: what in the loop truly requires live xLights? The answer, previewed here and verified in §2: only two things — the once-per-layout targetability probe, and the render itself. xLights renders effects (settings strings in the .xsq) into per-node pixels (the .fseq); the offline PreviewRenderer only replays an already-rendered .fseq. Everything else is either already file-based or trivially fakeable. The render is the crux, and this doc analyzes the three ways around it.

2. Current state

2.1 What the offline PreviewRenderer actually does

packages/xlights-core/src/xlights_core/preview/ is a three-file, xLights-free replayer:

  • fseq.py — load_fseq() (line 24) parses FSEQ v2 (PSEQ magic, line 28): fixed header (channels, frame count, step ms), a compression-block index, and either uncompressed channel data or zstd blocks (lines 42–73). It returns a dense uint8 [num_frames, channels] array — the exact per-node RGB of every frame, as rendered by xLights.
  • layout.py — parse_controllers() (line 57) walks xlights_networks.xml to build absolute channel ranges; parse_models() (line 113) walks rgbeffects.xml top-level <model> elements (Default preview only, line 126) and model_world_pixels() (line 248) places every pixel in world coordinates with per-DisplayAs geometry: Arches, Single Line, Custom grids, Horiz/Vert Matrix (with strand zig-zag), Cube, Star, Tree — everything else degrades to a line.
  • render.py — PreviewRenderer (line 29) precomputes an orthographic projection once, then serves render_frame() (still PNG, line 84), render_clip() (silent MP4 via an ffmpeg pipe, line 93), and brightest_frame_ms() (line 130, the "representative moment" picker the critique uses). Overlapping pixels take the max, not last-write (np.maximum.at, lines 73–77).

What it does not do — and this is the whole crux — is render effects. It has no notion of an effect, a settings string, a value curve, or a buffer style. Input is channel data that xLights has already rendered. Its fidelity limits versus the xLights house preview are also known: point-splat pixels (no bulb size/glow), no background image, approximate geometry (e.g. arch peak height is a 0.5 × span heuristic, layout.py:192), and only the Default layout group.

2.2 RealRender: why the offline render is not trusted as ground truth

pipeline/visual.py:155-158 — quoted verbatim:

class RealRender:
    """The REAL xLights render, exported via automation (media-attached sequences ONLY — a
    media-less export crashes xLights). Cached by the `.xsq` mtime; serves offset-corrected
    frames/clips via ffmpeg. Everything best-effort: consumers fall back to the offline renderer."""

It exists because the offline replay differs visually from what xLights shows (roadmap I8: "fseq metrics inherit offline-render fidelity (the reason RealRender exists)"). Its guardrail is the media-attach quirk — visual.py:171-174:

seq = await client.get_open_sequence()
if not (seq.get("media") or "").strip():
    return False                              # GUARDRAIL: never export media-less

backed by client.py:258-260: export_video_preview() "REQUIRES a media-attached sequence — exporting a media-less sequence crashes xLights with a bitrate-0 null-deref". And media cannot be attached live at all — effect_emitter.py:70-72 (docstring of apply_instructions):

Create a clean-slate ANIMATION sequence (replacing any open one) and additively place
the instructions. Audio is NOT attached here — `newSequence(mediaFile)` crashes xLights
(LoadAudioData modal); the song is attached by an offline `.xsq` patch at finalize instead.

An important consequence, verified by reading the flow (confirm live in the spike): because the in-run sequence is always a media-less ANIMATION (media is patched offline only at finalize, finalize.py:26-33, i.e. after the refine loop), the RealRender guardrail returns False mid-run and both the visual critique (visual.py:79, use_real) and the coverage sampler (visual.py:124-128) fall back to the offline PreviewRenderer. In other words, the refine loop's in-flight decisions already run on offline renders today — the live app's irreplaceable contribution mid-run is producing the .fseq, not the pictures.

Classified by when they matter:

  • Layout-setup-once (cached): the targetability probe. groups.py:53 caches per layout fingerprint (_fingerprint, lines 28–32, keyed on group and model names) — so this is a once-per-layout dependency, not per-run. A headless run against a known layout only needs the cache file targetable_groups_<fp>.json pre-seeded.
  • Per-run bookkeeping: get_show_folder, finalize save/close. All resolvable to plain paths/files in a headless mode.
  • Per-emit / per-refine-iteration — the real dependency: sequence lifecycle (close/new), placement (get_models+add_effect per instruction), render_all, save_sequence. Of these, only render_all is genuinely hard: placement is just building an effect list, and the .xsq is XML we already know how to patch — but only xLights can turn that XML into channel data.

2.4 What is already offline, and what the test fakes already cover

  • finalize.py is the existence proof for offline mutation. Media attach, canonical render order, and timing tracks are all patched into the saved .xsq with the file released — finalize.py:26: # patch the .xsq OFFLINE (live edits crash xLights), calling patch_xsq_media (media.py:70), patch_xsq_render_order, patch_xsq_timing_tracks. The .xsq format is fully writable by this codebase.
  • The offline renderer already carries QA and critique. The coverage sampler (make_lit_sampler, visual.py:105) and the visual critique both consume the .fseq through PreviewRenderer; per §2.2 this is the path actually exercised mid-run.
  • A fake client for CI already exists — several, in fact. tests/test_golden_pipeline.py:129 (_FakeClient with only get_group_names/get_show_folder), tests/test_refine.py:93 (only close_sequence, paired with a fake emitter), plus per-area fakes in test_targetable_groups.py:17, test_orchestrator.py:95, test_value_curves.py:43. These prove the logic runs headless today — but the fake emitter returns {"placed": ..., "rendered": True} without producing any channel data, so nothing downstream can measure the show. Fakes cover logic; they cannot cover rendering.

3. Proposed design

Headless mode needs a source of rendered .fseq data that does not require a human-attended GUI. Three options, then the spike that decides among them.

3.1 Option (a) — xLights batch / CLI render

What the repo actually says (verified): the automation surface has a batchRender command that this project has deliberately deferred, twice: openspec/changes/archive/2026-06-08-add-xlights-mcp-effect-editing/proposal.md:13 lists "batchRender" under non-goals, and openspec/changes/archive/2026-06-08-add-visual-critic/design.md:38-39 concluded "no batchRender, no extra render step — the .fseq is a side effect of the render+save we already do (renderAll is also safe/fast, ~2s)". So the repo has evidence that a batch-render concept exists in xLights, and evidence that renderAll on the current layout is cheap — but it has never run xLights without a GUI.

Not verified in this repo — must verify in the spike: xLights is known to ship a command-line render mode on some platforms (an xlights -render <seq.xsq>-style invocation / batch render entry point), and Linux builds exist (AppImage). Whether it (1) exists in the current release, (2) runs without a display or needs Xvfb/offscreen wx, (3) produces a .fseq equivalent to the GUI's renderAll+saveSequence, and (4) how long it takes, are precisely the spike questions. Do not design against it until S3 answers them.

Shape if it works (proposed, does not exist): a file-based emitter that writes effects straight into a .xsq (extending the finalize.py/media.py XML-patching pattern to effect placement — the .xsq stores the same settings/palette strings place_preset assembles, editing.py:63-78), then shells out to the CLI render:

# Proposed sketch — signatures only, contingent on S3
async def apply_instructions_headless(
    workdir: Path, instructions: list[EffectInstruction], *,
    duration_secs: int, layout: LayoutFiles, renderer: BatchRenderer,
) -> dict[str, Any]:
    """Write effects into a .xsq offline, invoke `xlights -render`, return the applied report."""

class BatchRenderer(Protocol):
    async def render(self, xsq: Path) -> Path: ...   # returns the .fseq path

Risks: writing .xsq effect XML ourselves replaces the live addEffect validation (targetability, worked=false on overlap — client.py:234-244) with our own correctness; the targetability probe stays a live-once-per-layout step; layer/row semantics must match what addEffect produces. This is real work (M–L on its own) — which is why it is a spike question, not a commitment.

3.2 Option (b) — reimplement effect rendering in Python: non-starter

Rendering the .xsq ourselves means reimplementing the xLights render engine: hundreds of effect types (the mined catalog alone spans dozens of stock types, each with its own settings vocabulary — see the E_*/B_* keys assembled in editing.py:63-78), value curves with their own DSL (knowledge/value_curves.py), per-model vs per-preview buffer styles (pipeline/render_style.py's resolve_buffer_style, and the roadmap's Horizon-3 note that a wrong buffer mode "silently breaks every sweep"), layer blend modes, transitions, group-canvas rendering, and per-version behavior drift in upstream C++. Exact parity is unattainable and approximate parity is worse than useless here — the whole point of the render is to be the ground truth the offline replay is measured against. Any divergence would be invisible until a human watches the real show. Permanently chasing the upstream renderer is an unmaintainable liability for a two-package Python project. Rejected without a spike task.

3.3 Option (c) — hybrid: cache a rendered .fseq, iterate offline on it

Keep one live render per "look generation", then iterate offline against the cached .fseq. Honest analysis of what that supports, given that the .fseq is a flat [frames, channels] array with every effect already baked in:

Iteration activityOffline on cached .fseq?Why
QA / objective score (coverage sampler), I8 Tier-0 fseq metrics, offline critique stills/clips, judge/checkpoint decisionsYesPure reads of channel data — this is exactly what runs today (§2.2)
Muting/dimming a group or a time window (drop an effect, reduce brightness)PartialZero/scale the group's channel range over the window — channel ranges are known from layout.py; but layered effects on the same target are baked together, so "remove layer 2 only" is impossible
Time-shifting a whole section (re-timing)PartialFrame-slice copy; boundaries show discontinuities for motion effects; sub-section effect re-timing inherits the same problem
Retargeting an existing look to a different groupMostly noCopying channel ranges ignores the target's geometry/buffer — a matrix look pasted onto arches is garbage
New effects, changed knobs/settings, palette changes, buffer-style changes, section regenerationNoAll require the render engine. And section regen is what _refine_loop actually does — regenerate_section (run.py:124) produces brand-new EffectInstructions

Conclusion: the hybrid cannot make the current refine loop headless, because every accepted revision regenerates instructions and needs a re-render. What it can do is restructure the loop so that evaluation and decision-making (report → judge → checkpoint → revision selection) run offline and renders are batched: accumulate all of an iteration's section regens, do one live emit+render, refresh the cache. That is not "no xLights"; it is "xLights as a render server touched once per iteration" — which is roughly what the loop already does, minus the per-placement get_models chatter. The hybrid's genuine wins are (1) CI evals against pre-rendered fixture .fseqs (the whole I8 Tier-0 metric suite, judge, and QA run hermetically on a checked-in fixture), and (2) trusting offline pictures for inner-loop decisions with one real-render gate at the end — exactly the caveat I8 already states ("keep one real-render check near the end of a run as the ground-truth gate").

3.4 The spike plan

  1. S1 — measure the offline-vs-real fidelity gap on a fixture. Take one finished, media-attached show (post-finalize, so RealRender engages). Script: (1) RealRender.refresh() → house-preview MP4; (2) for each section, sample N=8 beat-aligned timestamps plus brightest_frame_ms; (3) render each timestamp both ways — PreviewRenderer.render_frame(t) and RealRender.frame_png(t) (offset-corrected, visual.py:201-206); (4) compute per-frame global and per-group deltas — per-group masks come from projecting each group's member-model pixels through the same _project() the renderer uses — mean brightness delta, lit-fraction agreement at the sampler's threshold (>30, visual.py:140), and hue-histogram distance. Output a small JSON report per fixture.
  2. S2 — client-call inventory. Done in §2.3 of this document; re-run the grep at spike time and diff against the table (the codebase moves fast — 63 changes in ~5 weeks).
  3. S3 — prototype xLights batch/CLI render on the target platform. On macOS first (known-good install), then Linux (AppImage + Xvfb): discover the actual CLI flags (xlights --help / upstream docs), render the S1 fixture's .xsq, compare the produced .fseq byte-for-byte and channel-for-channel against the REST-rendered one, and time it against renderAll+saveSequence over REST (repo baseline: "~2s" for renderAll on this layout, per the add-visual-critic design). Also test: does it run with no display at all, or only under a virtual display?

3.6 Recommendation

Run the spike; expect the hybrid + fallback to be the near-term reality. The honest prior: option (a) is the only true headless path and is worth one focused day of S3 evidence before betting on it — but it carries the hidden cost of reimplementing addEffect-side validation as .xsq writing, so even a "yes" from S3 makes it an M–L follow-up, not a quick win. Meanwhile two cheap results are available regardless of S3:

  • CI evals on fixture .fseqs now. Check in one rendered fixture (layout XMLs + a small .fseq); run QA, I8 Tier-0 metrics, and judge flows against it hermetically. This needs zero new render capability and unblocks most of the "CI full-pipeline eval" value.
  • Keep live xLights, but batch the interactions (the explicit fallback): cache the get_models() list per emit instead of one REST call per placement (editing.py:82 — with ~270 placements this halves emitter round-trips), reuse I8's tiering so critique/QA reads stay offline, and investigate section-scoped renders (xLights' automation render command takes the whole sequence today; a scoped render is another S3 question). The refine loop already renders only once per iteration, so the win here is placement chatter and avoided re-critique, not render count.

If S3 surprises positively (headless Linux render, equivalent .fseq), promote option (a) to a proper OpenSpec change: it flips cloud execution and true CI full-pipeline evals from "impossible" to "engineering".

4. Implementation plan (spike protocol)

  1. Fixture prep — pick one finished show (e.g. the current test song), run the normal pipeline with --save-as, finalize (media attached), keep: <seq>.xsq, <seq>.fseq, xlights_rgbeffects.xml, xlights_networks.xml, the staged MP3, and the revision log.
  2. S1 script — scripts/spike_fj_fidelity.py (proposed, spike-only): build PreviewRenderer(fseq, rgb, net) and a RealRender(save_as, duration_s) against a live xLights with the fixture open; emit fidelity_report.json with per-section, per-group {brightness_delta, lit_fraction_agreement, hue_distance}. Add a --group-masks debug mode that writes the per-group projected masks as PNGs for eyeball sanity.
  3. S1 decision replay — re-run qa.evaluate with a sampler backed by each render path over the fixture's iteration history; count objective-score and keep/revert disagreements.
  4. S2 verification — re-grep client\. across packages/xlights-orchestrator/src/ and xlights_core/editing.py; diff against §2.3.
  5. S3 prototype — document the exact xLights version and invocation tried; on Linux: AppImage under Xvfb; capture (a) exit behavior with no display, (b) the produced .fseq, (c) wall time. Compare .fseqs with a small numpy diff script reusing load_fseq (max/mean per-channel delta, count of differing frames).
  6. Write-up + decision point — a short findings section appended to this doc (or a sibling f-j-spike-findings note) applying §3.5; if option (a) passes, open an OpenSpec change for the file-based emitter + BatchRenderer; if only the hybrid passes, fold its work into I8 (shared fixture, shared metrics) and implement the fallback batching (get_models cache in place_preset — one-line seam: pass a prefetched name set through apply_instructions).

5. Testing strategy

Hermetic-first, per repo convention (all current tests run without xLights, an API key, or the network):

  • The fidelity measurement is itself a script, not a session. spike_fj_fidelity.py takes paths + a base URL, writes JSON; re-runnable on any future layout or xLights version. Its pure parts (mask projection, delta metrics, .fseq diff) get hermetic unit tests against a tiny synthetic .fseq (the load_fseq format is simple enough to write in a fixture builder — uncompressed variant, fseq.py:42-47).
  • Fixture .fseq in CI. Regardless of the option chosen, land the checked-in fixture (small: low channel count or a truncated frame range) and a test that runs the sampler + Tier-0-style metrics over it — this is the durable artifact of the spike and the seed of the CI eval suite.
  • Live steps are gated and batched, mirroring the archived changes' pattern ("Live (batched): ..." tasks): S1's real-render half and S3 run only when XLO_LIVE=1-style gating is present, never in CI.
  • No production code changes required for the spike except, if convenient, the get_models prefetch seam — which gets a hermetic test with the existing fake-client pattern (tests/test_palette_realization.py:54 already fakes placement).

7. Acceptance criteria

This is a spike: acceptance is questions answered with evidence, not shipped features.

  • [ ] A fidelity_report.json for at least one fixture show: per-section, per-group brightness/lit-fraction/hue deltas between PreviewRenderer and RealRender, plus the decision-agreement rate for the coverage sampler.
  • [ ] The §2.3 client-call inventory re-verified by grep at spike time, with any drift noted.
  • [ ] A definitive, version-stamped answer on xLights batch/CLI render: exact invocation, platform behavior (macOS, Linux, with/without display), .fseq equivalence numbers, and wall time vs REST renderAll.
  • [ ] Confirmation (or refutation) of the §2.2 inference that RealRender never engages mid-run on media-less sequences — observed on a live run's logs.
  • [ ] A written go/no-go against the §3.5 criteria, with the chosen follow-up (option (a) OpenSpec change, hybrid folded into I8, or fallback batching) named.
  • [ ] A checked-in fixture .fseq + layout pair and one hermetic test exercising the sampler/metrics over it (the spike's durable artifact).

8. Open questions

  • Does the automation REST batchRender command (deferred by the MCP change) render saved sequences without opening them in the UI — and would that be a lighter path than the process-level CLI, while still requiring a running GUI app?
  • Can renderAll be scoped to a time range or a set of models (for section-scoped re-renders in the fallback), or is whole-sequence the only granularity?
  • Where does a Linux xLights write the .fseq — show folder, or an XDG equivalent of the macOS container? (resolve_artifact needs the answer.)
  • Is the targetability cache portable across platforms/installs of the same layout, or does targetability itself vary by xLights version (would force a per-environment probe)?
  • If option (a) lands, does the emitter keep the REST path as the attended-mode default (live watching has UX value) with headless opt-in, or become the single path?
  • How large can a CI fixture .fseq be before repo bloat matters — truncate frames, reduce channels, or store zstd-recompressed?
