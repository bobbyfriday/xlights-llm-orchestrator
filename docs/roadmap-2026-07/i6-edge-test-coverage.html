1. Why

The deterministic core is thoroughly tested (56 test files, a golden pipeline snapshot, hermetic refine-loop tests), but the edges — the places where a human, a shell, an HTTP client, or an LLM provider touches the system — are not:

  • cli.py (120 lines) has zero tests. A flag rename (--max-iterations → anything), a broken default (save_as=None if args.no_save else …), or a checkpoint-wiring regression (checkpoint=_auto_checkpoint if (args.refine and args.auto) else None, cli.py:29) ships silently. The CLI is the only entry point a user actually types.
  • models/registry.py has zero tests. It is the single routing point for every LLM call — model_string(), the XLO_PROVIDER override, model_snapshot() (recorded into every revision-log line), and the Anthropic-only _settings() path with its “never set temperature/top_p/top_k (removed on Opus → 400)” invariant (registry.py:60). A config.yaml typo or a broken override currently surfaces as a live-run failure.
  • brief_editor.py's HTTP layer is untested. tests/test_brief_editor.py covers save_brief/render_page as pure functions but never exercises do_GET/do_POST/_handler — the request parsing, Content-Length handling, error-to-400 mapping, and 404 routes. F-I (live progress UI) plans to extend this exact pattern.
  • The MCP server (packages/xlights-mcp, ~240 lines, 15 tools) has zero tests — the whole package. Its tools are thin wrappers over XLightsClient, which tests/test_client.py already proves testable against httpx.MockTransport; the same technique covers the server.
  • No schema-drift guard for agent outputs. Ten pydantic models are LLM output_types, and three of them are also persisted cache artifacts (song_description → MusicBrief, creative_brief → ShowPlan, instructions → list[EffectInstruction], run.py:391–433). A field rename or a new required field breaks every cached brief on disk — today that failure appears mid-run (or worse, is silently swallowed by the “stale/invalid cache shape → recompute” branch at run.py:395, quietly discarding an expensive panel run).

2. Current state

2.1 Test-suite conventions (what new tests must follow)

  • Hermetic by default. pyproject.toml: addopts = "-m 'not live'" with one marker: live: requires a running xLights instance (opt in with -m live). CI runs plain uv run --no-sync pytest — anything needing network, xLights, or an API key must be marked live or faked.
  • Fake agents via PydanticAI TestModel. The canonical helper (test_golden_pipeline.py:122–126): 
    def _test_agent(output):
    from pydantic_ai import Agent
    from pydantic_ai.models.test import TestModel
    return Agent(TestModel(custom_output_args=output.model_dump()),
                 output_type=type(output), system_prompt="")
    Simpler fakes use SimpleNamespace(output=…) returned from an object with an async run() (test_refine.py:110–115).
  • Fake xLights via httpx.MockTransport + recorded JSON fixtures (test_client.py:32–40: XLightsClient(client=httpx.AsyncClient(transport=MockTransport(handler), base_url="http://xlights.test"))), or minimal duck-typed clients (test_refine.py::_FakeClient — only close_sequence; test_golden_pipeline.py::_FakeClient — only get_group_names/get_show_folder).
  • Isolation knobs: XLO_CACHE_DIR is monkeypatched to tmp_path wherever the pipeline runs; guide files load best-effort so tests run without them.

3. Proposed design

3.1 tests/test_cli.py — argparse wiring

What to fake: the async entry points, not the parser. main(argv) is already injectable (cli.py:67). Monkeypatch cli._run/cli._regen with capture coroutines, cli.load_env and cli.has_llm_key to no-op/True, and for edit-brief monkeypatch brief_editor.serve. Better still, for run: monkeypatch cli.run_pipeline and XLightsClient (an async-context-manager stub) so the test asserts the exact kwargs the CLI passes into the pipeline — the wiring is the product here.

What to assert:

  • xlo run --song s.mp3 → use_cache=True, refine=False, max_iterations=3, log_revisions=True, timing_tracks=True, save_as=safe_name("s.mp3"), attended checkpoints (interpret_checkpoint=_interpret_review, design_checkpoint=_design_review, checkpoint=None).
  • --auto → both review gates None; --refine --auto → checkpoint=_auto_checkpoint; --refine alone → checkpoint=None (interactive default inside the loop) — the three-way matrix at cli.py:27–29 is the likeliest silent-regression site.
  • --no-save → save_as=None; --name X → save_as="X"; --no-cache/--no-log/--no-timing-tracks flip their booleans; --max-iterations 7 parses as int.
  • Guard exits: missing LLM key → SystemExit with the “No LLM key found” message (monkeypatch has_llm_key → False); edit-brief with neither --song nor --brief → SystemExit; nonexistent brief path → SystemExit mentioning “run `xlo run`”.
  • regen: --list (and omitted --section) prints sections via format_sections and does not require a key; FileNotFoundError/IndexError from _regen → SystemExit(str(exc)) (cli.py:113–116); a bad subcommand → argparse error (exit code 2).

3.2 tests/test_registry.py — model routing

What to fake: nothing external — the registry reads a YAML file and env vars. Two approaches: (a) test against the real config.yaml for invariants (every role has an entry for every provider named in providers:; model strings are prefixed anthropic:/google:), and (b) point _CONFIG at a fixture YAML for behavior tests. Because _cfg() is @functools.lru_cache (registry.py:21), fixture-based tests must call registry._cfg.cache_clear() in setup/teardown — encode that in a pytest fixture so the hazard is handled once.

What to assert:

  • active_provider(): default from YAML; XLO_PROVIDER=gemini (monkeypatch.setenv) wins.
  • model_string("judge") = "anthropic:claude-opus-4-8" under the default; "google:gemini-3.5-flash" under gemini; unknown role/provider → KeyError (documenting today's failure mode).
  • model_snapshot() returns all six roles under both providers, values shaped provider-prefix + model (the revision log and F-G depend on this shape).
  • _settings("director") → AnthropicModelSettings with anthropic_thinking={"type": "adaptive"} and anthropic_effort="high"; _settings("generator") has thinking but no effort; any role under provider="gemini" → None; and critically: no sampling keys ever present (the Opus-400 invariant, registry.py:60).
  • build_agent("judge", output_type=JudgeVerdict, system_prompt="x") constructs without network/API key (lazy — mirrors the comment at registry.py:65) and carries the expected model string.

3.3 tests/test_brief_editor_http.py — the stdlib HTTP form

What to fake: nothing — ThreadingHTTPServer(("127.0.0.1", 0), _handler(...)) on an ephemeral port in a daemon thread is hermetic (loopback only, port 0; no webbrowser since we build the server directly rather than calling serve()). Use urllib.request against http://127.0.0.1:{port}; tmp_path holds creative_brief.json + schema.

What to assert:

  • GET / → 200, text/html, body embeds the brief JSON and schema (the __BRIEF__/__SCHEMA__ substitutions) and the escaped path; GET /nope → 404 JSON.
  • POST /save with a valid edited brief → 200 {"ok": true}, the file on disk contains the edit, keeps $schema first and preserves un-rendered fields (group_motifs) — the file-level assertions reuse the existing unit test's brief fixture.
  • POST /save with a structurally invalid brief (e.g. sections not a list) → 400 with an error body truncated to ≤300 chars (brief_editor.py:201), and the file is untouched (the atomic .tmp-replace never ran).
  • POST /save with missing/zero Content-Length → 400, not a hang (the int(self.headers.get("Content-Length", 0)) path).
  • serve()-level: with the schema file absent, the page renders with SCHEMA={} (text-only form fallback, brief_editor.py:209–210) — testable via render_page composition or a handler built with schema={}.

3.4 tests/test_mcp_server.py — FastMCP tools against a fake client

What to fake: the Context and the client. The tools resolve their client via _client(ctx) → ctx.request_context.lifespan_context["client"] (server.py), so a stub is two SimpleNamespace layers deep:

def _ctx(client):
    return SimpleNamespace(request_context=SimpleNamespace(
        lifespan_context={"client": client}))

The fake client can be either (a) a duck-typed object with just the awaited methods (the test_refine.py style — simplest), or (b) a real XLightsClient over httpx.MockTransport reusing tests/fixtures/*.json (the test_client.py style — exercises serialization too). Recommend (a) for tool-logic tests and one (b)-style smoke test to prove the composition. Call the tool functions directly (await server.xl_get_models(_ctx(fake))) — FastMCP's @mcp.tool() leaves the underlying coroutine callable; if a FastMCP version pins the wrapper, fall back to mcp._tool_manager lookup or keep module-level plain functions registered explicitly (a one-line server refactor).

What to assert (grouped):

  • Read tools (xl_get_version, xl_get_show_folder, xl_get_models, xl_get_model, xl_get_controllers): pass-through shape — e.g. xl_get_models returns {"models": [...], "groups": [...]} from get_model_names/get_group_names; xl_get_model/xl_get_controllers return model_dump()ed dicts.
  • Error translation (_call): a fake raising XLightsConnectionError("down") surfaces as RuntimeError("XLightsConnectionError: down") — the typed-name contract MCP clients rely on; same for KnobValueError, ValueError, KeyError.
  • Write tools: xl_new_sequence forwards duration_secs/frame_ms/media_file/force verbatim (the force=True discard-guard is opt-in — assert default False reaches the client); xl_close_sequence forwards force/quiet; xl_save_sequence passes name=None through.
  • xl_add_effect_raw gates: end_ms <= start_ms → ValueError("bad timing…") before any client call; target not in get_models() → ValueError; client returning worked=false → RuntimeError("PresetPlacementError: …").
  • xl_add_effect/xl_validate_preset: monkeypatch server.place_preset/server.validate_preset to capture args (they're imported into the server's namespace, so patching the module attribute works) — assert knob/palette/layer forwarding.
  • Audio tools: with the import failing (monkeypatch xlights_core.audio import to raise), xl_analyze_song raises RuntimeError("audio extra not installed: …") — not an ImportError traceback.

3.5 The output-schema drift guard — tests/test_schema_drift.py + tests/fixtures/agent_payloads/

Mechanism: one checked-in, known-good JSON payload per schema, validated against the current model. The payloads are frozen artifacts — they never auto-regenerate; updating one is a deliberate act that stands in for “I am knowingly invalidating cached briefs on disk.”

tests/fixtures/agent_payloads/
  show_plan.json          section_plan.json      section_effects.json
  music_brief.json        structure_out.json     rhythm_out.json
  harmony_out.json        lyric_out.json         judge_verdict.json
  visual_findings.json    instructions.json      song_analysis.json
  revision_log_record.json
# tests/test_schema_drift.py (proposed)
CASES = [
    ("show_plan.json", ShowPlan), ("music_brief.json", MusicBrief),
    ("judge_verdict.json", JudgeVerdict), ("visual_findings.json", VisualFindings),
    ("section_effects.json", SectionEffects), ("section_plan.json", SectionPlan),
    ("structure_out.json", StructureOut), ("rhythm_out.json", RhythmOut),
    ("harmony_out.json", HarmonyOut), ("lyric_out.json", LyricOut),
    ("song_analysis.json", SongAnalysis), ("revision_log_record.json", RevisionLogRecord),
]

@pytest.mark.parametrize("name,model", CASES)
def test_known_good_payload_still_validates(name, model):
    payload = (FIXTURES / "agent_payloads" / name).read_text()
    model.model_validate_json(payload)          # a breaking model change fails HERE, not mid-run

def test_instructions_cache_shape():            # the list-of-dicts cache (run.py:427–429)
    items = json.loads((FIXTURES / "agent_payloads" / "instructions.json").read_text())
    [EffectInstruction.model_validate(x) for x in items]

Payload provenance: seed each fixture from a real cached artifact (a redacted song_description/creative_brief/instructions from the cache dir, trimmed to 1–2 sections) so the fixtures carry realistic optional-field usage — not from model_dump() of a freshly constructed object, which would only contain defaults and follow renames automatically. A second, cheaper assertion catches additive drift: compare model.model_json_schema() required-field sets against a small frozen manifest (required_fields.json) so “new required field” fails even if the payload happens to include it.

Failure protocol (documented in the test's docstring): when this fails you either (a) revert to a backward-compatible change (new field with default), or (b) intentionally bump: update the fixture and bump the cache key (the established pattern — see run.py:390 “Cache key bumped from music_brief so old flat briefs don't shadow the richer one”). Option (b) without a cache-key bump is the exact bug this guard exists to catch: the silent except Exception → recompute at run.py:394–396 for briefs, and a hard mid-run crash for creative_brief/instructions which validate without a try/except (run.py:411, 427–429).

3.6 CI coverage reporting

Advisory-first, matching the mypy posture (I4):

  • Add pytest-cov to [dependency-groups] dev; run uv run --no-sync pytest --cov=packages --cov-report=term-missing:skip-covered --cov-report=xml in .github/workflows/ci.yml.
  • Publish visibility without external services: a job-summary step (coverage report ≫ $GITHUB_STEP_SUMMARY) + upload coverage.xml as an artifact. No threshold initially — the roadmap's point is “the trend is visible in PRs”. A soft gate (--cov-fail-under) can come after a few weeks of baseline.
  • Expected effect: the four new test files convert today's 0%-covered edges (cli.py, registry.py, xlights_mcp/server.py, brief_editor handlers) into visible numbers, and the report highlights the next gaps (e.g. pipeline/media.py, config.py).

4. Implementation plan

  1. test_registry.py (smallest, zero fakes): real-config invariants + fixture-config behavior + the lru_cache clear fixture. ~15 tests.
  2. test_cli.py: capture-kwargs harness for run/regen, the checkpoint matrix, guard exits. ~12 tests. (If main() resists patching cleanly, a 3-line refactor extracting build_parser() is acceptable; do not restructure command dispatch.)
  3. test_brief_editor_http.py: ephemeral-port server fixture (yields base URL, shuts down in teardown), 6–8 request/response tests.
  4. test_mcp_server.py: _ctx helper + duck-typed fake client; read-tool shapes, _call translation, raw-effect gates, forwarding checks; one MockTransport composition smoke test. May require adding the mcp dependency to the root test environment (it's a workspace member — uv sync already installs it; verify CI does).
  5. Schema-drift guard: harvest + redact payload fixtures from a real run's cache, write test_schema_drift.py + required_fields.json, document the failure protocol in the module docstring.
  6. CI coverage: pytest-cov + summary + artifact; confirm runtime stays acceptable (coverage adds ~10–20%).

5. Testing strategy

This item is tests; the strategy is about keeping them within house rules:

  • Hermetic: no test opens a non-loopback socket, reads a real API key, or touches the user cache (XLO_CACHE_DIR → tmp_path where relevant). The brief-editor server binds 127.0.0.1:0 only. Nothing new is marked live.
  • Deterministic: registry tests clear the _cfg lru_cache around fixture-config use so ordering can't leak; CLI tests never spawn an event loop beyond asyncio.run on stub coroutines.
  • Anti-tautology: drift-guard fixtures are hand-frozen JSON, never regenerated by the test itself (contrast with the golden pipeline's explicit XLO_REGEN_GOLDEN escape hatch — mirror that: an XLO_REGEN_PAYLOADS=1 helper script is acceptable, an automatic rewrite is not).
  • Independence: each of the six steps lands as its own small PR; none depends on another.

7. Acceptance criteria

  • [ ] tests/test_cli.py exists; renaming any xlo run/regen/edit-brief flag or changing a default breaks a test; the --auto/--refine checkpoint matrix is pinned.
  • [ ] tests/test_registry.py exists; XLO_PROVIDER override, per-role model strings for both providers, model_snapshot() shape, and the no-sampling-params invariant are asserted.

8. Open questions

  1. Where should MCP tests live? tests/ is the workspace-root suite (single pytest run — simplest, matches today); a per-package packages/xlights-mcp/tests/ would be cleaner packaging but adds a second pytest target to CI. Recommend workspace root for now.
  2. Drift-guard scope for SongAnalysis: it's a large model (beats, stems, lyrics dict, instrumentation). A full fixture is bulky (~100 KB); a trimmed one may miss fields. Trim to one beat/segment/stem each and rely on the required-fields manifest for the rest?
  3. Should the drift guard also pin enum vocabularies (e.g. Decision.action, JudgeVerdict.verdict literals) that the revision log and checkpoints string-match on? Cheap to add to the manifest; slightly widens the “deliberate update” surface.
  4. Coverage backend: plain artifact + step summary (zero external deps) vs Codecov (PR annotations). Start with the former; revisit with F-G.
  5. test_orchestrator.py overlap: some pipeline-level wiring may already be asserted there — audit before writing test_cli.py to avoid duplicating pipeline-behavior assertions at the CLI layer (CLI tests should stop at the kwargs boundary).
