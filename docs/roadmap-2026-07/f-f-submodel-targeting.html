1. Why

Everything the orchestrator places today lands on whole models or groups of whole models. That caps choreography resolution at the prop: the mega tree is one target, the roofline is one line, a window frame is indivisible. The community craft this forecloses is spelled out in the repo's own guides:

"Split into submodels (vertical halves, rings) for drum-kit mapping and call-response." — xlights-sequencing-guide.md, line 179 (mega tree)

"Window/door submodels can pulse independently as rhythm cells." — xlights-sequencing-guide.md, line 198

"Submodels — finest-grained accents (tree topper, window frames, spinner rings)" — xlights-layering-rendering-guide.md, line 144 (the layering hierarchy's bottom tier)

"Defining submodels (tree rings/slices, spinner arms, window frames) is what unlocks surgical layering — and HD-prop vendors ship effects targeted at their submodel definitions … We have no documented submodel scheme yet." — xlights-layering-rendering-guide.md, line 272

Concretely, submodel targeting enables three families of gesture the current pipeline cannot express:

  • Tree zones/rings as a drum kit — kick on the bottom ring, snare answering on the mid band, cymbal hits on the topper: the instrument-mapped groove overlay (pipeline/beats.py::place_beat_accents) gains vertical structure inside its hero prop instead of flashing the whole cone.
  • Vertical build runs — a riser that climbs the tree ring by ring into a drop, the pixel-level version of the value-curve builds the guides call the workhorse.
  • Outline segment sweeps — spec §5.6's SEM_OUTLINE_SEGMENTS: roofline submodels (Roof_Left, Peak, Garage_Line) in left-to-right order so an intro can "draw" the house section by section (sequencing guide: "Outline chases that 'draw' the house are a great intro move").

The honest conditionality. docs/craft-roadmap.md line 11, verified against the user's file: "this user's rgbeffects.xml has 0 SubModels → submodel work is DEFERRED", and item 9 (line 74) marks the change deferred for exactly that reason. Nothing here benefits the current layout. F-F activates only when F-E onboards a layout that has submodels — vendor HD props ship with them, so any shared-tool user is likely to bring some — or when this user adds them (the layering guide's line 272 explicitly contemplates defining a scheme). This document exists so that design intent is captured now and the F-E classifier leaves the right seams; it deliberately proposes a minimal-first plan where everything buildable without submodel hardware is cheap and everything else is parked behind an explicit verification gate.

2. Current state

There is no submodel handling anywhere in the packages — and one guard that actively blocks it:

  • The spec preserves but doesn't use them. xlights-layout-semantics-spec.md §1 lists "<subModel> children (named pixel subsets, preserve these)" among the inputs, and §5.6 sketches the one concrete deliverable:
    "Do not flatten submodels. If OUTLINE models have submodels (Roof_Left, Peak, Garage_Line), additionally create SEM_OUTLINE_SEGMENTS containing the submodels in left-to-right order to enable section-by-section builds." (line 173)
    The 2026-06-10 add-layout-semantics change explicitly listed "submodel SEM_OUTLINE_SEGMENTS" as a non-goal.
  • layout_semantics.py has no submodel concept. The Prop dataclass (lines 23–40) carries no submodel field; build_sem_groups() (line 50) plans groups of model names only. (The F-E design adds Prop.submodels as a forward seam.)
  • The preview parser skips them. preview/layout.py::parse_models() (line 113) iterates models_el.findall("model") and never reads <subModel> children — the offline renderer can light a submodel's pixels (they're just parent-model channels) but has no mapping from submodel name to channel subset.
  • The write path rejects submodel targets today. xlights_core/editing.py::place_preset() (lines 83–84): 
    if target not in await client.get_models():
    raise ValueError(f"target {target!r} not in layout")
    client.get_models() (client.py:147) returns model and group names only, so even if xLights' addEffect accepted a submodel element, this client-side guard raises first. Every effect the pipeline places flows through place_preset (effect_emitter.py::apply_instructions, line 62), so this is a single, well-defined choke point.
  • The targetability probe covers groups only. pipeline/groups.py::targetable_groups() (line 35) probes get_group_names() — but that is exactly the right tool to extend: it exists because "targetability genuinely isn't derivable from the XML, so probing beats parsing" (roadmap assessment), which applies with even more force to submodels.
  • A partial workaround exists and is documented: the layering guide's Sub-Buffer row (line 98) — "splitting one prop into zones without making submodels" by redefining the effect's buffer region. It's per-effect, unnamed, and unordered; it can fake a "top half of the tree" but not an ordered ring walk or a named segment vocabulary. Noted as the alternative considered (§3.6).

3. Proposed design

3.1 Discovery during F-E classification (verifiable today)

Submodel discovery is an extension of the F-E classifier's parse_props(), not a new pass. Each <model>'s <subModel> children are parsed into records attached to the parent Prop:

# knowledge/layout_classify.py  (proposed extension; new dataclass in layout_semantics.py)
@dataclass
class SubModel:
    name: str                 # e.g. "Ring 1", "Roof_Left"
    parent: str               # owning model name
    node_ranges: str          # the raw range spec, e.g. "1-50,101-150"
    nodes: int                # count expanded from the ranges
    kind: str = ""            # derived segment semantics: RING | ZONE | HALF |
                              # SEGMENT | ARM | TOPPER | "" (unknown)
    order_hint: int | None = None   # trailing integer in the name ("Ring 3" → 3)

Two facts and one hedge about the XML shape: submodels are child elements of <model> carrying a name and a node-subset definition (typically a type="ranges" spec whose line0..lineN attributes hold comma-separated node ranges referencing the parent's node indices, or a sub-buffer rectangle). Since this repo contains zero examples (0 SubModels in the only real layout), the exact attribute grammar is pinned the same way F-E pins group layout modes: define one of each kind in the target xLights build, save, and commit the diff as a fixture before the parser merges. The parser must round-trip-preserve anything it doesn't understand — spec §1 says preserve, and the F-E writer never touches non-SEM_ elements anyway.

Segment semantics follow the F-E pattern — a name-heuristic table, inherited context from the parent's role:

SUBMODEL_KIND = [
    (("ring",),                       "RING"),     # tree rings (order = vertical)
    (("zone", "band", "tier"),        "ZONE"),
    (("half", "left", "right"),       "HALF"),
    (("roof", "peak", "garage", "gutter", "eave", "ridge"), "SEGMENT"),
    (("arm", "blade", "spoke"),       "ARM"),      # spinner arms
    (("top", "topper", "star"),       "TOPPER"),
]

Roles are inherited: a submodel of an OUTLINE model is an outline segment; a RING of a MEGA_TREE is a tree zone. Submodels never get independent roles in the §2 taxonomy — they refine their parent. Spatial derivation gives segments a left-to-right order (for SEGMENT/HALF kinds, by the mean x of their node subset once the parent's pixel geometry is known via model_world_pixels()) or a bottom-to-top order (RING/ZONE, by mean y or by order_hint). The manifest's PropRecord.submodels (already stubbed in the F-E schema as submodels: list[str] = []) upgrades to a list of records: {"name", "kind", "nodes", "order"} — a few hundred bytes for a typical layout, still within the §6 10 KB budget.

3.2 Targeting mechanics — the central investigation

Two candidate routes for getting an effect onto a submodel:

Route A — direct element targetRoute B — groups with submodel members (recommended)
Mechanismclient.add_effect(target="Tree/Ring 1", ...) — xLights' addEffect resolves target to a sequence element; whether it resolves submodel child elements (and under what name syntax) is unverified.xLights model groups may contain submodels as members (models="Tree/Ring 1,Tree/Ring 2" in the modelGroup element — the group dialog's "show submodels" affordance). The group is then an ordinary top-level element.
Client changesplace_preset's layout guard (editing.py:83) must learn submodel names — a new tolerant check (manifest lookup or a widened name set), plus whatever name syntax the API wants.None. The group's name appears in get_group_names(), flows through targetable_groups(), place_preset, the emitter, QA, and render-order untouched.
VerificationNew live probe logic; unknown failure modes (XLightsTargetMissing? silent worked=false?).The existing empirical probe (pipeline/groups.py::_probe, line 75) answers it per layout, cached by fingerprint — the same "probing beats parsing" doctrine the roadmap endorses.
ExpressivenessPer-submodel simultaneous distinct effects, arbitrary layering.Equivalent, via singleton groups (one submodel per group) where independence is needed; ordered multi-member groups where traversal is needed.

Decision: Route B first. It reuses every proven mechanism (the probe, the write lock, the emitter's layer allocation, canonical_order), needs zero client changes, and its one unknown — "does xLights accept addEffect on a group whose members are submodels?" — is answered by the probe in one disposable-sequence pass on real hardware. Route A stays a recorded investigation (one live experiment: try addEffect with Model/Submodel and with the bare submodel name; log status codes) because if it works it saves group proliferation for one-off accents; it is not on the critical path.

3.3 SEM_ group extensions (planning verifiable today)

build_sem_groups() grows a companion that emits submodel-membered groups, kept separate so layouts without submodels are byte-identical to today:

# knowledge/layout_semantics.py  (proposed)
def build_submodel_groups(props: list[Prop]) -> dict[str, list[str]]:
    """SEM_ groups whose members are 'Parent/SubModel' references.
    Emitted only for kinds with a choreography contract:

    SEM_OUTLINE_SEGMENTS      OUTLINE submodels of kind SEGMENT, left-to-right
                              (spec §5.6, ordered, layout mode = ordered traversal)
    SEM_TREE_ZONES            MEGA_TREE submodels of kind RING/ZONE, bottom-to-top
    SEM_TREE_ZONE_<i>         singleton per zone (i = 1..N, bottom-up) — the
                              drum-kit targets: distinct simultaneous effects
    SEM_TREE_TOPPER           TOPPER submodels (the cymbal)
    SEM_WINDOW_CELLS          WINDOW submodels — independent rhythm cells
    """

Design points:

  • Ordering is the value. SEM_OUTLINE_SEGMENTS and SEM_TREE_ZONES follow the _LTR doctrine — member order = traversal order, group layout mode = the ordered mode F-E pins in its slice-3.1 spike — so a chase draws the house left-to-right and a riser climbs the tree, with direction reversal via effect parameters (spec §5.2: no _RTL variants).
  • Singletons make the drum kit. Placing kick/snare/cymbal simultaneously on different zones requires three independent targets; per-zone singleton groups deliver that through Route B without any per-submodel element support. Cost: a handful of extra groups (N zones + topper), all probe-verified, all excluded from ensemble beds.
  • Ensemble hygiene: submodel groups never join SEM_ALL or the band/side ensembles — their parents are already there; double membership would double-render. They slot into canonical_order's accent/rhythm tiers (layout_semantics.py:113-124) so zone accents win overlaps against the parent's bed, the same occlusion contract the rest of the vocabulary relies on.
  • Manifest: each group lands in groups with ordered/layout_mode, exactly the §6 GroupRecord shape from F-E, so downstream consumers need no new schema.

3.4 Choreography hooks (design today, verify on hardware)

  • Drum-kit mapping (pipeline/beats.py): extend RhythmRoles (line 252) with optional zones: list[str] (bottom-up singleton zone groups) and topper: str | None. In place_beat_accents() (line 296), when zones exist and the section's follow_stem is drums: downbeats → bottom zone, backbeat positions (_backbeat_positions, line 287) → mid zone, and the sparkle layer's strongest-hit selection retargets from ACCENT_GROUPS to the topper. First cut needs no new audio analysis — it maps beat-grid structure, not per-drum separation. A true kick/snare split (spectral band classification of drum-stem onsets) is a separate, optional analyzer follow-up; kept out of v1 deliberately.
  • Vertical build runs (pipeline/weave.py): the weave's directional sweeps (line 356–363 forces sweep cells onto the group buffer so motion traverses the group) work unchanged on SEM_TREE_ZONES — a chase-family cell with an "up" direction on an ordered bottom-to-top group is the riser. The only new logic: the direction vocabulary for zone groups maps up/down instead of left/right.
  • Outline draw: an intro-scene recipe (scene cookbook candidate) placing a single slow chase on SEM_OUTLINE_SEGMENTS — pure data once the group exists.
  • Director awareness: the F-E layout block gains one line per submodel-bearing prop ("MegaTree: 1600 nodes, 4 zones bottom-to-top + topper — zone-mapped drum accents available"), and pulse_groups can name zone groups like any other group. No schema change.

3.5 Offline rendering & QA (verifiable today)

Because a submodel is just a channel subset of its parent, the offline PreviewRenderer can validate all of this without hardware: parse the node ranges → parent node indices → absolute channels (start_channel + 3*node, as render.py:43 computes) → synthesize frames lighting one zone/segment at a time → run the F-E sweep-centroid check vertically for SEM_TREE_ZONES (centroid y strictly increasing) and horizontally for SEM_OUTLINE_SEGMENTS. The F-E validator generalizes with one parameter (which axis). Coverage QA needs nothing: submodel-group effects light parent channels, which make_lit_sampler (pipeline/visual.py:105) already counts.

3.6 Alternative considered: Sub-Buffer zones

The layering guide (line 98) documents Sub-Buffer as zone-splitting "without making submodels". Rejected as the primary mechanism: it's a per-effect buffer rectangle (no named, reusable, ordered vocabulary; no group-canvas traversal across zones; invisible to the probe and the manifest). It remains useful as a per-placement escape hatch (e.g. "top half of the matrix"), possibly via extra_settings, but that's an independent feature.

5. Testing strategy

  • Hermetic (repo convention, no xLights/LLM):
    • Parser: fixture round-trip — names, ranges ("1-50,101-150" → 100 nodes), kinds, order hints; unknown submodel types preserved untouched; a model with zero submodels yields [].
    • Ordering: ring order from order_hint vs from mean-y geometry agree on the fixture; segment LTR order matches world-x; inverted-x flips it (shares F-E's invert_x tests).
    • Group plan: exact expected group set on the submodel fixture; empty delta on the real 0-submodel layout (the conditionality guard, asserted forever); singletons exclude ensemble membership; canonical_order tier placement.
    • Manifest: records serialize/load; size budget holds.
    • Choreography: select_rhythm_groups with zones present maps downbeat/backbeat/sparkle to bottom/mid/topper; without zones, byte-identical output to today (golden pipeline snapshot unchanged).
    • Offline validation: synthesized zone frames pass the vertical centroid check; a deliberately misordered group fails it.
  • Live (opt-in -m live, hardware-blocked): the phase-3 gate as a test module — group load, probe targetability, Route-A experiment (xfail-tolerated, result logged), one placed sweep per new group kind. Perceptual confirmation (does the riser read as climbing) stays a human watch, per the repo's live-verify convention.

7. Acceptance criteria

Verifiable today (phases 0–2):

  • ☐ The classifier parses <subModel> children from the authored fixture into SubModel records (names, expanded node counts, kinds, order) and preserves unknown submodel types untouched.
  • ☐ build_submodel_groups() emits SEM_OUTLINE_SEGMENTS (LTR), SEM_TREE_ZONES (bottom-up), per-zone singletons, and SEM_TREE_TOPPER on the fixture — and emits nothing on the current real layout (0 submodels → zero behavior change, asserted by test).
  • ☐ The manifest carries submodel records and submodel-group records within the 10 KB budget.
  • ☐ Offline validation lights each zone/segment via its channel subset and the axis-appropriate centroid check passes (and fails on a shuffled order).
  • ☐ xlo init-layout --submodels --dry-run shows the planned groups in the diff.
  • ☐ Golden pipeline snapshot unchanged; select_rhythm_groups without zones is byte-identical to today.

Hardware-blocked (phase 3+ — required before default-on):

  • ☐ Submodel-membered SEM_ groups load in xLights after restart and pass the targetable_groups() probe.
  • ☐ The Route-A direct-target experiment is run once and its result recorded in the change docs.
  • ☐ A chase on SEM_OUTLINE_SEGMENTS traverses left-to-right and a riser on SEM_TREE_ZONES climbs, confirmed by a real render + human watch.
  • ☐ Drum-kit mapping places zone-differentiated accents in a drums-prominent section on a real layout.

8. Open questions

  1. Member name syntax — the exact models= string for a submodel member (Parent/Sub is the expected convention; pinned by the phase-0 fixture, like F-E's layout-mode strings).
  2. Does getModels ever list submodels? If a future xLights exposes them, place_preset's guard passes for free and Route A becomes testable without client changes — worth re-checking per version bump.
  3. Zone granularity policy — merge many rings into 3–4 zones (readable at distance, fewer groups) vs faithful per-ring groups? Leaning merge, with the ring→zone map recorded in the manifest.
  4. Kick/snare separation — is beat-position mapping (downbeat/backbeat) good enough for the drum-kit illusion, or does the spectral split of drum-stem onsets earn its analyzer complexity? Decide after watching phase-4 output on hardware.
  5. Vendor submodel import — adopting vendor-compatible naming (sequencing guide lines 223, 263) so purchased HD props classify cleanly: a heuristic-table growth path or an import-mapping file?
  6. Should this user add submodels? The layering guide (line 272) argues yes for the mega tree/spinners; if so, F-F's phase 3 unblocks on the current display and the "conditional" label falls away — a cheap way to de-risk the whole item.
