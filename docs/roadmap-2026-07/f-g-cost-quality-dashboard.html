1. Why

The refine loop already writes a flight recorder — revision_log.jsonl, one record per iteration plus a finalize record — and that data has already paid for itself twice by hand: the REFINE_SKIP_OBJECTIVE = 88 threshold in pipeline/run.py:62 is annotated "Revision-log analysis (42 runs): drafts whose first-pass objective is ≥ this gained ≈0 over the whole loop", and the add-refine-skip-high-objective OpenSpec proposal quotes the same 42-run / 122-iteration analysis (runs starting ≥ 88 gained +0.6; runs below gained +19.2). That analysis was a one-off ad-hoc script; the numbers rot the moment the next tuning question arrives.

Meanwhile the project runs an active cost-reduction campaign (tiered model routing, guide-extract trimming, the skip gate, I8's tiered visual critique) justified entirely by estimated cost, because — per roadmap I1 — there is no token accounting anywhere. Once I1 threads per-role token counts and a cost summary into the revision log, every run becomes evaluation data. F-G is the harvest: a small, deterministic analyzer that answers, without any LLM or xLights, the questions the team keeps asking manually:

  • What did this show cost, and what did each objective point of refinement cost?
  • Does the loop actually improve scores, per iteration, per song, per model routing?
  • Which sections churn (get regenerated repeatedly), and how often do revisions regress and get reverted?
  • How often does the skip gate fire — is 88 still the right threshold?
  • Which model tier (planner vs worker) dominates spend? (This is F-H's comparison axis too.)

2. Current state

2.1 What one iteration record contains

revision_log.py:33-52 defines the canonical JSONL schema (verbatim):

class RevisionLogRecord(BaseModel):
    run_id: str
    iteration: int
    song_key: str
    ts: str
    kind: Literal["iteration", "finalize"] = "iteration"
    objective_score: int
    advisory_score: int
    findings: list[LogFinding] = []
    judge: dict | None = None                 # {"score": int, "verdict": str}
    revisions: list[LogRevision] = []
    regenerated_sections: list[int] = []
    obj_before: int | None = None
    obj_after: int | None = None
    obj_delta: int | None = None
    reverted: bool = False
    human_decision: str | None = None
    models: dict[str, str] = {}
    review_bundle: str | None = None

Every field the quality half of the dashboard needs already exists: score trajectory (obj_before/obj_after/obj_delta), churn (regenerated_sections, revisions with origin: "judge" | "backstop"), reverts (reverted), the Judge's view (judge), and a per-role model snapshot (models, filled from model_snapshot() in models/registry.py:38-45, e.g. {"director": "anthropic:claude-opus-4-8", ...}). The module docstring (revision_log.py:1-6) fixes the contract the analyzer relies on: JSONL is canonical/programmatic, Markdown is the human view, and logging is pure observability.

2.2 What the loop records, and the two gates the dashboard must surface

Records are written by the _record closure in _refine_loop (pipeline/run.py:201-209). The skip-high-objective gate (run.py:211-216) short-circuits the whole loop and leaves a distinctive fingerprint — a finalize record at iteration 0 with human_decision="skip-high-objective":

    if skip_objective is not None and best_obj >= skip_objective:
        # already good → accept the draft without spending judge/critic/regen iterations
        log.info("refine skipped: first-pass objective %d ≥ %d (already good)", best_obj, skip_objective)
        _record(0, await _report(st.applied), None, kind="finalize",
                obj_after=best_obj, human_decision="skip-high-objective")
        return

The objective-regression revert (run.py:275-284) marks an iteration whose regeneration made things worse; REGRESS_MARGIN = 1 (run.py:58) and STALL_LIMIT = 2 (run.py:59) govern it:

        reverted = obj < best_obj - REGRESS_MARGIN
        if reverted:                                      # objective REGRESSION → revert it
            st.instructions, st.applied = list(best), best_applied

The iteration record then carries obj_before/obj_after/obj_delta and reverted=True (run.py:290-296). Other terminal states also leave markers: an accept/stop decision logs human_decision="accept"|"stop" with obj_delta=0 (run.py:233-234); the plateau detector logs human_decision="plateau" (run.py:240-241); every run that enters the loop ends with a kind="finalize" record carrying the best objective (run.py:304-305). Two stop causes are not distinguishable today: the stall stop (run.py:297-298) and the max_iterations cap both just fall through to finalize — see §3.6.

2.3 Where the log lives — the cache layout

pipeline/cache.py (entire file is 24 lines) defines the layout:

def cache_root() -> Path:
    return Path(os.environ.get("XLO_CACHE_DIR", "data")) / "orchestrator"

def song_key(path: str) -> str:
    h = hashlib.sha1()
    with open(path, "rb") as f:
        for chunk in iter(lambda: f.read(1 << 20), b""):
            h.update(chunk)
    return h.hexdigest()[:16]

and run_pipeline wires the log per song (run.py:455-459):

        revlog, run_id = NullRevisionLog(), "run"
        if log_revisions:
            run_id = datetime.now(timezone.utc).strftime("%Y%m%dT%H%M%SZ")
            base = _cache_root() / key
            revlog = RevisionLog(base / "revision_log.jsonl", base / "revision_log.md")

So the on-disk tree the analyzer walks is:

data/orchestrator/                        # or $XLO_CACHE_DIR/orchestrator
├── targetable_groups_<fp>.json          # layout-fingerprint caches (pipeline/groups.py:53) — NOT songs
└── <song_key>/                          # 16-hex sha1 prefix of the MP3 bytes
    ├── song_analysis.json / song_description.json / description.md
    ├── creative_brief.json / creative_brief.md / instructions.json
    ├── revision_log.jsonl                # ← canonical input; APPEND-mode across runs
    ├── revision_log.md
    └── visual_review/iter<N>/            # frame bundles referenced by review_bundle

Two consequences matter. First, RevisionLog.write opens with "a" (revision_log.py:108), so one revision_log.jsonl accumulates many runs — grouping key is (song_key, run_id), never the file. Second, the song's human name appears nowhere in the key; the analyzer can recover a display name from the sibling description.md / creative_brief.md or just show the key.

2.4 What the scores mean

qa/__init__.py:16-35 composes them: objective_score is the rounded mean of sync, placement, rules (plus coverage when a rendered-frame sampler exists — lines 24-29), and it alone drives the revert/stall gate; advisory_score is the variety subscore and only informs the Judge/human. The Judge's own JudgeVerdict (refine.py:36-39: score, verdict: "accept"|"iterate"|"stop", revisions) is logged as the judge dict. The dashboard must keep the objective/advisory distinction visible — conflating them would misreport the loop's actual gate.

2.5 The CLI and the missing pieces

cli.py uses argparse (not click): main() at cli.py:67-116 builds subparsers edit-brief, run, regen and dispatches on args.cmd. There is no analysis/reporting code anywhere in the package, no token/usage capture (a repo-wide grep for usage in src/ returns nothing), and models/config.yaml has role→model routing (lines 5-23) but no pricing table. Existing test coverage for the log is tests/test_revision_log.py: pure-unit record/rendering tests plus hermetic _refine_loop integration via an in-memory _MemLog and injected fakes — the exact conventions F-G's tests should follow.

3. Proposed design

3.1 Shape: a reporting.py module behind an xlo report subcommand

Module vs standalone script. A standalone script under scripts/ would have to re-declare the record schema or import the package anyway; it would drift the way the original 42-run analysis script did (it no longer exists in the repo). A package module gets the real RevisionLogRecord for parsing, the real cache_root() for discovery, and pytest coverage for free. Recommendation: module. Proposed: packages/xlights-orchestrator/src/xlights_orchestrator/reporting.py — a sibling of revision_log.py, deliberately not under pipeline/ (it never touches a live run; it reads what the pipeline wrote). If it outgrows one file (~250 lines expected), split into reporting/{load,metrics,render_text,render_html}.py later.

CLI subcommand vs script entry point. An xlo report subparser costs ~10 lines in the existing argparse structure, inherits load_env(), and — unlike run/regen — must not gate on has_llm_key() (cli.py:99-101): reporting is deterministic and works offline. Recommendation: xlo report subcommand, consistent with the roadmap's other operability surfaces (xlo regen, the planned xlo init-layout).

Proposed public API (all signatures are proposed, none exist yet):

3.2 The I1 contract this design consumes

I1 (token & cost telemetry) is described in the roadmap as: capture result.usage() from PydanticAI agent runs, thread it into revision_log.py per-role/per-iteration, and add a per-run cost summary from tokens × published rates in models/config.yaml. F-G designs against these concrete field names (to be finalized in I1's own doc; proposed here, additive and default-valued so every pre-I1 record still validates):

# Proposed I1 extension — revision_log.py
class RoleUsage(BaseModel):
    input_tokens: int = 0
    output_tokens: int = 0
    requests: int = 0                # >1 when a role runs multiple times in one iteration

class RevisionLogRecord(BaseModel):
    ...                              # existing fields unchanged
    usage: dict[str, RoleUsage] = {}     # role -> tokens spent in THIS iteration
    cost_usd: float | None = None        # this iteration, from usage × pricing
    # finalize records additionally aggregate:
    run_usage: dict[str, RoleUsage] = {} # whole-run per-role totals
    run_cost_usd: float | None = None

and a pricing table in models/config.yaml, keyed by the bare model IDs already used in roles: (rates in USD per million tokens; values below are placeholders for I1 to fill from published price lists):

# Proposed I1 extension — models/config.yaml
pricing:                       # USD per 1M tokens
  claude-opus-4-8:        {input: 0.00, output: 0.00}
  claude-sonnet-4-6:      {input: 0.00, output: 0.00}
  gemini-3.1-pro-preview: {input: 0.00, output: 0.00}
  gemini-3.1-flash-lite:  {input: 0.00, output: 0.00}
  gemini-3.5-flash:       {input: 0.00, output: 0.00}

The analyzer resolves a record's role to a model via the record's own models snapshot (already logged per iteration — correct even across provider switches mid-history), strips the provider: prefix, and looks up pricing. It prefers the record's stored cost_usd when present (rates as-of-run-time beat rates as-of-report-time) and only recomputes from tokens when asked (--reprice) or when cost_usd is absent but usage is present.

Graceful degradation on pre-I1 logs is a hard requirement — the existing corpus (the 42-run history) predates I1. Because all new fields default to empty, old lines parse cleanly. Per run, the analyzer sets has_cost: bool; when false, every cost cell renders —, cost-derived metrics (cost/show, cost/point, tier attribution) are omitted from that run's row, aggregate cost totals count only costed runs (with an explicit "12 of 42 runs have cost data" caveat line), and all quality metrics (trajectory, churn, reverts, skip rate) work at full fidelity. A proxy-cost mode is deliberately rejected: estimating tokens for old runs would fabricate exactly the numbers this tool exists to make real.

3.4 Output formats: compute once, render twice

Both formats are worth shipping, and the marginal cost of the second is one pure function over the same Report model. Terminal is the daily driver — it composes with the CLI workflow (xlo run --refine then xlo report --song …) and works over SSH; it uses plain str.format column alignment (the package depends on pydantic-ai only — no rich/tabulate, and this feature does not justify adding one). Static HTML is where trajectory and churn become legible — per-run score sparklines and per-section churn bars as inline SVG, no JavaScript, no external resources, using this design doc's own CSS variables so it inherits light/dark. It is also the natural artifact to attach to an OpenSpec change or an F-H A/B comparison. Recommendation: ship both — render_text prints by default; --html [PATH] writes report.html (default: cache_root()/report.html). Anything divergent between the two renderers is a bug in the split, not a feature: all arithmetic lives in summarize_run/build_report, renderers only format.

3.5 CLI wiring

# Proposed — cli.py additions inside main()
p = sub.add_parser("report", help="cost & quality dashboard from the revision logs")
p.add_argument("--song", default=None, help="one song (path; keyed via song_key) — default: all cached songs")
p.add_argument("--cache-dir", default=None, help="override the cache root (default: $XLO_CACHE_DIR/data)")
p.add_argument("--html", nargs="?", const="", default=None, help="also write a static HTML report [PATH]")

Dispatch is synchronous (no asyncio.run, no XLightsClient, no has_llm_key() gate). --song reuses _song_key from pipeline/cache.py exactly the way _edit_brief already does (cli.py:52-64). Discovery via root.glob("*/revision_log.jsonl") naturally ignores the top-level targetable_groups_*.json layout caches (pipeline/groups.py:53) because those are files, not song directories.

3.6 One tiny logger assist (optional but cheap): stop_reason

The finalize record at run.py:304-305 — _record(iters, final, None, kind="finalize", obj_after=best_obj) — cannot distinguish a stall stop from the max_iterations cap (both fall through the loop), and design escalations (run.py:252-265) are visible only in log.info, not the record. Proposed, additive, default-valued (so it rides along with I1's schema touch or I3's loop decomposition):

# Proposed — revision_log.py
    stop_reason: str | None = None   # skip-gate | accept | stop | plateau | stall | cap
    redesigned_sections: list[int] = []   # design escalations this iteration

The dashboard treats these as best-effort: absent → the stop-reason mix reports stall-or-cap as one bucket for old runs. This is the only change F-G wants inside the pipeline; everything else is read-only.

4. Implementation plan

  1. Land after I1's schema. Confirm final field names (usage, cost_usd, run_usage, run_cost_usd, pricing) against I1's implementation; adjust §3.2 names if I1 chose differently. F-G's loader must match I1's writer exactly.
  2. reporting.py — load layer. discover_logs, load_records (line-tolerant: RevisionLogRecord.model_validate_json per line, count-and-skip on ValidationError/JSONDecodeError), group_runs (ordered dict keyed by run_id, records in file order).
  3. reporting.py — metrics layer. RunSummary (run_id, song_key, iterations, first/final objective, advisory, trajectory list, churn Counter, revert count, stop_reason, has_cost, per-role usage/cost, cost_per_show, cost_per_point) and summarize_run; Report + build_report aggregating per-song and fleet-wide (skip-rate, stop-reason mix, tier attribution, finding mix). Pricing loaded via a small pricing() accessor next to _cfg() in models/registry.py (added by I1; F-G only reads it).
  4. Renderers. render_text(report) — fixed-width tables, the costed-runs caveat line, — for missing cost. render_html(report) — one self-contained page, this doc's CSS block, tables + inline-SVG sparklines (score trajectory per run) and horizontal churn bars; escape all detail strings.
  5. CLI. The report subparser and dispatch branch per §3.5; --json prints report.model_dump_json(indent=1).
  6. Logger assist (optional). stop_reason + redesigned_sections per §3.6 — coordinate with I3, which is rewriting _refine_loop's seams anyway.
  7. Docs. One paragraph in README.md's CLI section; note in the roadmap scorecard that F6 is closed by F-G.

5. Testing strategy

Hermetic-first, matching tests/test_revision_log.py: no LLM, no xLights, no network — the analyzer is pure functions over files, so this is easy. New file tests/test_reporting.py, fixture JSONL under tests/fixtures/revision_logs/.

  • Fixture construction two ways. (a) Programmatic: build RevisionLogRecord objects and write them through the real RevisionLog writer into tmp_path — guarantees writer/reader agreement forever. (b) Checked-in raw JSONL fixtures: one pre-I1 file (fields exactly as today's schema — copy real lines, keys anonymized) and one post-I1 file with usage/cost_usd — these freeze backward compatibility: if a future schema change breaks parsing of historical logs, a test fails, not a report.
  • Load layer: malformed line is skipped and counted; multiple run_ids in one file group correctly and preserve order; discover_logs on a synthetic tmp_path tree (two <song_key>/ dirs + a decoy targetable_groups_x.json) finds exactly two logs — with XLO_CACHE_DIR monkeypatched, per the cache_root() env contract.
  • Metrics: golden-value tests per metric on a hand-built run: trajectory 71→78→76(revert)→90 yields revert count 1, gain 19, cost/point = Σcost/19; skip-gate fixture (single finalize, human_decision="skip-high-objective", iteration 0) counts as a skip and contributes no cost/point; zero-gain run renders ∞ (no gain); mixed pre/post-I1 corpus reports has_cost correctly and totals only costed runs.
  • Renderers: substring assertions on render_text (the caveat line, the — cells) mirroring test_render_md_shows_origin_and_scores; render_html output contains no http:///https:///<script (self-containment guard) and escapes a finding detail containing <b>&.
  • CLI: main(["report", "--cache-dir", str(tmp)]) end-to-end via capsys, and --json round-trips through Report.model_validate_json — this doubles as a down payment on roadmap I6's "cli.py argument wiring" gap.
  • Loop integration (only if §3.6 lands): extend the existing _MemLog-style _refine_loop tests to assert stop_reason on stall vs cap.

7. Acceptance criteria

  • [ ] xlo report exists, runs with no LLM key, no xLights, no network, and exits 0 on an empty cache with a clear "no revision logs found under <root>" message.
  • [ ] Against a cache with post-I1 logs it prints, per run: iterations, first→final objective, advisory, cost per show, cost per objective point, revert count, stop reason.
  • [ ] Fleet aggregates include skip-gate hit rate, stop-reason mix, per-section churn, finding mix by source, and per-role/per-tier cost attribution.
  • [ ] Pre-I1 logs (fixture-pinned) parse fully; cost cells show —; quality metrics are complete; the costed-runs caveat line appears.
  • [ ] --html writes one self-contained file (no external URLs, no scripts) that renders correctly in light and dark; --json round-trips through the Report model.
  • [ ] Multiple runs appended to one revision_log.jsonl are reported as separate runs keyed by run_id; a malformed line never aborts the report.
  • [ ] All of the above covered by hermetic tests in tests/test_reporting.py with fixtures under tests/fixtures/revision_logs/; the full suite stays green.
  • [ ] No behavioral change to the pipeline itself beyond (optionally) the additive stop_reason/redesigned_sections fields, which default such that every existing test and log line remains valid.

8. Open questions

  • Where does I1 put non-refine spend? Analyst panel, synthesizer, and Director first-pass calls happen before _refine_loop — today the revision log only exists during refine (run.py:449-459). If I1 logs pre-loop usage as an iteration 0-style record (or a new kind="generate"), cost-per-show is truly whole-show; if not, F-G must label its number "refine cost" honestly. To be settled in I1's design.
  • Cache-hit accounting: a run resumed from cached brief/instructions spends nothing on those stages — should RunSummary mark which stages were cache hits (I1 could log a zero-usage marker) so cost-per-show comparisons across runs are apples-to-apples?
  • Song display names: derive from description.md's first heading, or add a tiny meta.json (song filename) to the cache dir at run time? The latter touches the pipeline; the former is heuristic but read-only.
  • Should stop_reason ride with I1 or I3? I3 rewrites _refine_loop into named guard functions — the natural moment to make each guard record its own name.
