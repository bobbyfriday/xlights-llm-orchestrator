## ADDED Requirements

### Requirement: Capture per-role LLM token usage per run
The system SHALL capture the token usage reported by each LLM agent run into a run-scoped accumulator, attributed to the calling role, for every agent invocation in the pipeline (interpret, design, generation, redesign, judging, and visual critique).

Capture SHALL cover the shared generation call site so that first-pass generation, refine-loop regeneration, and manual regeneration are all measured, and the accumulator SHALL reach every invocation — including the concurrently-run analysts — without changing pipeline function signatures.

#### Scenario: Every LLM call site is measured
- **WHEN** the pipeline runs and each role's agent produces a result carrying usage
- **THEN** the run-scoped accumulator holds per-role input/output/cache token counts and request counts for every role that ran

#### Scenario: Generation is measured on every path
- **WHEN** a section is generated on the first pass, regenerated by the refine loop, or regenerated manually
- **THEN** the tokens for that generation are attributed to the generator role in the accumulator

#### Scenario: Concurrent analysts each record
- **WHEN** the analyst panel runs its analysts concurrently
- **THEN** each analyst's usage is captured into the same run's accumulator

### Requirement: Telemetry capture is best-effort and never breaks a run
Usage capture SHALL be best-effort: a missing accumulator, an agent result without usage data, or any error during capture SHALL be a defined no-op that does not raise, so a run produces the same result whether or not telemetry succeeds.

#### Scenario: Result without usage is a no-op
- **WHEN** capture is attempted on an agent result that exposes no usage (e.g. a test fake)
- **THEN** nothing is recorded and no error is raised

#### Scenario: No active accumulator is a no-op
- **WHEN** capture is attempted with no run-scoped accumulator installed
- **THEN** it silently does nothing and the run proceeds unchanged

### Requirement: Prices are configuration data and cost is derived
The system SHALL read per-model token prices from a configuration price table keyed by bare model id, and SHALL derive run cost purely from captured token counts times those prices, so re-pricing requires only a data change and never a code change.

A model absent from the price table SHALL make the derived cost unknown (not zero), and cache-read and cache-write tokens SHALL be priced at their own rates distinct from uncached input.

#### Scenario: Cost derived from tokens and prices
- **WHEN** every role with usage runs on a priced model
- **THEN** the estimated cost equals the sum over roles of token counts times that role's model rates, with cache tokens priced at the cache rates

#### Scenario: Unpriced model makes cost unknown
- **WHEN** any role with nonzero usage runs on a model with no price-table entry
- **THEN** the derived cost is unknown rather than zero
