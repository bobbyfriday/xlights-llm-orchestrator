1. Why

Both remaining craft-roadmap.md items — Matrix narrative Text (item 8) and lyric-driven Faces (item 7) — have been blocked since June on the same wall: every effect the pipeline can place must come out of the mined preset catalog, and Text/Faces (and Pictures/Video/Shader) are not in it and structurally cannot be. The June roadmap scorecard row is blunt: "F1 Matrix Text / F2 Faces — ❌ Open — still blocked on an asset-bound placement path."

The block is not an oversight; it is a consequence of a correct design decision. The catalog is mined from community .xsq files precisely so that every settings string we emit is valid by construction — observed working in a real show. That guarantee cannot extend to effects whose settings reference things outside the settings string: image/video file paths on someone else's disk, face definitions in someone else's rgbeffects.xml, timing-track names in someone else's sequence. A mined Text look whose value curve reads "Timing Track: Verse 1" is worse than useless in our sequence — it silently renders nothing.

So the fix is not "stop excluding them from mining." It is a second placement route: for the narrow set of asset-bound effects we actually want (Text now, Faces next), construct the settings string from scratch in code, bind its external references to resources we ourselves create (our own timing tracks, the layout's own face definitions), validate the string with the settings parser we already have, and carry it through the existing emitter seam. This document designs that route.

2. Current state

2.1 The exclusion, in three layers

Layer 1 — mining. knowledge/constants.py:9:

# Effect types whose settings reference external resources not guaranteed to exist
# in a target sequence. Excluded from mining.
ASSET_BOUND_TYPES = frozenset({"Faces", "Pictures", "Video", "Shader", "DMX"})

knowledge/xsq_extractor.py::extract_file (lines 73–89) skips them at the source, plus a second, subtler filter: any effect whose settings carry an active timing-track value curve is skipped too (_has_active_timing_track_curve, lines 51–56, built on settings.classify_value_curve / constants.is_timing_track_curve_type):

if name in ASSET_BOUND_TYPES:
    skipped["asset_type"] += 1
    continue
...
if _has_active_timing_track_curve(parse_settings(s)):
    skipped["timing_track_vc"] += 1
    continue

The committed catalog's own metadata records the cost: presets/looks.json meta shows "skipped": {"asset_type": 453, "timing_track_vc": 197} across the 17-file corpus. Note a nuance the roadmap wording papers over: Text is not in ASSET_BOUND_TYPES — it simply never survived mining (community Text is typically lyric text bound to timing tracks, caught by the second filter, or absent from this corpus's top-level files). The committed catalog's 37 looks_by_type keys include neither Text nor Faces. Either way the practical effect is identical: no Look, no placement.

Layer 2 — agent vocabulary. agents/catalog.py::placeable_effect_types() (line 15) is get_library().list_effect_types() minus KNOWN_REJECTED_TYPES — so the Director and Generator are never even offered Text/Faces, QA's ENERGY_BAND has no entries for them, and weave.canon_effect_type / beats._accent_look (which guard on candidate_look_ids(effect_type) being non-empty) silently drop or fall back if an LLM hallucinates one. This defense-in-depth is good and should stay: F-B does not add these types to the LLM's free-choice menu; the new deterministic passes (F-C/F-D) are the only authors.

Layer 3 — placement. Every placement goes through editing.place_preset (editing.py:35–94), whose first act is a library lookup:

lib = library or get_library()
look = lib.get_look(effect_type, look_id)      # raises KeyError for Text/Faces
...
settings = lib.assemble(look, knob_values)     # validates knobs per constraint

For an unmined type, get_look raises KeyError — and in effect_emitter.apply_instructions, KeyError is in _SKIPPABLE (line 16), so the instruction would be quietly recorded in skipped and the show goes on. There is no path by which an EffectInstruction can deliver a settings string that didn't come from a mined Look.

2.2 The seams we get to reuse

  • place_preset's override mechanics (editing.py:67–78): extra_settings entries override same-keyed pairs in the assembled string (first-occurrence-wins is handled) and append the rest; values may contain | (value curves) but never ,, so comma-splitting is safe. The direct path can reuse this exact merge logic.
  • Palettes are already independent of looks. colors.palette_from_colors builds the C_BUTTON_Palette1..N=,C_CHECKBOX_Palette1..N= string from named/hex colors — nothing preset-bound. Text/Faces colors ride this unchanged.
  • The wire format is trivial. client.add_effect(target, effect, settings, palette, layer, start_ms, end_ms) (client.py:223–244) takes an opaque settings string; xLights doesn't care where it came from.
  • The parser round-trip exists. knowledge/settings.py: parse_settings/serialize_settings (lines 13–30) and constants.classify_kind (key-prefix → slider/choice/checkbox/valuecurve/text) give us syntactic validation of any string we synthesize.
  • The scratch-sequence validator exists. editing.validate_preset (lines 97–177) proves a placement against live xLights on a disposable sequence — the pattern the direct path copies for its live smoke test.
  • Buffer style stays orthogonal. The emitter injects B_CHOICE_BufferStyle via resolve_buffer_style (effect_emitter.py:113–114) regardless of provenance.

3. Proposed design

3.1 Principle: templates in code, references we own

A "direct effect" is an effect whose settings string is built by a code-owned template — a curated, hand-verified analogue of a mined Look — where every external reference points at a resource the pipeline itself creates or verifies:

  • Text: no external file at all when using OS font rendering; the only "asset" is the glyph content, which is a plain settings value. Fully self-contained → the easiest case, and why F-C is the first consumer.
  • Faces: two external references — the target model's face definition (lives in the user's rgbeffects.xml; must be detected, never invented) and a phoneme timing track (which F-D's pipeline itself writes, so the name is ours). Placeable only when both exist → gated, not assumed.
  • Pictures/Video/Shader: explicitly OUT (§3.6).

3.2 New module: xlights_core/knowledge/direct_settings.py (proposed)

All code below is proposed, not existing. The module owns the per-type templates and builders:

"""Code-owned settings templates for asset-bound effect types (Text, Faces).

These types cannot ride the mined preset catalog (ASSET_BOUND_TYPES / no corpus
looks): their settings reference resources outside the string. Each builder
constructs a settings string from scratch, binding references only to resources
the caller proves exist. Round-trip-validated via settings.parse_settings.
"""

DIRECT_TYPES = frozenset({"Text", "Faces"})   # the only types the direct path serves

def build_text_settings(
    text: str,
    *,
    movement: str = "none",        # "none" | "left" | "up"  (validated against _TEXT_DIRS)
    font_size: int = 12,           # px; clamp to the target matrix height upstream
    bold: bool = True,
    speed: int = 10,               # scroll speed when movement != "none"
    center: bool = True,
) -> str: ...

def build_faces_settings(
    *,
    timing_track: str,             # the phoneme track THIS run writes (F-D)
    face_definition: str,          # a definition verified present on the target model
    eyes: str = "Auto",
    outline: bool = True,
) -> str: ...

Text setting keys. The Text effect's panel maps to E_* keys in the standard self-describing scheme (classify_kind already understands all the prefixes). The working set the builder emits:

KeyKindPurpose / values
E_TEXTCTRL_TexttextThe rendered string itself (the one mandatory key)
E_FONTPICKER_Text_FontotherLegacy wx font descriptor (kept at its default)
E_CHOICE_Text_Fontchoice"Use OS Fonts" — avoids xLights' bundled bitmap fonts
E_CHOICE_Text_DirchoiceMovement: "none", "left", "right", "up", "down" (+ vector variants)
E_TEXTCTRL_Text_SpeedtextScroll speed for moving text
E_CHOICE_Text_Effectchoice"normal" (others: vert/rotate variants — out of scope v1)
E_CHOICE_Text_Countchoice"none" (countdown modes — out of scope v1)
E_CHECKBOX_TextToCentercheckboxCenter static text on the buffer
E_SLIDER_Text_XStart / E_SLIDER_Text_YStartsliderStatic offsets (default 0)

Verification protocol for the exact key list (a hard task in the implementation plan, because key names drift across xLights versions and must not be trusted from memory): author one Text effect by hand in the pinned xLights version, save, and read the settings string out of the .xsq's EffectDB — the extractor's own parsing utilities (parse_settings) make this a five-line script. The observed string becomes the frozen template; the builder only varies the table's keys. This is exactly how the mined catalog earns "valid by construction," applied to a corpus of one hand-authored probe.

Faces setting keys (same verification protocol; the F-D design doc covers semantics): the builder binds E_CHOICE_Faces_TimingTrack (the phoneme track name), the face-definition choice key observed in the probe (E_CHOICE_Faces_FaceDefinition in current builds), phoneme source mode, eyes, and outline. Unlike Text, the builder requires its two reference arguments and performs no defaulting — a Faces string with a dangling reference renders a blank prop.

3.3 Carrying it: EffectInstruction.direct_settings (proposed schema addition)

Add one defaulted field to EffectInstruction (show_plan.py:63–80):

class EffectInstruction(BaseModel):
    target: str
    effect_type: str
    look_id: str                      # "" allowed when direct_settings is set
    ...
    direct_settings: str = ""         # PROPOSED: full settings string built by
                                      # knowledge.direct_settings; when non-empty the
                                      # emitter bypasses the preset library entirely

Defaulted → cached instruction lists (instructions stage cache, golden fixture) validate unchanged. The generator LLM is not told about the field (it isn't in any prompt); only deterministic passes populate it. QA treats a direct-settings instruction like any other (it has a real target/effect_type/span); rules.ENERGY_BAND simply has no entry for Text/Faces, which means "unconstrained" by construction (qa/rules.py:15 comment).

Alternative considered and rejected: a magic look_id="direct:…" sentinel carrying the payload — saves a field but hides a load-bearing behavior switch inside a string convention, and look_id is threaded through logging, review bundles, and skip reasons where the full settings string would be noise.

3.4 Placement: editing.place_direct (proposed) + emitter branch

A sibling of place_preset that skips assembly but keeps every guard:

async def place_direct(
    client: XLightsClient,
    target: str,
    effect_type: str,
    settings: str,                      # from knowledge.direct_settings builders
    *,
    palette_colors: list[str] | None = None,
    extra_settings: dict[str, str] | None = None,
    layer: int = 0,
    start_ms: int,
    end_ms: int,
) -> str:
    """Place a code-templated (non-catalog) effect. Validates the settings string
    syntactically (parse_settings round-trip, known key kinds), merges
    extra_settings with the same first-occurrence-wins override as place_preset,
    resolves the palette via palette_from_colors, then add_effect. Raises
    PresetPlacementError on worked=false, ValueError on bad timing/target."""

Shared code: the extra-settings merge loop and the timing/target guards get extracted from place_preset into private helpers (_merge_extra_settings(settings, extra), _check_timing_and_target(...)) both functions call — no behavior change to the preset path, provable by the existing tests/test_editing.py and the golden test.

effect_emitter.apply_instructions branches per instruction (around line 116):

if ins.direct_settings:
    await place_direct(client, ins.target, ins.effect_type, ins.direct_settings,
                       palette_colors=ins.palette_colors or None,
                       extra_settings=extra,     # still carries B_CHOICE_BufferStyle
                       layer=layer, start_ms=ins.start_ms, end_ms=ins.end_ms)
else:
    await place_preset(...)                      # exactly today's call

Layer accounting (_free_layer/_top_layer/occupancy) and _SKIPPABLE handling are identical for both branches — a failed direct placement degrades to a logged skip, never a crashed run, the same best-effort posture as everything else in the emitter.

3.5 Validation: three rings

  1. Syntactic (hermetic, every build). Round-trip through the existing parser: serialize_settings(parse_settings(s)) == s; every key classifies to a known kind via classify_kind (no "other" except the audited font-picker key); no active timing-track value curves unless the builder deliberately bound one (reuse _has_active_timing_track_curve's logic — inverted: for Faces the track reference is required and its name must equal the track the run writes).
  2. Reference (hermetic for Text, gated for Faces). Text: none needed. Faces: face_definitions(rgb_path, model) -> list[str] (proposed helper in knowledge/layout_semantics.py or a new knowledge/faces.py — it parses rgbeffects.xml faceInfo elements) must return the referenced definition, and the timing-track name must be one the pipeline schedules for writing.
  3. Live (opt-in, -m live). A validate_direct(client, effect_type, settings) mirroring validate_preset's scratch-sequence protocol (clean slate → place → render_all → discard), used once per template per xLights upgrade, not per run.

3.6 Explicitly out of scope: Pictures, Video, Shader (and DMX)

These stay excluded, deliberately:

TypeWhy not now
Pictures / VideoSettings embed file paths; a real feature needs asset acquisition, show-folder copying, path rewriting (the catalog doc itself warns "imports break paths; fix via Bulk Edit Path"), resolution gating (<~50 px canvases turn detailed imagery "to mush"), and content curation — a full asset-management subsystem with no LLM-pipeline leverage until we have assets worth showing.
ShaderReferences .fs shader files (same asset problem), heavy render cost, and per-GPU behavior differences the offline preview renderer cannot reproduce — validation story doesn't exist.
DMXDevice-bound by definition; this layout drives pixels.

The design keeps the door open: each future type is one more builder in direct_settings.py plus its reference ring — no new plumbing.

4. Implementation plan

  1. Probe & freeze the Text template. Hand-author one static and one scrolling Text effect in the pinned xLights version; extract both settings strings from the saved .xsq via parse_settings; commit them as the frozen template + a fixture. Reconcile the §3.2 key table against reality; document any version-gated keys (the DROP_KEYS mechanism at editing.py:22 is the precedent for stripping stale keys).
  2. knowledge/direct_settings.py with DIRECT_TYPES, build_text_settings (and a build_faces_settings skeleton that raises until F-D lands its probe), each returning strings that pass ring-1 validation; unit tests per builder including clamped/invalid inputs (negative font size, unknown movement → ValueError).
  3. Refactor editing.py: extract _merge_extra_settings + guards from place_preset; add place_direct. Existing test_editing.py must pass untouched (behavior- preserving for the preset path — the golden test corroborates).
  4. Schema + emitter: add EffectInstruction.direct_settings: str = ""; branch apply_instructions; extend the emitter's fake-client test (pattern of tests/test_orchestrator.py / test_client.py fakes) asserting a direct instruction reaches add_effect with the exact string, correct layer accounting, and skip-on-failure parity.
  5. Guard rails in QA/realization: assert (test, not runtime) that DIRECT_TYPES ∩ placeable_effect_types() == ∅ so the LLM menu never grows these types silently if the catalog is ever re-mined with different filters; add a placement-rules note that direct types bypass candidate_look_ids-based guards (weave._valid_recipes, beats._accent_look) by never entering recipes.
  6. validate_direct live check (marked live): place the frozen Text template on the Matrix model on a scratch sequence, render, assert worked; wire into the live-verify checklist.
  7. Docs: a short section in docs/architecture/README.md (or the knowledge package docstring) describing the two placement routes and the rule for choosing (mined look when one exists; direct only for DIRECT_TYPES, authored only by deterministic passes).

Estimated blast radius: 2 new files, ~4 edited; no changes to run.py, QA scoring, or any agent prompt. F-C/F-D then consume build_*_settings + direct_settings without further core changes.

5. Testing strategy

  • Hermetic-first (pytest -m "not live"): builder unit tests (round-trip, key-kind audit, override merge, error cases); emitter branch tests against a fake client capturing add_effect calls; schema back-compat test loading a pre-change cached instructions JSON (no direct_settings key) and the golden fixture unchanged — this change must not perturb the golden, since nothing emits direct instructions yet.
  • Template fixtures: the hand-probed settings strings live in tests/fixtures/; a test asserts build_text_settings output differs from the frozen probe only in the deliberately variable keys (text, dir, speed, size) — the "corpus of one" invariant.
  • Parser property tests: for generated inputs, parse_settings(build_text_settings(...)) yields unique keys, values contain no commas, and glyph text with commas/equals is rejected or escaped explicitly (settings strings are CSV of KEY=VALUE — a lyric containing "," would corrupt the string; the builder must strip or substitute, and a test pins that decision).
  • Live (opt-in): validate_direct against a running xLights; visual eyeball that glyphs are legible on the Matrix at distance (F-C's checklist item, but the placement smoke test belongs here).

7. Acceptance criteria

  • ☐ knowledge/direct_settings.py exists with DIRECT_TYPES = {"Text", "Faces"} and a build_text_settings whose output round-trips through parse_settings/serialize_settings and whose every key classifies to a known kind.
  • ☐ The Text template is frozen from a hand-authored probe .xsq committed as a fixture; the builder varies only the documented knobs.
  • ☐ EffectInstruction.direct_settings (default "") exists; pre-change cached instruction JSON and the golden fixture load and compare unchanged.
  • ☐ editing.place_direct places a direct instruction via client.add_effect with palette from palette_from_colors and the same extra-settings override semantics as place_preset (parity test).
  • ☐ apply_instructions routes on direct_settings, with identical layer accounting and skip-on-failure behavior for both branches (fake-client test).
  • ☐ Text and Faces remain absent from placeable_effect_types(); a test asserts the invariant.
  • ☐ A live-marked validate_direct proves the frozen Text template places (worked=true) and renders on a scratch sequence.
  • ☐ Hermetic suite green: pytest -m "not live"; golden untouched.

8. Open questions

  • Should the direct path also serve synthesized non-asset strings someday (e.g. an effect type mined with too few looks)? Recommend no — the catalog is the source of truth for anything minable; keep DIRECT_TYPES minimal.
  • Comma handling in glyph text: substitute (e.g. "," → " ·") or reject? Recommend substitute-with-log — a lyric fragment should never fail a run. Needs the probe to confirm whether xLights itself escapes commas in E_TEXTCTRL_Text when saving (if it does, mirror its escape exactly).
  • Does addEffect accept multi-line text (newlines in a query/body value) for stacked static text? Probe during step 1; v1 can restrict to single-line.
  • Where does the Faces reference-detection helper live — knowledge/layout_semantics.py (already parses rgbeffects.xml) or a new knowledge/faces.py? Leaning layout_semantics.py for cohesion with the other rgbeffects readers; F-D decides.
  • Should skipped direct placements surface in the roadmap-I5 degradation summary with a dedicated reason? (Cheap once I5 lands; flag it there.)
