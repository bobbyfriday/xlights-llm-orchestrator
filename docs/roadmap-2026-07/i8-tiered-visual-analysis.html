1. Why

Every refine iteration currently pays the most expensive model in the fleet to look at up to six stills and six ten-second video clips, with two full markdown guides in its system prompt — and everything it says is advisory only. Meanwhile the critic's four judgment aspects (coverage, color, motion, energy — the VisualFinding.aspect literal in agents/visual_critic.py:36) are largely computable from data the pipeline already has: the .fseq file is the exact per-node RGB of every frame as xLights rendered it (xlights_core/preview/fseq.py parses it into a uint8 [frames, channels] array), and the layout gives the node→group mapping. make_lit_sampler (pipeline/visual.py:105–142) already samples this data — but pointwise, three timestamps per section, reduced to a single lit-pixel count.

We are paying pro-tier prices for an advisor, while a free, deterministic reading of the same render data — one that could legitimately enter the objective gate — goes uncomputed.

2. Current state

2.1 The critique path

make_visual_critique(client, *, save_as, song_key, cache_root, critic=None, max_sections=6, clip_secs=10, enable_video=True, real=None) (visual.py:48) builds the async callable the refine loop invokes each iteration (run.py:224–228, best-effort, findings appended to the QA report). Per call it:

  1. await client.save_sequence(save_as) — flush the current .fseq (visual.py:63); resolve it from the macOS sandbox container, not the show folder (module docstring, media.resolve_artifact).
  2. Construct PreviewRenderer(fseq, rgbeffects, networks) (visual.py:74) — the offline renderer.
  3. use_real = real is not None and await real.refresh(client) (visual.py:79) — prefer the real xLights render when exportable.
  4. Per brief section (max 6): pick the brightest frame offline, take the still and a ≤10s clip from the real render when available, else offline (visual.py:80–89).
  5. Run the multimodal agent, convert to advisory Findings, persist a human review bundle to cache_root/song_key/visual_review/iterN/ (stills, clips, findings.json, review.md) (visual.py:93–99).

qa/coverage.py consumes the sampler at exactly three timestamps per section (_SAMPLES = (0.25, 0.5, 0.75), line 16), takes the max, and scores sections with intensity ≥ MIN_INTENSITY = 0.5 against EXPECTATION = 0.6 × intensity of the show's own peak; under FAIL_FRACTION = 0.5 of expectation → an objective error finding. When a sampler is present, coverage joins the objective average (qa/__init__.py:24–28) — the precedent this design generalizes: rendered-pixel metrics already gate, they're just crude. Note the current sampler renders a full 1280×720 PNG per sample through PIL just to count pixels (visual.py:136–140) — the fseq array could answer the same question with one vectorized comparison.

parse_models (preview/layout.py:113–165) yields per-model start_channel/n_pixels (Default-preview models only, line 126); group membership lives in the same rgbeffects.xml under <modelGroups>, which xlights_core/knowledge/layout_semantics.py already walks (e.g. patch_sem_gridsize, patch_view) — so the node→group index is a small join of two existing parsers.

RealRender (visual.py:155–215) exists because the offline projection is an approximation of what xLights shows: it exports the true render via client.export_video_preview (media-attached sequences only — a media-less export crashes xLights, the guardrail at visual.py:172–174), caches by .xsq mtime, and serves offset-corrected frames/clips via ffmpeg. Consumers fall back to offline.

3. Proposed design

Three tiers, cheapest first; each tier only escalates what it can't settle.

3.1 Tier 0 — deterministic fseq metrics (every iteration, ~free, no LLM)

New pure-numpy analysis over (header, frames) + a group channel index. All signatures below are proposed.

3.1.1 The group channel index

# xlights_core/preview/metrics.py  (proposed — core-side, pure, no orchestrator imports)
def group_channel_index(rgbeffects_path, networks_path,
                        groups: list[str] | None = None) -> dict[str, np.ndarray]:
    """group name -> int64 array of R-channel offsets for every member-model pixel.
    Join of parse_models() (start_channel, n_pixels) with <modelGroups> membership
    (models="a,b,c" attribute). Channels bounded against header.channels like
    PreviewRenderer.__init__ (render.py:48). Unknown/empty groups omitted."""

3.1.2 The per-group time series (computed once per iteration)

class FseqSeries:
    """Per-group brightness/lit-fraction/motion series over the whole song. One pass, vectorized.
    A 4-min song at 50ms frames × ~20k channels is a ~92 MB uint8 array — loaded once,
    reduced immediately to per-group float32 series (~4800 × n_groups)."""
    def __init__(self, fseq_path, rgbeffects_path, networks_path,
                 lit_threshold: int = 30):        # same threshold as make_lit_sampler
        header, frames = load_fseq(fseq_path)
        # per node: v[t, n] = max(R,G,B) — brightness proxy, matches the sampler's max(axis=2)
        # B[g][t]   = v[t, nodes_g].mean()                     (brightness)
        # L[g][t]   = (v[t, nodes_g] > lit_threshold).mean()   (lit fraction)
        # M[g][t]   = |v[t] - v[t-1]|[nodes_g].mean()          (frame-delta motion)
    def section_slice(self, start_ms, end_ms) -> ...          # frame-index window

3.1.3 The five metric families

MetricComputation (per section)Replaces / catches
Coverage (fseq:coverage)Mean + max of the global lit fraction over the whole section window, plus per-group lit fractions. Scored against energy expectation exactly as qa/coverage.py does (peak-relative, intensity-gated) — but from the full series instead of 3 samples, and attributable: “SEM_ARCHES dark for 82% of the chorus”.Generalizes coverage.evaluate; catches “dark during the peak” between the sample points; kills the render-a-PNG-to-count-pixels sampler.
Motion (fseq:motion)Per-group mean frame-delta M[g] over the section, normalized by that group's brightness (so a dim-but-moving chase isn't scored static). Section score = lit-weighted mean across groups; flag when a section with intensity ≥ 0.5 has near-zero normalized delta.The critic's “static during a high-energy moment”, with exact group attribution the critic can't give. Also the pixel-level ground truth I7 needs (instruction counts are the proxy; this is the photons).
Music-sync (fseq:sync)Cross-correlate the positive part of d/dt of global (and per-pulse-group) brightness with a beat impulse train from SongAnalysis.beats, ±1 frame tolerance. Score = correlation at lag 0 normalized against a circular-shift baseline (shuffle-robust). Complements qa/sync.py, which scores instruction timestamps; this scores whether the render actually pulses on the grid (an on-beat instruction whose effect fades in late fails here and nowhere else).New capability: sync as a rendered score, not an authored one.
Color / palette adherence (fseq:color)Hue histogram (e.g. 24 bins, lit nodes only, vectorized RGB→hue) per section vs the planned section palette (names resolved through xlights_core.knowledge.colors, as brief_editor._color_hex_map does). Two checks: (a) adherence — histogram mass within Δhue of planned hues; (b) distinctness — when the palette promises k distinct hues, the rendered histogram must show ≥ min(k, 2) separated peaks. Check (b) is the “gold + amber + warm white reads as one color” failure as arithmetic: three planned colors whose pairwise hue distance is under ~20° collapse to one perceived color.The critic's color aspect, plus a failure it demonstrably misses at 640px.
Section-signature similarity (fseq:rhyme, fseq:range)Per-section signature vector: brightness stats, motion stats, hue histogram, per-group activity vector. Cosine similarity between sections sharing a repetition_map label → pixel-level repetition-rhyme (chorus 2 should look like chorus 1); spread between the quietest and loudest signatures → pixel-level dynamic-range. These are the rendered-side complements of the instruction-level repetition-rhyme/dynamic-range metrics the improve-musicality proposal defines for qa/musicality.py — same names, two layers of the same question.Directly measures F-A Phase 1's goal (repeats must rhyme) and Phase 2's (contrast must exist).

3.1.4 Orchestrator-side wrapper and the objective gate

# xlights-orchestrator/qa/fseq_metrics.py  (proposed)
def evaluate(plan, analysis, series: FseqSeries | None,
             *, repetition_map=None) -> tuple[dict[str, float], list[Finding]]:
    """subscores {'fseq_coverage':…, 'fseq_motion':…, 'fseq_sync':…, 'fseq_color':…}
    + Findings with metric='fseq:*'. series None / unreadable → ({}, []) — neutral,
    never gate blind (the qa/coverage.py posture, line 31)."""

Wiring: qa.evaluate(...) (qa/__init__.py:16) grows an optional fseq_series keyword next to sampler; the refine loop's _report closure (run.py:172–184) constructs/refreshes the FseqSeries after the save-flush (reusing the mtime-invalidation pattern from make_lit_sampler, visual.py:132–137) and passes it through. Unlike the LLM critique, these findings may be objective=True — they are deterministic and reproducible, the stated bar for the gate. Rollout in two steps: land everything objective=False (advisory) for a few logged runs, calibrate thresholds against the revision log, then flip coverage/motion/sync to objective (color and rhyme likely stay advisory — taste-adjacent). Escape hatch: XLO_FSEQ_METRICS=0 disables the whole family, mirroring XLO_REFINE_SKIP_OBJECTIVE's pattern. When fseq_series is supplied, make_lit_sampler/3-point coverage retire (the series strictly dominates them).

3.2 Tier 1 — contact sheets on a cheap model (flagged/changed sections only)

  • The artifact: render_contact_sheet(section, n=8) — 6–9 beat-aligned frames (downbeats spread evenly across the section, from SongAnalysis.beats, rather than one brightest frame) tiled into a single ~512px montage with tiny timestamp captions. One image ≈ 1–2k tokens vs ~2.5k+ per clip + ~1.2k per still today. Optional ninth tile: a per-pixel standard-deviation heatmap over the section's frames (np.std across the section window, colormapped) — motion rendered as a static picture, so even a stills-only model perceives dynamics. Natural home: a method on PreviewRenderer (preview/render.py) beside render_frame/render_clip, reusing _project/_frame_img.
  • The prompt: drop both guides (with_guides → plain prompt): the critique needs the four aspects and the musical context, not 51k chars of sequencing pedagogy. Saves ~13k tokens per call by itself. Keep the per-section context JSON (visual_critic.render_input shape) and the “judge against the music at that moment” framing — that instruction is what makes darkness-in-a-lull acceptable, and it must survive the trim.
  • The model: a new visual_critic_lite role in models/config.yaml routed to the worker tier (anthropic: claude-sonnet-4-6 / gemini: gemini-3.1-flash-lite), built via the existing build_agent path so model_snapshot() records it.
  • The scope — only what changed or what Tier 0 flagged: hash each section's instruction slice (sha256 of the sorted model_dump_jsons — the stage-cache idiom) and keep {section_index: hash} across iterations. Tier 1 critiques a section iff its hash changed since its last critique or Tier 0 raised a finding on it. Today the critique re-judges all 6 sections every iteration even though regen is section-scoped; after this, iteration 2 of a one-section revision sends one montage. Unchanged sections carry their previous findings forward, tagged stale (or drop them — open question 3).

3.3 Tier 2 — today's still+clip on the pro model, rarely

The current make_visual_critique path survives unchanged as the escalation: full still + 10s clip, planner-tier model, guides optional. Triggers:

  • Tier disagreement: Tier 0 says error but Tier 1 says the section fits its moment (or vice versa) — the case that genuinely needs gestalt judgment.
  • Churn: a section revised ≥2 times per the loop's ledger (the anti-oscillation record already tracks this) keeps failing — ask the strong model why.
  • Final gate: one whole-show pass near the end of the run (see §3.4) — the “random vs intentional” judgment is the one thing that needs a strong multimodal model, and it doesn't need asking six times per iteration.

3.4 The offline-vs-real fidelity caveat

Precision matters here: the .fseq channel data is the true render — xLights wrote it — so Tier 0 measures ground-truth per-node values, not an approximation. What is approximate is everything downstream of the channels: (a) the offline projection (orthographic splat, render.py:54–78) vs xLights' preview drawing — geometry, splat size, background — which is why RealRender exists and why Tier 1/2 prefer real frames when real.refresh() succeeds; (b) staleness — metrics must run only after the save-flush (the ordering _report already enforces at run.py:174–179); (c) scope — the offline parsers include Default-preview models only (layout.py:126), so props parked in other layout groups are invisible to both metrics and montages. Therefore: keep one real-render check per run — the Tier 2 final gate runs on RealRender frames whenever the media-attached export is possible — as the ground-truth backstop for anything the offline path miscounts.

4. Implementation plan

  1. preview/metrics.py: group_channel_index (modelGroups join, channel-bound checks per render.py:48) + FseqSeries (B/L/M series, section slicing). Unit tests against synthetic fseq/layout fixtures (§5). No behavior change anywhere yet.
  2. qa/fseq_metrics.py: the five metric families → subscores + Finding(metric="fseq:*", objective=False); thread fseq_series through qa.evaluate and _report; retire the 3-point sampler path when a series is present (coverage numbers must be shown equivalent-or-better on a real run first). Add XLO_FSEQ_METRICS kill switch.
  3. Calibration pass: run 2–3 refine-enabled songs; compare fseq findings against the same iterations' LLM findings and the review bundles in the revision log; set thresholds; document them in the module like qa/coverage.py does.
  4. Tier 1: render_contact_sheet (+ std-heatmap tile), lite prompt + visual_critic_lite role, section-hash tracking, make_tiered_critique factory; swap the default in run_pipeline (run.py:451–454) behind the same visual_critique=None seam so injected fakes keep working.
  5. Tier 2 demotion: escalation triggers (disagreement, churn-≥2, final gate on RealRender); the old make_visual_critique becomes the escalation implementation.
  6. Flip the gate: promote fseq:coverage/motion/sync to objective=True after step 3's evidence; announce in docs/usage.md (env knobs) and regenerate nothing — the golden test doesn't cover refine.
  7. OpenSpec: this modifies the visual-critique and show-refinement capabilities (tiering; objective rendered metrics) — write the change doc; I1's token telemetry, if landed, provides the before/after cost evidence for the archive.

5. Testing strategy

Hermetic-first; the building blocks already exist in tests/test_preview.py: _write_fseq (writes a valid zstd-compressed PSEQ from raw bytes) and _minimal_layout (tiny rgbeffects/networks XML). Extend, don't reinvent:

  • Fixture generator: a helper that synthesizes an fseq with known physics — e.g. 2 groups × 4 nodes, 100 frames at 50ms: group A blinks full-white exactly on a 500ms beat grid, group B holds constant dim red, group C stays dark. Assertions: motion ranks A ≫ B ≈ 0; sync score for A ≈ max and for B ≈ baseline; coverage flags C only when its section intensity ≥ 0.5; lit-threshold agrees with make_lit_sampler's (>30) on the same frames.
  • Group index: modelGroups fixture with 2 groups + 1 model in both → channels bounded, overlap allowed, unknown group omitted; a model with an unresolvable StartChannel is skipped exactly as parse_models skips it.
  • Color: frames of pure red vs planned ["Red", "Green"] → adherence ~0.5 and a missing-green finding; three near-identical warm hues vs a 3-color palette → distinctness failure (the gold/amber/warm-white case as a literal fixture).
  • Rhyme/range: duplicate a section's frames → similarity ≈ 1.0 for shared labels; invert brightness for one section → dynamic-range > 0.
  • Contact sheet: deterministic frame selection from a fixed beat list; output PNG dimensions and tile count; std-heatmap tile is non-uniform iff frames differ.
  • Tier routing (no LLM): fake lite/pro critics counting calls (the test_refine.py duck-type style) + a scripted Tier 0: assert iteration 2 after a single-section regen critiques exactly one section; disagreement triggers exactly one Tier 2 call; unchanged+unflagged sections trigger none; XLO_FSEQ_METRICS=0 yields today's behavior.
  • Neutrality: missing fseq / zstd error / empty groups → ({}, []) and an unchanged objective score (the never-gate-blind posture, mirroring test_refine.py::test_sync_empty_beats_neutral).
  • Live (marked live): one end-to-end run asserting the review bundle contains metrics.json + montages and that the real-render final gate fired — the only non-hermetic piece, following test_real_render.py's precedent.

7. Acceptance criteria

  • [ ] preview/metrics.py computes per-group brightness/lit-fraction/motion series from a fixture fseq with unit-tested known-physics results.
  • [ ] qa/fseq_metrics.py emits the five metric families as subscores + fseq:* findings; unreadable inputs are neutral (never gate blind); XLO_FSEQ_METRICS=0 disables cleanly.

  • [ ] Tier 1 sends one ≤2k-token beat-aligned montage per changed-or-flagged section to a worker-tier visual_critic_lite role, with the guides removed from its prompt; unchanged sections are not re-critiqued.
  • [ ] Tier 2 (today's still+clip, planner tier) fires only on disagreement, ≥2-revision churn, or the once-per-run final gate — which runs on the real render when the media-attached export is possible.
  • [ ] Per-run visual-analysis token spend drops >10× on a like-for-like refine run (measured via I1 telemetry if landed, else by counted calls × estimated sizes in the revision log).
  • [ ] Review bundles still contain everything a human needs: montages, metrics.json, and any Tier 2 media; hermetic suite green; golden fixture untouched.

8. Open questions

  1. Brightness proxy: max(R,G,B) (matches the sampler) vs luma (0.2126R+0.7152G+0.0722B)? Max is consistent with existing gates; luma is perceptually truer for the color metrics. Proposal: max for coverage/motion (continuity with coverage.py), luma only inside fseq:color.
  2. Where does sync live long-term? fseq:sync and qa/sync.py both feed the objective average — double-counting sync-ish signal. Merge them (rendered sync supersedes instruction sync when a series exists), or keep both as independent parts?
  3. Carry-forward vs drop for unchanged sections' Tier 1 findings — carrying keeps the Judge informed but risks staleness; dropping under-informs on defects the loop chose not to fix. (Leaning: carry, tagged, and let the plateau detector's signature ignore tagged findings.)
  4. Should Tier 0 metrics also run at generate time (before refine) to gate the skip-high-objective decision (run.py:211–216) with rendered evidence — the skip currently trusts instruction-side QA plus the 3-point sampler only? Cheap once the series exists; changes the meaning of REFINE_SKIP_OBJECTIVE's calibration.
  5. Montage color fidelity vs downscale: 512px across 9 tiles ≈ 170px per frame — enough for coverage/motion gestalt, marginal for palette nuance. Does Tier 1 get one extra full-res still for its worst Tier-0-scored moment?
  6. F-J alignment: headless/preview-only iteration wants exactly this stack (fseq + metrics, no live app). Should FseqSeries construction avoid any client dependency from day one (yes — it only needs paths) and should that constraint be written into the module docstring as a contract?
