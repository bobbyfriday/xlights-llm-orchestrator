1. Why

The project has been running an active cost-reduction campaign — guide-extract trimming, tiered per-role model routing (models/config.yaml routes analysts/generators to Sonnet-tier and the director/synthesizer/judge/visual-critic to Opus-tier), and the refine-loop skip-high-objective gate (REFINE_SKIP_OBJECTIVE = 88 in pipeline/run.py:62, justified by "revision-log analysis (42 runs)") — without a single token being counted anywhere in the codebase. Every cost claim to date is an estimate. The visual-critique cost analysis in roadmap item I8 ("roughly 30–40k multimodal tokens per iteration") is likewise an eyeball figure.

PydanticAI already computes what we need: every AgentRunResult carries a usage() method returning a RunUsage dataclass with request/response token counts, including prompt-cache splits. We throw it away at all seven call sites — the codebase's universal pattern is (await agent.run(prompt)).output, which discards the result wrapper (and the usage on it) in the same expression.

Meanwhile, the revision log (revision_log.py) already records a per-iteration models snapshot ({"director": "anthropic:claude-opus-4-8", ...} via models/registry.py::model_snapshot(), line 38) — the JSONL is one field away from being a per-run cost ledger. This item adds that field, a price table, and a per-run summary, turning every future run into evaluation data for the F-G dashboard and making the F-H provider A/B comparison meaningful (you cannot A/B Gemini vs Claude on cost/quality without cost numbers).

2. Current state

2.1 What PydanticAI gives us (verified against the installed version)

The workspace resolves pydantic-ai>=1.0 to pydantic_ai 2.5.0 (checked in the synced .venv, 2026-07-05). pydantic_ai.usage.RunUsage is a dataclass with exactly these fields:

RunUsage.incr() is an in-place accumulator PydanticAI itself uses to sum usage across a multi-request run — we can reuse it rather than writing our own summation. Note that input_tokens is the uncached remainder: total prompt size is input_tokens + cache_write_tokens + cache_read_tokens. Multimodal (visual-critic) requests report image tokens inside input_tokens.

2.3 The revision log today

revision_log.py defines RevisionLogRecord (line 33) with per-iteration scores, findings, revisions, and a per-role model snapshot — but no usage:

class RevisionLogRecord(BaseModel):
    run_id: str
    iteration: int
    song_key: str
    ts: str
    kind: Literal["iteration", "finalize"] = "iteration"
    ...
    models: dict[str, str] = {}          # role -> "provider:model" (model_snapshot())
    review_bundle: str | None = None

Records are written by the _record() closure in _refine_loop (pipeline/run.py:201), which is invoked at: the skip-high-objective early return (line 214), accept/stop (line 233), plateau (line 240), every applied iteration (line 290), and the finalize record (line 305). Writes are best-effort — RevisionLog.write() swallows all exceptions with a log.warning (line 112) — and hermetic tests use NullRevisionLog.

Two structural gaps matter for telemetry: (a) the revision log only exists during refine — the panel (interpret), director (design), and first-pass generation run before RevisionLog is constructed at pipeline/run.py:455–459; (b) a --no-log/hermetic run has no log at all. So the revision log alone cannot be the usage collector — usage needs a run-scoped accumulator that outlives (and pre-dates) the refine loop, with the revision log as one view of it.

2.4 Model routing and where prices belong

models/config.yaml is the single data-driven routing table (role → per-provider model id), loaded once via functools.lru_cache in registry._cfg(). It already carries per-role model ids for both providers, so it is the natural home for a price table — a price is a property of a model id, exactly the granularity this file already speaks:

3. Proposed design

3.1 Overview

Three pieces, all additive:

  1. A run-scoped usage collector (telemetry.py, proposed new module) with a one-line capture helper called at each of the seven .run() sites.
  2. A JSONL schema extension: per-iteration usage deltas on RevisionLogRecord, and run totals + cost on the finalize record; plus a standalone usage.json artifact and a log.info summary line so non-refine runs get telemetry too.
  3. A price table in models/config.yaml and a pure cost function estimate_cost().

3.2 The collector (proposed: xlights_orchestrator/telemetry.py)

A contextvar-scoped accumulator, so no signature in the pipeline changes (the panel is reached through the injected interpret callable whose signature (song_path, sa) -> MusicBrief we do not want to disturb; contextvars propagate through asyncio.gather, so the concurrent analysts each record correctly):

# PROPOSED — new module xlights_orchestrator/telemetry.py
from __future__ import annotations
import logging
from contextvars import ContextVar
from pydantic import BaseModel

log = logging.getLogger(__name__)

class RoleUsage(BaseModel):
    """Accumulated tokens for one role (JSONL-stable shape)."""
    requests: int = 0
    input_tokens: int = 0
    output_tokens: int = 0
    cache_read_tokens: int = 0
    cache_write_tokens: int = 0

class UsageLog:
    """Run-scoped accumulator: totals per role + a delta window for the revision log."""
    def __init__(self) -> None:
        self.total: dict[str, RoleUsage] = {}
        self._delta: dict[str, RoleUsage] = {}

    def record(self, role: str, usage) -> None: ...        # add a RunUsage into both maps
    def drain_delta(self) -> dict[str, RoleUsage]: ...     # pop + reset the delta window
    def snapshot(self) -> dict[str, RoleUsage]: ...        # copy of totals

_current: ContextVar[UsageLog | None] = ContextVar("xlo_usage_log", default=None)

def start_run() -> UsageLog: ...          # create + install; returns the log
def current() -> UsageLog | None: ...

def record(role: str, result) -> None:
    """Capture result.usage() into the active UsageLog. BEST-EFFORT — a fake agent
    without .usage(), or no active log, is silently fine (log.debug only)."""
    try:
        ul = _current.get()
        if ul is None: return
        u = result.usage()
        ul.record(role, u)
    except Exception as exc:  # noqa: BLE001 — telemetry never breaks a run
        log.debug("usage capture skipped for %s: %s", role, exc)

Design decisions and the alternatives rejected:

DecisionAlternative consideredWhy this way
Explicit telemetry.record(role, res) at each call siteWrap Agent in registry.build_agent() so capture is automaticThe wrapper would miss context we want later (section index, iteration) and would not cover injected test fakes anyway; seven explicit lines are greppable and match the codebase's "plain async stages" style. build_agent stays a pure constructor.
ContextVar, not a State fieldThread a collector through Statepanel.run_panel and make_visual_critique's closure don't see State construction; contextvar reaches every await point under run_pipeline without signature churn. State gets a read-only view at finalize instead.
Capture requires the wrapper, so call sites split the one-linerKeep (await agent.run(p)).outputEach instrumented site becomes two lines: res = await agent.run(p) / telemetry.record("judge", res); verdict = res.output. Mechanical, no behavior change.
Per-role granularity in the log; per-site context deferredRecord section_index/iteration per event nowThe revision-log record already carries iteration; delta-per-record gives iteration attribution for free. Per-section attribution is a cheap later extension (an optional events list) once F-G asks for it — see Open questions.

3.4 JSONL schema extension (backward & forward compatible)

# revision_log.py — extended (PROPOSED fields marked)
class RevisionLogRecord(BaseModel):
    ...
    models: dict[str, str] = {}
    review_bundle: str | None = None
    usage: dict[str, RoleUsage] = {}          # NEW — per-role tokens SINCE THE PREVIOUS RECORD
    usage_total: dict[str, RoleUsage] = {}    # NEW — populated on kind="finalize" only
    cost_usd: float | None = None             # NEW — finalize only; None when any price unknown

The _record() closure in _refine_loop adds usage=usage_log.drain_delta() to every record, so each iteration record carries exactly the tokens spent producing it (judge + critic + regens + redesigns of that iteration). The finalize record (pipeline/run.py:305) additionally carries usage_total=usage_log.snapshot() and cost_usd=estimate_cost(...). Tokens spent before the loop (panel, director, first-pass generation) appear in the delta of the first record written — acceptable for iteration analysis, and fully broken out in usage.json (below). Example finalize line:

{"run_id":"20260705T031500Z","iteration":3,"kind":"finalize","objective_score":91,...,
 "models":{"director":"anthropic:claude-opus-4-8","generator":"anthropic:claude-sonnet-4-6",...},
 "usage_total":{"generator":{"requests":9,"input_tokens":142000,"output_tokens":31000,
                             "cache_read_tokens":0,"cache_write_tokens":0},
                "visual_critic":{"requests":3,"input_tokens":96500,"output_tokens":4100,...},
                "judge":{...},"analyst":{...},"synthesizer":{...},"director":{...}},
 "cost_usd":2.41}

Markdown view: _render_md() gains one line in the tail block, mirroring the models line: **Tokens:** generator 142k→31k · critic 97k→4k · … · **$2.41** on finalize records only (iteration records stay uncluttered).

3.5 Per-run cost summary and the price table

Price table — a new top-level pricing: block in models/config.yaml, keyed by bare model id (same ids the roles block uses), all values in USD per million tokens. Anthropic list prices below are current published rates (2026-06); cache-read is 0.1× input and cache-write 1.25× input per Anthropic's caching pricing. Gemini rows must be filled from Google's published rates at implementation time — do not guess them; a missing entry makes cost None, never zero:

# models/config.yaml — PROPOSED addition
# USD per 1M tokens. Cost is DATA, not code: reprice here, no code change.
# A model absent from this table yields cost_usd: null (unknown ≠ free).
pricing:
  claude-opus-4-8:   {input: 5.00, output: 25.00, cache_read: 0.50, cache_write: 6.25}
  claude-sonnet-4-6: {input: 3.00, output: 15.00, cache_read: 0.30, cache_write: 3.75}
  # TODO(fill from Google's published rates before flipping XLO_PROVIDER=gemini runs):
  # gemini-3.1-pro-preview:  {input: ..., output: ...}
  # gemini-3.1-flash-lite:   {input: ..., output: ...}
  # gemini-3.5-flash:        {input: ..., output: ...}

Cost function — pure, in models/registry.py next to the config loader (it owns _cfg()):

# PROPOSED — models/registry.py
def price_for(model_id: str) -> dict[str, float] | None:
    """Bare-model-id row from config.yaml's pricing table; None if unlisted."""
    return _cfg().get("pricing", {}).get(model_id)

def estimate_cost(models: dict[str, str], usage: dict[str, "RoleUsage"]) -> float | None:
    """Σ over roles of tokens × the role's model rates. `models` is the same
    role -> 'provider:model' map model_snapshot() produces (strip the prefix).
    Returns None if ANY role with nonzero usage lacks a price row."""

Summary line — at the end of run_pipeline() (after step 6, before return st), unconditionally (refine or not):

log.info("run usage: %s tokens in / %s out across %d requests — est. $%.2f",
         total_in, total_out, total_requests, cost)   # "$unknown" when cost is None

usage.json artifact — written best-effort next to the stage caches at cache_root()/<song_key>/usage.json (pipeline/cache.py paths), one JSON object per run appended-or-overwritten as a list keyed by run_id: {run_id, ts, provider, models, usage_total, cost_usd}. This is what covers non-refine runs (no revision log) and gives F-G a second, simpler input. Guarded by the same log_revisions-independent best-effort pattern used for the song_analysis cache (pipeline/run.py:379–384).

3.6 Backfill / compatibility with existing logs

  • Old JSONL lines validate unchanged: the three new fields default to {}/{}/None, so RevisionLogRecord.model_validate_json() over a pre-I1 log succeeds. This must be pinned by a test (§5).
  • No retroactive backfill. Token counts are unrecoverable for past runs; the F-G analyzer must treat absent usage_total as "unknown", rendered distinctly from $0.00. Convention: cost_usd is None ⇒ unknown; 0.0 is reserved for genuinely-zero (fully cached, zero-LLM) runs.
  • Detection heuristic for the analyzer: records with usage == {} and usage_total == {} and a ts before the I1 ship date are pre-telemetry; after it, an empty map on a finalize record means "no LLM calls" (e.g. skip-high-objective on fully cached stages).
  • The markdown view is append-only prose — no compat concern.

3.7 How this unblocks F-G and F-H

  • F-G (cost & quality dashboard): the dashboard is "an analyzer over the revision-log JSONL". With per-iteration usage deltas + obj_delta already on each record, it can compute cost-per-run, cost-per-iteration, cost-per-objective-point-gained, and which role dominates spend (the I8 hypothesis — visual critic ≫ everything — becomes measurable), with zero further instrumentation. Roadmap sizes F-G at "S once I1 lands" for exactly this reason.
  • F-H (provider A/B harness): an A/B run sets XLO_PROVIDER=gemini and re-runs a fixture song; the finalize records of the two runs then differ in models, usage_total, cost_usd, objective_score, and iteration — the complete diff the harness needs. The pricing table keyed by bare model id (provider-agnostic) is what makes the dollar comparison symmetric.
  • I8 (tiered visual critique): before/after visual_critic usage numbers turn ">10× cheaper" from an estimate into a measured claim.

4. Implementation plan

  1. Create packages/xlights-orchestrator/src/xlights_orchestrator/telemetry.py with RoleUsage, UsageLog (record/drain_delta/snapshot), the _current ContextVar, and module functions start_run(), current(), record(role, result). record() maps RunUsage fields → RoleUsage (requests, input_tokens, output_tokens, cache_read_tokens, cache_write_tokens; audio fields ignored for now) and is wrapped in the repo-standard best-effort except Exception with log.debug.
  2. Instrument the seven call sites (§3.3): split each (await agent.run(...)).output into result-then-record-then-output. Import telemetry at module top per existing import style.
  3. Install the collector: telemetry.start_run() at the top of run_pipeline() (pipeline/run.py:345, before step 1) and regen_section() (pipeline/regen.py:105).
  4. Extend RevisionLogRecord with usage, usage_total, cost_usd (defaults as §3.4); extend _render_md() with the finalize tokens/cost line.
  5. Wire the deltas: in _refine_loop, thread the active UsageLog (fetched once via telemetry.current()) into _record(); add usage= to every record and usage_total=/cost_usd= to both finalize call sites (skip-high-objective at line 214 and the end-of-loop record at line 305).
  6. Add the pricing: block to models/config.yaml (Claude rows real; Gemini rows commented TODO) and price_for()/estimate_cost() to models/registry.py.
  7. Summary + artifact: end of run_pipeline() — log.info summary and best-effort usage.json write under cache_root()/key/.
  8. Docs: one paragraph in docs/usage.md (where the cache layout is described) documenting usage.json, the JSONL fields, and the None-vs-zero cost convention.

5. Testing strategy

All hermetic (repo convention: pytest with addopts = "-m 'not live'"; agents are injected fakes — see tests/test_refine.py's stubbed Judge/regen pattern).

  • tests/test_telemetry.py (new):
    • UsageLog accumulation: two record()s for one role sum; drain_delta() empties the window but not totals.
    • Fake result objects: SimpleNamespace(output=..., usage=lambda: SimpleNamespace(input_tokens=100, output_tokens=10, cache_read_tokens=0, cache_write_tokens=0, requests=1)) — proves capture needs only duck-typed .usage().
    • Fakes without .usage() never raise: telemetry.record("judge", SimpleNamespace(output=1)) is a no-op — this is the guarantee that keeps every existing fake-agent test green without modification.
    • ContextVar isolation: no active log ⇒ no-op; concurrent asyncio.gather tasks record into the same run's log (panel-shaped test).
  • tests/test_revision_log.py (extend):
    • Round-trip a record with usage/usage_total/cost_usd through JSONL.
    • Back-compat pin: a stored pre-I1 JSONL line (string literal in the test) still validates and yields usage == {}, cost_usd is None.
    • Markdown finalize line renders tokens + cost; iteration records don't.
  • tests/test_orchestrator.py / test_refine.py (extend): run the existing hermetic pipeline with fakes whose .usage() returns fixed numbers; assert the finalize record's usage_total matches the sum and that per-iteration deltas partition it.
  • Cost math unit tests: estimate_cost() with the real config.yaml Claude rows (142k in / 31k out on sonnet-4-6 ⇒ $0.891); any role with usage on an unpriced model ⇒ None; zero usage ⇒ 0.0; cache tokens priced at the cache rates.
  • Golden pipeline (test_golden_pipeline.py): must stay byte-identical — telemetry writes no stage-cache artifacts and the golden test injects fakes without .usage(), so nothing changes; run it to confirm.
  • Live smoke (opt-in, -m live): one real analyst run asserting usage_total["analyst"].input_tokens > 0.

7. Acceptance criteria

  • ☐ All seven .run() call sites record usage; a grep for ).output adjacent to agent.run( finds no un-instrumented LLM site.
  • ☐ A refine run's JSONL contains per-iteration usage deltas and a finalize record with usage_total and cost_usd; the deltas sum to the total.
  • ☐ A non-refine run (and xlo regen) produces usage.json and the log.info cost summary line.
  • ☐ models/config.yaml has a pricing: block; Claude rows priced ($5/$25 opus-4-8, $3/$15 sonnet-4-6 per MTok + cache rates); unlisted model ⇒ cost_usd: null.
  • ☐ A pre-I1 JSONL line validates against the extended RevisionLogRecord (pinned by test).
  • ☐ Every pre-existing hermetic test passes unmodified (fakes without .usage() are no-ops).
  • ☐ Cached-stage runs report only the tokens actually spent (a fully-cached run shows zero generator/director/analyst usage).
  • ☐ uv run ruff check and the golden pipeline snapshot stay green.

8. Open questions

  1. Per-analyst split — record analysts as one analyst role (matches routing/pricing keys) or as analyst:structure etc.? Proposal: one role now; add an optional details: dict[str,int] on RoleUsage if F-G wants the split.
  2. Per-section / per-stage attribution — F-G's "which sections churn" view could use section_index on generator events. Defer: needs an events list rather than role maps; the record shape above leaves room (usage stays a map; an events field can be added compatibly).
  3. Gemini pricing source of truth — config comment says flash-lite was "live A/B'd"; who owns keeping Google's rates current, and should estimate_cost warn (once) when an active provider has unpriced models?
  4. Thinking tokens — Anthropic bills thinking as output; verify PydanticAI folds them into output_tokens (expected) or exposes them in details, and pin whichever with the live smoke test.
  5. usage.json shape — list-of-runs per song key (proposed) vs one file per run under visual_review-style subdirs? List-per-song is simpler for the dashboard's scan.
