1. Why

Every model-routing decision in models/config.yaml today is justified by an informal, undocumented experiment. The evidence is literally a YAML comment. From packages/xlights-orchestrator/src/xlights_orchestrator/models/config.yaml (lines 11 and 20), verbatim:

  generator:                                  # per-section effects (worker tier)
    anthropic: {model: claude-sonnet-4-6, thinking: adaptive}
    gemini:    {model: gemini-3.1-flash-lite}   # live A/B'd: valid structured output, rules 92
  judge:                                      # scores the draft + scoped revisions (planner)
    anthropic: {model: claude-opus-4-8, thinking: adaptive, effort: high}
    gemini:    {model: gemini-3.5-flash}      # scoring a structured QA report ≠ pro-tier work

"live A/B'd: valid structured output, rules 92" is one hand-run comparison, on one song, at one point in time, recorded as a comment. There is no way to re-run it, no record of what was held constant, and no cost dimension at all (the June roadmap's F7 was explicitly "blocked on missing token telemetry"). Meanwhile the config header advertises exactly the switch an A/B harness needs (config.yaml lines 1–3, verbatim):

# Per-role model routing. Pick the provider with `default_provider` or the XLO_PROVIDER
# env var (no code change). Each role lists a model per provider. IDs are bare.
default_provider: anthropic   # override at runtime with XLO_PROVIDER=gemini

F-H formalizes that manual practice: run a fixture song through two (or more) provider routings under controlled conditions, and report per-role and overall differences in QA scores, refine convergence, wall time, and — once I1 lands — token cost. The output is decision support for config.yaml: "route the judge to gemini-3.5-flash" becomes a claim backed by runs you can point at and repeat.

2. Current state

2.1 Provider routing is global, env-switched, consulted at agent-construction time

models/registry.py (72 lines) is the whole routing layer. The provider in effect is a single global (lines 26–28):

def active_provider() -> str:
    """Provider in effect: XLO_PROVIDER env override, else config default."""
    return os.environ.get("XLO_PROVIDER") or _cfg().get("default_provider", "anthropic")

model_string(role, *, provider=None) (line 31) and _settings(role, *, provider=None) (line 48) both already accept an explicit per-call provider — but nothing passes it: build_agent (lines 64–71) calls both with defaults, so XLO_PROVIDER is the only runtime switch and it flips all six roles at once (director, generator, analyst, synthesizer, judge, visual_critic). There is no per-role override today. The env var is read at each call, not cached (_cfg() caches the YAML via functools.lru_cache, line 21, but active_provider() re-reads os.environ every time), and agents are built lazily inside the pipeline (e.g. director_mod.director_agent() at pipeline/run.py:413, judge_mod.judge_agent() at run.py:142, the panel inside _default_interpret at run.py:77) — so changing the env between in-process pipeline runs genuinely re-routes them. model_snapshot() (registry lines 38–45) records role → provider:model into every revision-log record via run.py:466.

Sampling control: there is none, deliberately. Registry docstring (lines 4–5): "sampling params (temperature/top_p/top_k) are never set (removed on Opus 4.8 → 400)", enforced at line 60 ("Deliberately never set temperature/top_p/top_k") and asserted by tests/test_orchestrator.py::test_registry_claude_settings_thinking_no_sampling (line 56). Gemini gets provider defaults (_settings returns None for non-anthropic, line 51). No seed, no temperature — runs are not reproducible; section 3.5 takes this seriously.

2.2 Stage caching is keyed by song content only — provider is NOT in any cache key

pipeline/cache.py is 23 lines; the entire key scheme (lines 10–23, verbatim):

def cache_root() -> Path:
    return Path(os.environ.get("XLO_CACHE_DIR", "data")) / "orchestrator"

def song_key(path: str) -> str:
    h = hashlib.sha1()
    with open(path, "rb") as f:
        for chunk in iter(lambda: f.read(1 << 20), b""):
            h.update(chunk)
    return h.hexdigest()[:16]

def cache_path(key: str, stage: str) -> Path:
    return cache_root() / key / f"{stage}.json"

Call sites (all of them, from grep): run.py:380 (song_analysis, written for xlo regen rehydration), run.py:391 (song_description — the LLM panel output), run.py:409 (creative_brief — the Director's LLM output), run.py:426 (instructions — the generator's LLM output), run.py:474 (post-refine instructions rewrite), plus regen.py:46,61–62,69,71,122 which rehydrate the same four stages, and cli.py:59 (edit-brief locates the cached brief). The read path for the LLM stages, e.g. run.py:391–394:

    mb_cache = _cache_path(key, "song_description")
    if use_cache and mb_cache.exists():
        try:
            st.music_brief = MusicBrief.model_validate_json(mb_cache.read_text())

This is the design issue at the heart of F-H. The cache key is sha1(audio bytes)[:16] + stage name — nothing else. Provider and model appear nowhere. Concretely, today:

xlo run --song x.mp3 --auto                     # anthropic (default) — populates the cache
XLO_PROVIDER=gemini xlo run --song x.mp3 --auto # "gemini run" — silently REUSES anthropic's
                                                # song_description, creative_brief, instructions

The second run's revision log would even record models: {director: google:gemini-3.1-pro-preview, ...} (from model_snapshot()) against outputs that Claude produced. Any naive A/B built on top of xlo run is invalid unless it passes --no-cache — and --no-cache throws away the one thing the arms should share (see below). Note the two caches that are provider-independent and correctly shared today: song_analysis (deterministic audio feature extraction — in fact run_pipeline re-analyzes every run regardless of use_cache; the JSON at run.py:380 exists only for regen.py:71 rehydration) and the targetable-groups probe cache, keyed by a layout fingerprint, not the song (pipeline/groups.py:53: targetable_groups_{_fingerprint(names, model_names)}.json).

2.3 The comparison metrics already exist (except cost)

qa/__init__.py::evaluate (lines 16–35) produces the deterministic QAReport: objective_score (mean of sync/placement/rules, plus coverage when a frame sampler exists), advisory_score (variety), per-metric subscores, and findings. Only the objective drives the refine loop's revert/stall gate; both are logged. revision_log.py::RevisionLogRecord (lines 33–52) captures per-iteration objective_score, advisory_score, obj_before/obj_after/obj_delta, reverted, regenerated_sections, judge score/verdict, a ts timestamp, and — crucially for A/B — models: dict[str, str], the per-role routing snapshot. The JSONL at cache_root()/<song_key>/revision_log.jsonl is append-only across runs (run.py:457–459), with run_id a UTC timestamp. So per-arm score trajectories, iteration counts, and convergence behavior are already recorded and already attributable to a routing — what is missing is (a) a safe way to produce the two arms, and (b) an analyzer. Token cost is entirely absent (I1; no result.usage() capture exists anywhere — grep confirms no RunUsage references in packages/).

2.4 CLI and test conventions the harness must fit

cli.py uses one argparse parser with subcommands run, regen, edit-brief (lines 67–116), each a small async _cmd(args) driven by asyncio.run inside an async with XLightsClient(). Unattended mode is already a first-class path: --auto wires checkpoint=_auto_checkpoint and passes None for the interpret/design gates (cli.py:27–29), so no input() blocks. Tests: the repo is hermetic-first — test_golden_pipeline.py drives run_pipeline end-to-end with TestModel agents, a _FakeClient, an injected analyze and emitter, and XLO_CACHE_DIR pointed at tmp_path; the single opt-in marker is live (pyproject.toml: addopts = "-m 'not live'", markers = ["live: requires a running xLights instance (opt in with -m live)"]).

3. Proposed design

3.1 Harness shape: CLI command vs pytest suite

Option A — xlo ab CLI commandOption B — pytest-marked eval suite (e.g. -m ab)

SemanticsProduces data (score/cost distributions) — the honest output of a stochastic comparisonpytest wants pass/fail; any threshold assertion on stochastic LLM output is a flaky test by construction

A long-running CLI with progress output is the right UXSame runtime buried inside a test runner; no incremental output; a Ctrl-C loses everything; CI cannot run it anyway until F-J (headless mode) exists
Repeats/armsNatural flags (--repeat, repeated --arm)Parametrize-able but awkward to vary from the command line

Recommendation: Option A, the CLI command, with a thin hermetic pytest layer under the normal (non-marked) suite. The pytest layer does not run providers at all — it tests the harness itself with the same fakes the golden test uses: arm-spec parsing, per-arm env isolation and restoration, cache namespacing (section 3.4), and summary computation over synthetic revision_log.jsonl lines. A pytest-based live eval suite is the wrong tool until F-J makes a CI-runnable pipeline possible; if that lands, the same run_ab() entry point can be invoked from an ab-marked test without redesign.

3.2 xlo ab — command surface (Proposed)

xlo ab --song fixtures/song.mp3 \
       --arm anthropic \
       --arm gemini \
       --arm "gemini+judge=anthropic,visual_critic=anthropic" \
       --repeat 3 --max-iterations 3 [--name-prefix ab] [--keep-sequences]
  • --arm SPEC (repeatable, ≥2): SPEC = base_provider *("+" role "=" provider). anthropic flips everything; gemini+judge=anthropic is the per-role mixing experiment (Gemini generator/director, Claude judge). Roles and providers validate against _cfg()["roles"] / _cfg()["providers"] so a typo dies at parse time, before any spend.
  • --repeat N (default 1): N pipeline runs per arm, interleaved A,B,A,B,… (not A,A,B,B) so slow drift in the environment (xLights session state, network) spreads across arms.
  • Always unattended: the harness hard-wires refine=True, checkpoint=_auto_checkpoint, interpret_checkpoint=None, design_checkpoint=None, log_revisions=True — an A/B with a human in the loop is measuring the human.
  • Key preflight: unlike xlo run's has_llm_key() (either key suffices, config.py:25–26), ab checks that every provider named by any arm has its key (ANTHROPIC_API_KEY / GEMINI_API_KEY) and refuses to start otherwise — failing arm B after arm A spent money is the worst outcome.

New module packages/xlights-orchestrator/src/xlights_orchestrator/pipeline/ab.py (Proposed):

# Proposed — pipeline/ab.py
@dataclass(frozen=True)
class ArmSpec:
    label: str                      # the literal --arm string
    provider: str                   # base provider
    role_overrides: dict[str, str]  # role -> provider

def parse_arm(spec: str) -> ArmSpec: ...

@contextmanager
def arm_env(arm: ArmSpec):
    """Set XLO_PROVIDER (+ XLO_PROVIDER_<ROLE> per override) for the duration of one
    pipeline run; restore prior values on exit, even on failure."""

async def run_arm(arm: ArmSpec, song: str, *, client, analysis: SongAnalysis,
                  max_iterations: int, save_as: str | None) -> ArmRunResult: ...

async def run_ab(song: str, arms: list[ArmSpec], *, repeat: int, ...) -> ABResult: ...

run_arm calls run_pipeline in-process under arm_env. This works because of the facts established in §2.1: agents are constructed lazily inside the run, and the registry consults os.environ at construction time. Arms run sequentially — the XLightsClient mutation lock and single live app make parallel arms impossible, and the process-global env swap requires it anyway (enforce with a comment and the context manager; do not attempt concurrency here).

3.3 Per-role provider overrides in the registry (Proposed)

The mixing experiment needs role-level routing at runtime. The registry already has the per-role provider= parameter on model_string/_settings; what is missing is a resolution rule and its plumbing. Extend models/registry.py:

# Proposed — models/registry.py
def provider_for(role: str) -> str:
    """Provider for a role: XLO_PROVIDER_<ROLE> > XLO_PROVIDER > config default."""
    return (os.environ.get(f"XLO_PROVIDER_{role.upper()}")
            or os.environ.get("XLO_PROVIDER")
            or _cfg().get("default_provider", "anthropic"))
  • model_string line 33 and _settings line 49 change provider or active_provider() → provider or provider_for(role).
  • model_snapshot() (lines 38–45) switches from one global prov = active_provider() to provider_for(role) per role — this is what makes a mixed arm's revision-log models field truthful (e.g. {"generator": "google:gemini-3.1-flash-lite", "judge": "anthropic:claude-opus-4-8", ...}).
  • build_agent needs no signature change — it routes through model_string(role) which now resolves per-role.
  • active_provider() stays (used as the fallback and by any external callers) but becomes the base of provider_for.

Env vars (rather than a new function parameter threaded through six agent factories) match the config header's stated philosophy — "no code change" runtime re-pointing — and mean the mechanism also works for plain manual runs: XLO_PROVIDER_JUDGE=gemini xlo run … is useful on its own, independent of the harness. Role names in the env var are the literal config.yaml role keys, uppercased; provider_for validation errors (unknown provider for a role) surface as the same KeyError path model_string raises today.

3.4 Cache isolation: model fingerprint in the LLM-stage cache key (the fix)

Requirements derived from §2.2: (a) two arms must never read each other's LLM-stage artifacts; (b) both arms should share the audio analysis and the targetable-groups probe (provider-independent, expensive — stems separation and live probing respectively); (c) xlo regen and xlo edit-brief, which rehydrate the same caches (regen.py:46,61–71, cli.py:59), must keep working and must pick up the artifacts of the current routing.

Proposed change to pipeline/cache.py — add a models-fingerprint namespace, opt-in per stage:

# Proposed — pipeline/cache.py
def models_fingerprint() -> str:
    """Stable 8-hex digest of the ACTIVE role->provider:model routing."""
    from ..models.registry import model_snapshot
    snap = model_snapshot()                      # per-role after 3.3
    blob = ",".join(f"{k}={v}" for k, v in sorted(snap.items()))
    return hashlib.sha1(blob.encode()).hexdigest()[:8]

def cache_path(key: str, stage: str, *, models: bool = False) -> Path:
    base = cache_root() / key
    if models:                                   # LLM-stage artifacts: provider/model-scoped
        base = base / f"m-{models_fingerprint()}"
    return base / f"{stage}.json"

One whole-snapshot fingerprint (rather than per-stage "only the roles that feed this stage") is deliberate: it over-invalidates slightly (changing only the judge model re-namespaces the generation caches) but it is impossible to get wrong, and stage inputs are transitively coupled anyway (instructions realize the brief, which renders the description). Human-facing companions (description.md, creative_brief.md, creative_brief.json) follow their stage: they are written next to the cache file (run.py:402,416–418), so they land in the namespace directory automatically. No legacy-path read fallback: falling back to the un-namespaced path re-opens the exact cross-provider reuse bug this fixes. Existing caches are simply cold after the change — acceptable, they regenerate on the next run (and regen.py's FileNotFoundError message already says "run xlo run first").

This fix is valuable independent of the harness: it turns XLO_PROVIDER=gemini xlo run from silently-wrong (§2.2) into correct, and it should land first (see §4).

3.5 What is held constant, and the honest nondeterminism inventory

Held constant across arms, by construction:

  • Audio analysis — the harness analyzes once and injects the same SongAnalysis object into every arm via run_pipeline's existing analyze parameter (run.py:326,350–351). This is stronger than "shared cache": identical by identity, and it skips the expensive stems separation on every repeat.
  • Layout and group vocabulary — same live xLights instance, and the harness warms the targetable-groups probe cache once before arm 1 so no arm pays (or races) the probe (groups.py:25: _SETTLE_SECS = 0.8, commented "racy — mirrors the emitter").
  • Refine parameters — same --max-iterations (default 3, cli.py:81); the skip gate XLO_REFINE_SKIP_OBJECTIVE (run.py:62–70, default 88) is recorded in the arm result, since a first pass at/above it skips refinement entirely (run.py:211–216) and makes iteration counts incomparable if it differs.
  • Deterministic downstream — the weave expansion, beds, accents, occlusion guard, and QA scoring are pure functions of the LLM outputs (that is what the golden test proves), so differences in scores are attributable to the LLM stages plus render sampling.

Not constant, and not controllable today — the harness compares distributions, not outputs:

3.6 Metrics and the comparison report

Per run (one row per run_id, extracted from revision_log.jsonl plus harness timing): first-pass objective_score and advisory_score; final scores (the kind == "finalize" record, run.py:305); iteration count; per-iteration obj_delta trajectory; revert count (reverted flags) and plateau/stall exits (human_decision values "plateau"/"skip-high-objective", judge accept/stop); regenerated-section counts; QA subscores per metric (sync/placement/rules/variety/coverage — the per-role diagnostic: a generator change moves rules/placement, a director change moves variety/coverage patterns); wall time per stage and total, measured by the harness (the log's ts gives per-iteration deltas but not stage splits).

Without I1 the harness fully supports: scores, subscores, iteration counts, convergence shape, wall time. With I1 (per-role, per-iteration token counts captured from PydanticAI result.usage() into the same JSONL records, priced from a rate table in config.yaml) it adds: tokens and dollars per run, per role, and the roadmap's real target metric, cost per objective point gained. The harness should land after I1 (roadmap sequencing step 8 vs step 1) but nothing in its design blocks on it — cost columns simply appear when the records carry usage.

Results land in the F-G surface, not a new format. The F-G cost & quality dashboard is an analyzer over revision-log JSONL with terminal/HTML report output. Because arms share one per-song revision_log.jsonl (§3.4) and every record carries the per-role models dict (truthful after §3.3), A/B analysis is a group-by in that analyzer — group records by run_id, label each run by its models signature, aggregate per label. The harness additionally writes a tiny manifest, cache_root()/<song_key>/ab_runs.json (Proposed): {ab_id, song_key, arms: [{label, run_ids: [...], models, wall_time_s, ...}]} — so the reporter can distinguish "runs that belong to this deliberate A/B" from ordinary historical runs, and so interrupted A/Bs are visible. xlo ab prints the terminal summary itself (medians and ranges per arm, per-metric deltas) and defers anything richer to F-G's HTML report, which gains an "A/B" section keyed off the manifest.

3.7 Statistical honesty with n=1 songs

One song × one run per arm — the current manual practice, per the config.yaml comment — supports almost no conclusion: with unseeded sampling, within-arm variance on objective_score is nonzero and unmeasured, so a 3-point between-arm gap is uninterpretable. The harness's posture:

  • Fixture set: a small committed set of 3–5 short songs spanning the axes the pipeline branches on — vocal vs instrumental (different panel/synthesizer path, run.py:354–377), high vs low energy (drives the skip gate and peak treatments), simple vs complex sectioning. Short (≤90s) keeps per-arm wall time and spend sane. These live next to tests/fixtures/ conventions but are real audio, so likely data/ab_fixtures/ with a README, not in-repo MP3s if licensing forbids.
  • Repeats: --repeat 3 minimum before believing any delta; report per-arm median and min–max range for every metric. Within-arm range is the noise floor: an arm-vs-arm delta smaller than either arm's range is reported as "indistinguishable".
  • Valid conclusions: categorical failures ("gemini flash-lite emitted invalid structured output in 2/3 runs"), consistent large gaps (non-overlapping ranges across ≥3 repeats on ≥2 songs), cost ratios (token counts are far less noisy than scores). Invalid: "provider A is better", any single-run comparison, extrapolation beyond the fixture songs or the current prompt set — a prompt edit invalidates prior A/B data (the models fingerprint does not hash prompts; note this in the report header).
  • No significance tests in v1 — with n ≤ 5 songs × 3 repeats, a sign-test veneer would imply rigor the data cannot carry. Ranges and honest wording instead.

5. Testing strategy

Hermetic-first, per repo convention (no key, no xLights, no network; the default suite excludes live):

  • Registry: XLO_PROVIDER_JUDGE=gemini + XLO_PROVIDER=anthropic → model_string("judge").startswith("google:") while model_string("director").startswith("anthropic:"); model_snapshot() reflects the mix; unknown role/provider raises. Mirrors test_registry_reroute_via_env (tests/test_orchestrator.py:64–67), via monkeypatch.setenv.
  • Cache isolation: with XLO_CACHE_DIR at tmp_path (the golden test's pattern, test_golden_pipeline.py:167), run the fake-agent pipeline under provider A, flip to provider B, assert the B run does not read A's creative_brief/instructions (distinct m-* dirs on disk) while song_analysis and targetable_groups_* paths are shared. This is the regression test for the §2.2 bug and the most important test in the change.
  • Arm spec parsing: round-trips, validation failures, env set/restore including on exception inside arm_env.
  • Harness orchestration: run_ab over the golden-test fakes (TestModel agents, _FakeClient, fake emitter, injected analysis) with 2 arms × 2 repeats: assert 4 runs, interleaved order, distinct run_ids, one shared revision_log.jsonl, correct ab_runs.json, and truthful per-arm models in the records.
  • Summary math: summarize_runs over hand-written JSONL (reuse test_revision_log.py record-building style) — medians, ranges, revert counts, the "indistinguishable when ranges overlap" wording rule.
  • Regen compatibility: regen.load_cached_state finds the namespaced artifacts written by a run under the same routing, and raises the existing FileNotFoundError under a different routing.
  • Live (opt-in, -m live, manual): one short two-arm, one-repeat run against real providers and xLights as the end-to-end verify before archiving the change — the harness's own output doubles as the evidence.

7. Acceptance criteria

  • XLO_PROVIDER=gemini xlo run --song x.mp3 after an anthropic run does not reuse anthropic's song_description/creative_brief/instructions caches (regression test exists), while re-using the shared song_analysis and targetable_groups_* caches.
  • XLO_PROVIDER_<ROLE> overrides route exactly one role; model_snapshot() and revision-log models reflect the true mix.
  • xlo ab --song S --arm anthropic --arm gemini --repeat 2 completes unattended, produces 2×2 runs with distinct namespaced caches, one shared revision_log.jsonl, an ab_runs.json manifest, and a terminal summary showing per-arm median/range for first/final objective, advisory, iterations, reverts, and wall time.
  • A mixed arm (gemini+judge=anthropic) runs and is labeled truthfully everywhere.
  • Refusal before spend: missing key for any named provider, malformed arm spec, unknown role.
  • An interrupted A/B leaves a valid manifest covering completed runs.
  • All new logic (parsing, env isolation, cache namespacing, summaries) covered by hermetic tests; default pytest run stays key-less and xLights-less; golden pipeline test unchanged.
  • Post-I1: per-arm token/cost columns and cost-per-objective-point appear with no harness change beyond the summary function.

8. Open questions

  • Should the models fingerprint also hash the agents' system prompts (or a package version), so prompt edits auto-invalidate LLM caches and A/B comparability is machine-checked rather than a report caveat? (Broader than F-H — it changes cache behavior for everyone.)
  • Fixture songs: real licensed MP3s under data/ (not committed) vs synthesized short audio committed to the repo — synthesized audio is hermetic but may not exercise stems/lyrics paths meaningfully.
  • Same-provider A/B (e.g. anthropic vs anthropic with a different effort, or sonnet-vs-opus per role): the arm grammar covers providers only; extending it to inline model/settings overrides (anthropic+generator.model=claude-haiku-4-5) is easy to parse but starts duplicating config.yaml — alternatively support --config path/to/alt-config.yaml per arm.
  • Interaction with F-J (headless mode): once offline-render iteration exists, should xlo ab gain a --headless flag and a CI/pytest entry point? The run_ab() seam is designed so this is additive.
  • Does the visual critic belong in A/B arms at all pre-I8? It is the most expensive role, advisory only, and doubles arm cost; a --no-visual flag (mapping to visual_critique=None… note run_pipeline currently builds a real critique when None is passed and refine=True, run.py:451–454 — an explicit off switch would be a small new parameter).
